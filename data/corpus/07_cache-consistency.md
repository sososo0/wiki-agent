# Cache Consistency

## Write-Through Caching

Write-through is a cache update pattern where data is written to both the cache and the backing store before the write operation completes. This ensures cache consistency at the cost of write latency, since every write must wait for both the cache and persistent storage to acknowledge. The pattern is straightforward to reason about and prevents stale data, but introduces higher write latencies compared to write-behind approaches. Write-through is suitable for systems where consistency is prioritized over write performance, such as financial transactions or user account updates. The main tradeoff is that write operations block until both layers acknowledge, which can bottleneck under high write throughput.

## Write-Behind Caching

Write-behind (also called write-back) caching decouples cache writes from persistent storage writes by immediately acknowledging writes to the client while asynchronously flushing data to the backing store. This improves write latency but introduces a consistency window where the cache contains uncommitted data. If the cache fails before flushing, uncommitted writes are lost. Write-behind requires careful handling of flush ordering and failure scenarios. It's appropriate for use cases where eventual consistency is acceptable, such as logging, metrics, or non-critical user preferences. The pattern significantly improves write performance at the expense of durability guarantees and implementation complexity.

## Cache Invalidation by Event

Event-driven cache invalidation clears or updates cache entries in response to application events that modify underlying data. When an operation changes data, an event is published that triggers invalidation across relevant caches. This pattern ensures consistency is maintained actively rather than passively through TTL expiry. Event-based invalidation requires tight coupling between the data modification logic and cache invalidation logic, typically implemented through message queues or event buses. The pattern works well in systems with clear data ownership and predictable access patterns, but scales poorly when many caches depend on the same data, as each cache must subscribe to and process invalidation events.

## Time-To-Live Expiry

TTL-based expiry automatically removes cache entries after a specified duration, ensuring stale data is eventually discarded without explicit invalidation logic. The cache entry is considered valid for the TTL period and then automatically evicted or marked invalid. This approach requires no coordination between data producers and consumers, making it simple to implement and scale. However, TTL expiry introduces a consistency window where stale data may be served. The ideal TTL duration depends on the acceptable staleness threshold; shorter TTLs improve consistency but increase cache miss rates and backend load. TTL expiry is appropriate for read-heavy data that changes infrequently, such as user profiles or configuration that allows some staleness.

## Stale-While-Revalidate

Stale-while-revalidate is a cache consistency pattern that serves stale cache data to clients immediately while asynchronously refreshing the cached value in the background. When a cache entry expires, instead of blocking to fetch fresh data, the stale value is returned and a background refresh is initiated. This provides low latency responses even when data needs updating, though clients may briefly receive stale information. The pattern is effective for use cases where eventual consistency is acceptable and immediate availability is valued, such as public web content or frequently accessed reference data. Implementing stale-while-revalidate requires mechanisms to differentiate between serving stale data and triggering refreshes, typically through separate expiry timestamps or background workers.

## Cache-Aside Pattern

Cache-aside (also called lazy loading) shifts cache population responsibility to the application. When data is requested, the application first checks the cache; if missing, it fetches from the backing store and populates the cache before returning. This pattern is flexible and works well with any backing store, but places consistency responsibility on application logic. Cache-aside doesn't guarantee fresh data—stale entries remain cached until explicitly invalidated or TTL expires. The pattern creates additional complexity around cache misses and requires careful handling of the check-then-set race condition using atomic operations. Cache-aside is common in systems with diverse data sources or where cache population logic varies by data type.

## Distributed Cache Coherence

Distributed cache coherence refers to mechanisms ensuring consistency across multiple cache instances in a distributed system. When data is cached at multiple locations, coherence protocols ensure all caches reflect the same current state. Approaches include write-invalidate (invalidate copies when data changes), write-update (propagate new values), or lazy invalidation through versioning. Coherence is fundamentally challenging in distributed systems due to network latency and partitions. Strong coherence guarantees require significant coordination overhead and can bottleneck performance. Most distributed systems accept eventual consistency through TTL expiry or event-based invalidation rather than strict coherence protocols. Implementing coherence requires careful consideration of fault tolerance and the acceptable consistency window.

## Negative Caching

Negative caching stores the result of a cache miss or failed lookup, recording that a value does not exist or that a request failed. Rather than repeatedly querying for non-existent data, negative cache entries prevent redundant backend lookups. For example, caching that a user ID doesn't exist prevents repeated database queries for that ID. Negative caches must have explicit expiry since the absence of data may change over time. The pattern significantly reduces backend load for queries on non-existent data but requires careful TTL tuning—overly long durations may cache absence for too long if the data is later created. Negative caching is valuable in systems with high lookup traffic and non-existent data queries, such as auth systems or distributed caches.

## Cache Stampede Prevention

Cache stampede (also called thundering herd) occurs when a popular cache entry expires, and multiple concurrent requests simultaneously miss the cache and flood the backend with identical queries. Each request independently detects the miss and attempts to fetch and refresh the cache. Without prevention, the backend receives a sudden traffic spike potentially causing outages. Stampede prevention techniques include probabilistic early expiry (refreshing before true expiry), request coalescing (blocking concurrent misses until first fetch completes), or using background refresh workers. The choice depends on consistency requirements and system scale. Request coalescing provides strong consistency but blocks clients; probabilistic expiry prevents backend load but may refresh unnecessarily. Most distributed systems employ multiple techniques layered together.

## Lease-Based Consistency

Lease-based cache consistency grants a cache the exclusive right to serve a data item for a limited time period (a lease). After lease expiry, the cache must revalidate with the source before serving. Leases allow reasonably long consistency windows while preventing permanent stale reads. When data changes, the source can revoke leases early, forcing immediate revalidation. The pattern is used in systems like distributed file systems where clients cache file metadata. Leases balance consistency with cache efficiency by allowing autonomous caching within lease bounds. Implementation requires lease state management at both client and server, and mechanisms for early revocation. Leases are most suitable where the lease holder needs some autonomy but the data source must maintain control.

## Version-Based Invalidation

Version-based invalidation decorates data with version numbers or timestamps, allowing caches to detect stale entries without explicit invalidation messages. When a cache entry is requested, its version is compared with the current version; if mismatched, the entry is refreshed. This decouples cache invalidation from explicit notification and is resilient to missed invalidation events. Version-based approaches scale well since caches can independently check freshness. However, every access requires version comparison, adding latency and complexity. Versions can be simple counters, timestamps, or content hashes. This pattern is effective in systems where data is frequently accessed and staleness detection must be reliable, such as distributed caches or shared memory systems.

## Consistency Hashing for Cache Nodes

Consistency hashing is a technique for distributing cache entries across multiple cache nodes while minimizing remapping when nodes are added or removed. Traditional hashing maps keys directly to nodes; when a node is added, most keys must remap, causing cache misses. Consistency hashing maps both keys and nodes to a hash ring, ensuring that adding a node only affects a small subset of keys. This reduces cascading cache misses during cluster scaling. Consistency hashing doesn't directly address consistency between cache and backing store, but it improves cache efficiency during topology changes. The pattern is standard in distributed caching systems like Redis Cluster or Memcached clusters. Trade-offs include slightly more complex implementation and potential for uneven load distribution if not carefully tuned.

## Read Repair

Read repair is a consistency mechanism where applications detect and correct inconsistencies when reading cached or replicated data. When a read occurs, if the value is stale or inconsistent, it is repaired (refreshed) as part of the read operation. This pushes consistency responsibility to readers rather than requiring proactive invalidation. Read repair improves consistency for frequently accessed data and is lazy in the sense that repairs only occur on reads. However, rarely accessed data may remain stale indefinitely, and reads may incur repair latency. Read repair is commonly used in distributed databases and replicated systems alongside other consistency mechanisms. The pattern is most effective when combined with other strategies like TTL expiry or version checking.

## Two-Phase Invalidation

Two-phase invalidation extends cache invalidation with explicit acknowledgment from cache nodes, ensuring invalidation is reliably received before completing the invalidation operation. The first phase sends invalidation requests to all relevant caches; the second phase waits for acknowledgments. This prevents the race condition where an invalidation message is lost and a cache continues serving stale data while the source believes it has been invalidated. Two-phase invalidation provides strong consistency guarantees but introduces latency and requires reliable acknowledgment mechanisms. It's suitable for systems where consistency is critical, such as shared resource management or distributed locks. The pattern can be simplified through timeout mechanisms, though this reintroduces small consistency windows if timeouts occur.

## Probabilistic Early Expiry

Probabilistic early expiry, also called xfetch, refreshes cache entries before their true expiry time based on a probabilistic function of time. Rather than waiting until TTL expires, cache entries have a refresh probability that increases as expiry approaches. This spreads refresh load over time rather than concentrating it at the expiry moment, preventing cache stampedes. The technique uses a small background refresh rate early in an entry's lifetime and accelerates refresh probability as true expiry approaches. With tuning, most cache hits remain within the expiry window while avoiding thundering herd issues. The pattern trades some excess refreshes (entries refreshed before strictly necessary) for better backend load distribution. Probabilistic expiry is particularly valuable for high-traffic cache entries with long TTLs.

## Multi-Tier Cache Consistency

Multi-tier caching uses multiple cache levels (L1 near clients, L2 shared, L3 backing store) to reduce latency and backend load. However, maintaining consistency across multiple tiers is challenging—ensuring L1 caches don't serve stale data when L2 or backing store are updated requires coordination. Common approaches include cascading invalidation (invalidating all tiers when data changes) or independent expiry with eventual consistency (each tier has its own TTL). Cascading invalidation provides stronger consistency but creates tight coupling between tiers. Independent expiry is simpler but requires accepting broader consistency windows. Tier-specific consistency policies are often tuned—L1 caches may use shorter TTLs than L2 for tighter local consistency, while L2 accepts longer staleness. Multi-tier consistency requires careful analysis of staleness tolerance at each tier.

## Conditional Writes for Consistency

Conditional writes verify that a cache entry hasn't been modified before allowing an update, catching and preventing lost updates from concurrent modifications. The write succeeds only if the current cached version matches an expected version (checked via version number, timestamp, or CAS—compare-and-swap). If the condition fails, the write is rejected and the application must retry, typically after refreshing. Conditional writes prevent one writer from silently overwriting another writer's changes. The pattern is essential in cache-aside and write-behind implementations where concurrent updates are possible. However, under high contention, conditional write failures increase and applications may need retry logic with backoff. Conditional writes are a core component of optimistic locking strategies and are supported natively by most distributed caches.

## Refresh-on-Access

Refresh-on-access extends the TTL of a cache entry each time it's accessed, keeping hot data fresh indefinitely while cold data eventually expires. Rather than fixed expiry times, entries' expiry times are extended on every hit. This adapts cache behavior to access patterns—frequently accessed data remains cached longer while rarely accessed data is evicted sooner. The pattern reduces overall invalidation operations for hot data and provides graceful degradation under storage pressure. However, it can mask data staleness for active users if refresh happens less frequently than the source data changes. Refresh-on-access requires per-access overhead to update expiry times and is most effective when access patterns are skewed (few items accessed frequently). The pattern is commonly used in session caching and reference data caching.

## Tombstone Records for Deletions

Tombstone records are cache entries marking data as deleted rather than completely removing the entry. Instead of immediately removing a cache entry when source data is deleted, a tombstone entry indicates deletion with metadata like the deletion timestamp. Tombstones prevent subsequent cache misses and redundant backend lookups for deleted data. Eventually, tombstones themselves expire after a grace period. This pattern is particularly valuable in distributed systems where deletion messages may not reach all cache replicas. Tombstones allow systems to distinguish "never cached" from "exists and is deleted" and support deletion semantics in distributed caches. The tradeoff is additional memory used by tombstone entries and complexity in cache eviction policies. Tombstones are essential in systems with eventual consistency and distributed replication.

## Cache Coherence Protocols

Cache coherence protocols are formal mechanisms ensuring consistency in distributed caches, drawn from multiprocessor architectures and adapted for distributed systems. Common protocols include write-invalidate (invalidating other copies on write), write-update (propagating new values), and variants combining both. Invalidation protocols minimize write traffic at the cost of read latency after writes. Update protocols reduce read latency but increase write traffic. Practical distributed systems rarely implement full coherence protocols due to network latency and partition tolerance, but borrow concepts. Instead, systems typically use weaker consistency models with explicit invalidation or TTL-based expiry. Understanding coherence protocols is valuable for reasoning about consistency tradeoffs, though most distributed systems implement simplified variants rather than strict protocols. Protocols provide a vocabulary for discussing consistency properties and tradeoffs.

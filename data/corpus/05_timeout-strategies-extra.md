# Timeout Strategies

## Circuit Breaker Timeout Coordination

Circuit breakers and timeouts serve complementary functions in preventing cascading failures. A circuit breaker should open based on error rates or latency patterns, while timeouts act as a fail-fast mechanism for individual requests. Coordinate these by setting timeout values lower than the circuit breaker's detection threshold. For example, if a service typically responds in 100ms, set individual request timeouts to 500ms, but configure the circuit breaker to open if 50% of requests exceed 2 seconds over a 30-second window. This ensures individual slow requests fail quickly without blocking the circuit breaker's aggregated health assessment. When timeouts consistently trigger before the circuit breaker activates, the downstream service may be degraded but not fully failing—in this case, reduce timeout values or adjust circuit breaker sensitivity. Avoid setting timeouts longer than circuit breaker detection windows, as this defeats the circuit breaker's purpose and allows requests to accumulate while waiting for failure signals.

## Timeout Retry Budget Interaction

Retries and timeouts interact critically in distributed systems. Each retry consumes time from your overall latency budget, so timeout values must account for retry attempts. If you retry up to 3 times with a 1-second timeout per attempt, the maximum end-to-end latency is 3 seconds—which may exceed acceptable SLOs. Implement retry budgets that limit total retry attempts per request. For instance, allow retries only on specific transient errors (connection timeouts, 503 responses), not on application-level failures. Set timeout values such that `timeout × max_retries + inter-retry_delay ≤ acceptable_latency_budget`. Document which errors trigger retries to prevent retry amplification on the upstream side. Consider exponential backoff with jitter between retries to reduce thundering herd effects. Monitor retry rates in production; consistently high retry rates indicate either aggressive timeout settings or upstream instability requiring investigation.

## Per-Endpoint Timeout Overrides

Different endpoints within the same service often have different latency characteristics. Database migrations, report generation, and simple lookups all have distinct performance profiles. Implement per-endpoint timeout configuration allowing service owners to override global defaults. Store overrides in a configuration file or service mesh layer (e.g., Istio VirtualService) rather than hardcoding them. Example: a `/health` endpoint might use a 100ms timeout, while `/export-report` uses 30 seconds. Document the rationale for each override. Use monitoring to detect when actual latencies approach or exceed configured timeouts—this indicates the override needs adjustment. Avoid excessive fragmentation; if many endpoints require unique timeouts, the underlying service architecture may need optimization. Provide tooling for teams to test timeout changes before deployment, such as synthetic load testing with instrumentation to measure p99 latencies.

## Timeout Testing in CI/CD

Timeout behavior must be validated in continuous integration to catch configuration errors before production. Implement tests that verify timeout enforcement: create test cases that simulate slow backends (using sleep or network delays) and confirm requests fail within expected time windows. Use tools like toxiproxy or tc (traffic control) to inject latency into test environments. Test the interaction between timeouts and retries—verify that a request with 3 retries fails within the total timeout budget, not `timeout × 3`. Include tests for timeout behavior under load; timeout accuracy may degrade under resource contention. Verify that timeout errors are distinguishable from other failures in logs and metrics. Create tests for timeout configuration changes; deploy a new timeout value and confirm it takes effect without requiring service restarts. Document expected timeout values in test assertions so future developers understand intent. Avoid flaky tests by using deterministic delays rather than variable network conditions when possible.

## gRPC Deadline vs HTTP Timeout Semantics

gRPC and HTTP handle timeouts differently, creating confusion in polyglot systems. gRPC uses deadlines—absolute timestamps by which an operation must complete—propagated across service boundaries via the `grpc-timeout` metadata header. HTTP uses timeouts—relative durations for individual requests. When gRPC services call HTTP backends, deadline-to-timeout conversion is required: subtract current time from deadline to calculate remaining duration. Languages differ in deadline handling; Go's context package enforces deadlines, while other runtimes may implement them permissively. HTTP intermediaries (proxies, load balancers) may not propagate gRPC timeout headers, requiring explicit configuration. Set gRPC deadlines at entry points (API gateways or user-facing services) and let them propagate downstream. For HTTP-to-gRPC bridges, ensure timeout values align; a gRPC service with a 5-second deadline should not call an HTTP endpoint with a 10-second timeout. Test timeout behavior with mixed protocol stacks to verify deadlines propagate correctly and enforce at the right layers.

## Timeout Configuration for Batch Jobs vs Interactive Requests

Batch jobs and interactive requests have fundamentally different timeout requirements and should not share configuration. Interactive requests (API calls, web requests) require short timeouts—typically 5-30 seconds—to maintain responsiveness and bound resource consumption. Batch jobs (data exports, report generation, nightly syncs) may run for minutes or hours and need much longer timeouts or no timeout at all. Use separate configuration namespaces for batch and interactive paths. For batch jobs, consider removing hard timeouts entirely and instead implement progress monitoring—if a batch job hasn't updated state in 2 hours, manually investigate rather than terminating it. If timeouts are required, set them based on the 99th percentile of historical execution time plus buffer. Interactive requests should use timeouts based on user-acceptable latency, not actual average execution time. Implement exponential backoff for batch retries to reduce infrastructure load during failures. Monitor timeout expiration separately for each category—high timeout rates on interactive requests indicate performance problems, while timeout rates on batch jobs may indicate resource contention or code bugs requiring investigation.

# Circuit Breaker Patterns

## Closed State Fundamentals
The closed state represents normal operation where requests flow through to the downstream service without interruption. In this state, the circuit breaker monitors failures but does not block traffic. Each failed request increments a failure counter; each successful request typically resets or decrements this counter. The circuit breaker remains closed until the failure count reaches a configured threshold within a time window. This is the default operational mode and should represent the majority of uptime for healthy systems. The closed state requires minimal overhead—typically just incrementing counters on each request.

## Open State Behavior
The open state occurs when failure thresholds are exceeded, causing the circuit breaker to stop forwarding requests to the failing service. Instead of attempting calls, the circuit breaker immediately rejects new requests, typically returning an error or executing a fallback. This prevents cascading failures by failing fast rather than accumulating timeout delays. The open state protects downstream resources from being exhausted by requests to an unresponsive service. A timeout mechanism must eventually transition the circuit to half-open to allow recovery attempts. The open state can significantly reduce latency by eliminating wasted calls to unavailable services.

## Half-Open State Mechanics
The half-open state allows the circuit breaker to test whether a failing service has recovered. After a configured timeout from entering the open state, the circuit transitions to half-open and allows a limited number of test requests through. If these test requests succeed, the circuit returns to closed; if they fail, it returns to open and resets the timeout. This state prevents the circuit from remaining open indefinitely and enables automatic recovery. Most implementations limit the number of concurrent test requests in half-open state to avoid overwhelming a recovering service. The half-open state is typically brief, lasting only as long as needed to determine service health.

## Failure Threshold Configuration
Failure thresholds define the conditions that trigger a transition from closed to open state. Common threshold types include: absolute count (e.g., 5 consecutive failures), percentage-based (e.g., 50% of requests fail), and time-windowed (e.g., 3 failures within 10 seconds). Threshold selection requires understanding your service's normal error rate and acceptable degradation. Too low a threshold causes false positives and unnecessary circuit trips; too high a threshold delays protection against real failures. Thresholds should reflect the specific characteristics of each downstream dependency, as different services have different reliability profiles. Testing threshold values under realistic load conditions is essential before production deployment.

## Success Threshold in Half-Open State
Success thresholds determine how many successful test requests are required to transition from half-open back to closed. Common approaches include: single success (transition immediately), count-based (e.g., 2 successful requests), and percentage-based (e.g., 80% of test requests succeed). Lower success thresholds enable faster recovery but risk returning to closed too quickly if the service is partially recovered. Higher success thresholds provide more confidence but delay returning to normal operation. The success threshold should be more conservative than the failure threshold—it's acceptable to require stronger evidence of recovery than of failure. Some implementations use different test request counts based on the service's history.

## Timeout Duration in Open State
The timeout duration determines how long a circuit remains open before transitioning to half-open to allow recovery testing. This duration should be long enough for the failing service to recover but short enough to detect recovery promptly. Recommended values typically range from seconds to minutes depending on the service's expected recovery time. Too short a timeout causes frequent oscillation between open and half-open; too long a timeout prolongs the outage. Some implementations use exponential backoff, increasing the timeout on repeated failures. The timeout should be configurable per-service and adjustable based on operational experience.

## Error Budget Integration
Error budgets represent the acceptable number of errors within a time period, and circuit breakers help preserve error budgets by preventing cascading failures. When a service approaches its error budget limit, circuit breakers can proactively trip to prevent further error accumulation from a failing dependency. This approach differs from threshold-based triggering by considering the service's overall reliability goals. Implementing error budget-aware circuit breakers requires tracking both the local service's error budget and understanding dependency error budgets. This integration helps prevent one service's failure from consuming the entire error budget of dependent services.

## Bulkhead Pattern Coordination
Bulkheads isolate resources (threads, connections, memory) to prevent a single failing dependency from consuming all resources. Circuit breakers complement bulkheads by preventing requests from reaching the bulkhead's resource pool when the dependency is unhealthy. When combined, circuit breakers fail fast before the bulkhead queue fills up, and bulkheads contain resource exhaustion from slow failures that circuit breakers haven't yet detected. Typical implementation: configure separate thread pools or connection pools per downstream service, then apply circuit breakers around those pools. This layered approach provides defense-in-depth against cascading failures.

## Fallback Strategy Selection
Fallback strategies define what happens when a circuit is open or a request fails. Options include: returning cached data, using a default value, queuing the request for later processing, or returning a degraded response. The best fallback depends on the use case—an e-commerce site might serve stale product prices, while a payment system might queue transactions. Fallbacks should be fast and reliable; a slow or failing fallback defeats the purpose of the circuit breaker. Some services implement multiple fallback layers, trying cached data first, then defaults, then queuing. Not all failures warrant fallback—some require immediate error propagation to the caller.

## Retry vs Circuit Breaker Trade-offs
Retries involve resending failed requests immediately or after brief delays, while circuit breakers prevent sending requests during known failures. Retries can succeed if the failure is transient (brief network glitch, temporary overload), but excessive retries on permanent failures waste resources and increase latency. Circuit breakers are more efficient for sustained failures but may incorrectly assume recovery isn't happening. Combined approach: use retries for transient failures (typically 1-3 quick retries), then fall back to circuit breaker logic. Retry configuration (count, delay strategy) and circuit breaker thresholds should be tuned together to avoid conflicts. Exponential backoff with jitter in retries reduces thundering herd effects.

## Timeout vs Circuit Breaker Distinction
Timeouts terminate individual requests that exceed a duration threshold, while circuit breakers prevent entire categories of requests based on aggregate failure patterns. A timeout protects against slow services but doesn't prevent repeated attempts to a completely unreachable service. Circuit breakers protect against cascading failures across many requests but may trip on transient issues if thresholds are too sensitive. Both mechanisms should be used together: timeouts catch slow responses within normal operation, circuit breakers handle systemic failures. Timeout duration should be shorter than circuit breaker recovery attempts—if a request times out, it should contribute to the circuit breaker's failure count.

## Synchronous vs Asynchronous Triggering
Synchronous circuit breaker triggering evaluates thresholds on each request, adding latency proportional to threshold logic complexity. Asynchronous triggering uses background threads or event streams to monitor metrics and update circuit state independently of request flow. Synchronous approaches are simpler but incur per-request overhead; asynchronous approaches reduce per-request cost but require additional infrastructure. Hybrid approaches exist: track failures synchronously with fast counters, evaluate thresholds asynchronously. The choice depends on request volume, acceptable latency, and operational complexity constraints. For high-throughput systems, asynchronous evaluation may be necessary to avoid introducing measurable latency.

## Metrics and Monitoring Integration
Effective circuit breakers expose metrics including: transition events (closed→open→half-open), failure counts and rates, success rates in half-open, and time spent in each state. These metrics enable detection of problematic patterns, like a service oscillating between open and half-open. Monitoring should alert operators to circuit breaker state changes, as each transition indicates a potential problem. Key metrics to track: mean time to detection (time from failure start to circuit opening), mean time to recovery (time from recovery start to circuit closing), and false positive rate. Dashboards should clearly indicate which circuits are open and which services are affected.

## Per-Service vs Global Circuit Breakers
Per-service circuit breakers maintain separate state for each downstream dependency, enabling fine-grained control—one failing service doesn't affect requests to other services. Global circuit breakers apply to entire classes of requests (e.g., all database calls), providing simpler configuration but less precision. Most distributed systems use per-service circuit breakers to prevent correlated failures. However, some systems implement global circuit breakers at higher levels to provide defense-in-depth. Choosing between them depends on whether failures are typically isolated to specific services or affect broader categories. Many implementations use both: per-service circuit breakers for direct dependencies and global circuit breakers for resource protection.

## State Persistence and Recovery
Circuit breaker state (closed/open/half-open) is typically held in memory and lost on process restart. Persistent state allows circuit breakers to remain open across restarts, preventing immediate reconnection to still-failing services. However, persistence introduces operational complexity and requires careful handling of stale state after recovery. Some systems use distributed coordination (etcd, Consul) to share circuit state across instances, preventing the thundering herd when all instances attempt recovery simultaneously. More commonly, state is local to each instance, and each instance independently learns that a service is failing. For stateless services, eventual consistency of circuit state is acceptable since independent detection provides similar protection.

## Cascading Failure Prevention
Circuit breakers prevent cascading failures by stopping the propagation of errors upstream. When service A depends on service B, and B fails, A's circuit breaker to B opens, allowing A to degrade gracefully rather than exhausting its own resources trying to reach B. If service A is depended on by service C, C's circuit breaker to A prevents C from being affected by B's failure. This layered protection requires circuit breakers at multiple levels of the architecture. Cascading failure prevention is most effective when combined with fallback strategies—otherwise, the failure simply propagates in different form.

## Jitter and Thundering Herd
When multiple service instances detect a dependency failure simultaneously, they may all transition to open state at once. When the timeout expires, they all attempt recovery in half-open state, creating a thundering herd that can overwhelm the recovering service. Jitter—randomizing timeout duration slightly per instance—spreads recovery attempts over time. Typical jitter adds 0-20% variance to base timeout duration. Some implementations use exponential backoff with jitter, increasing timeouts between repeated transitions to open. Another approach: limit the number of instances that can be in half-open state simultaneously, queuing recovery attempts. Thundering herd effects are particularly problematic in microservice architectures with many instances.

## Multi-Level Circuit Breaker Hierarchies
Complex systems may implement circuit breakers at multiple levels: per-dependency, per-domain, and per-datacenter. A request might check multiple circuit breakers before being allowed to proceed. Multi-level hierarchies provide fine-grained control but increase complexity. Lower-level circuit breakers detect failures first; higher-level circuit breakers prevent upper services from being overwhelmed. Hierarchies should be designed to fail gracefully—if a lower-level circuit is open, the higher-level circuit should not immediately open unless it detects its own failures. Testing multi-level hierarchies requires careful scenario planning to ensure dependencies don't create deadlocks.

## Adaptive Threshold Tuning
Static thresholds may be suboptimal as service characteristics change over time. Adaptive thresholds adjust based on historical performance: increasing thresholds during high-error periods (allowing more failures before opening), decreasing during stable periods (being more protective). Adaptive approaches require careful implementation to avoid oscillation or overreacting to temporary variations. Simple adaptive strategies include: if the circuit opens frequently, increase the threshold; if it rarely opens, decrease the threshold. More sophisticated approaches use machine learning to predict optimal thresholds based on time of day, service version, or other contextual factors. Adaptive tuning requires historical data collection and offline analysis before deploying changes.

## Testing Circuit Breaker Behavior
Testing circuit breakers requires scenarios including: single failures, sustained failures, intermittent failures, slow responses, and concurrent requests. Chaos engineering tools can inject failures into specific dependencies while observing overall system behavior. Testing should verify: correct state transitions, proper fallback execution, metric accuracy, and recovery behavior. Load testing should include scenarios where multiple services fail simultaneously. Testing should also verify that circuit breaker configuration is correct for each dependency. Automated tests should run regularly; manual chaos experiments should be performed quarterly or when architecture changes significantly.

## Recovery Patterns and Strategies
Service recovery after circuit breaker opening can follow several patterns: graceful degradation (reduced functionality with fallbacks), eventual consistency (queuing updates for later processing), or graceful shutdown (rejecting new requests while draining existing ones). Optimal recovery pattern depends on service type—real-time services might use degradation, batch services might queue work, and front-end services might display error messages. Circuit breakers should coordinate with service-level recovery mechanisms, gradually increasing traffic as the service proves stability. Some implementations implement health checks independently from circuit breakers, allowing services to declare readiness before accepting normal traffic.

## Documentation and Runbook Requirements
Effective circuit breaker deployment requires documentation of: configured thresholds for each dependency, expected behavior during failures, monitoring and alerting setup, runbook procedures for manual intervention, and escalation paths. Runbooks should include steps for: investigating why a circuit opened, determining if manual recovery is needed, and verifying recovery before closing the circuit. Documentation should explain circuit breaker trade-offs to stakeholders unfamiliar with distributed systems reliability. Alert runbooks should be specific—not just "Circuit breaker X is open" but "Circuit breaker X is open, check status of service Y at dashboard link Z." Documentation should be version-controlled and updated whenever thresholds or recovery procedures change.

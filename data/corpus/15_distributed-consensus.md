# Distributed Consensus

## Overview of Distributed Consensus

Distributed consensus is the process by which a group of independent nodes reach agreement on a single value or sequence of values, despite potential failures, network delays, and message loss. Consensus mechanisms form the foundation of fault-tolerant distributed systems by ensuring all non-faulty nodes eventually agree on the same state. The consensus problem is fundamental to building reliable systems because it enables nodes to coordinate decisions without requiring a centralized authority. Consensus algorithms trade off availability, consistency, and partition tolerance according to system requirements. Common use cases include state machine replication, distributed locking, leader election, and maintaining distributed ledgers.

## Paxos: Basic Algorithm

Paxos is a family of consensus algorithms designed to handle asynchronous networks where nodes may fail and messages may be lost or reordered. The algorithm operates in two main phases: prepare and accept. In the prepare phase, a proposer sends a proposal number to acceptors; acceptors respond if they haven't seen a higher proposal number. In the accept phase, the proposer sends the proposed value with the highest proposal number, and acceptors accept it if they haven't promised to ignore it. Paxos guarantees safety (only one value is chosen) and liveness (a value will eventually be chosen if a proposer keeps trying). Paxos is notoriously complex to understand and implement correctly, which has made it less popular than newer alternatives like Raft for production systems.

## Raft Consensus Algorithm

Raft is a consensus algorithm designed to be more understandable than Paxos while maintaining equivalent fault tolerance. It divides the problem into three subproblems: leader election, log replication, and safety. Nodes operate in discrete terms and maintain a replicated log of commands. The leader replicates entries to followers and coordinates commits. If the leader fails, followers elect a new leader. Raft handles network partitions by preventing the minority partition from committing new entries. The algorithm is easier to reason about than Paxos because the state space is more constrained. Raft is widely used in production systems including etcd, Consul, and CockroachDB. A single Raft cluster typically consists of 3, 5, or 7 nodes to tolerate 1, 2, or 3 failures respectively.

## Leader Election Fundamentals

Leader election is the process by which nodes in a distributed system agree on a single node to coordinate operations. In consensus-based systems, the leader is responsible for proposing new values, replicating state, and coordinating commits. Leader election must satisfy safety (at most one leader is elected per term) and liveness (a new leader will be elected if the current leader fails). Election algorithms use timeouts to detect leader failure; a follower that doesn't receive heartbeats from the leader starts an election. Most algorithms require majority quorum agreement to elect a new leader, preventing split-brain scenarios. Leader election is not instantaneous; there is a period of unavailability while a new leader is being elected. The cost of leader election includes network round trips and the time required for majority agreement.

## Raft Leader Election Process

In Raft, nodes in follower state increment their current term and transition to candidate state to initiate an election. A candidate sends RequestVote RPCs to all other nodes. Voters grant votes to the first candidate that requests their vote in a given term, provided the candidate's log is at least as up-to-date as the voter's log. A candidate wins the election when it receives votes from a majority of nodes and becomes leader. If an election times out without a winner, a new election begins with an incremented term. Raft uses randomized election timeouts to reduce conflicts when multiple candidates run simultaneously. The election timeout typically ranges from 150ms to 300ms, with the broadcast time much smaller. Leader election in Raft is guaranteed to eventually succeed as long as the cluster remains reachable.

## Quorum-Based Reads and Writes

A quorum is a subset of nodes whose agreement is sufficient to make a decision. In consensus systems, requiring quorum agreement for both reads and writes ensures consistency across the cluster. For a cluster of N nodes, a typical quorum size is ⌈N/2⌉ + 1. Writing to a quorum guarantees that any subsequent read will see the written value because quorums overlap. Reading from a quorum ensures observing the most recent write. Quorum-based systems tolerate up to ⌊(N-1)/2⌋ node failures. The tradeoff is that every operation must contact a quorum of nodes, increasing latency compared to single-node operations. Some systems optimize reads by allowing reads from the leader without waiting for quorum, accepting potential staleness. Quorum sizes can be tuned; smaller quorums improve latency but reduce fault tolerance.

## Leader-Based Reads

Leader-based reads allow clients to read directly from the leader without quorum agreement. This approach reduces latency because a read operation completes in a single round trip to the leader. However, it requires the leader to be up-to-date and to confirm it is still the current leader. A leader can serve reads safely only after confirming that it has written an entry in its current term and the entry has been committed. This ensures the leader knows about any writes that have been acknowledged to clients. In Raft, the leader can serve a read after it has replicated an entry in the current term. If the leader becomes partitioned and another leader is elected, the old leader will not serve reads after learning about the new term. Leader-based reads are common in production systems because the latency improvement justifies the complexity.

## Quorum Reads in Distributed Systems

Quorum reads require contacting a majority of nodes and using the value with the highest version or timestamp. Quorum reads provide stronger consistency guarantees than leader-based reads because they are resilient to leader failures and network partitions. A quorum read can proceed even if the current leader is partitioned, as long as a quorum of nodes is reachable. The tradeoff is higher latency; quorum reads require communicating with multiple nodes. Some systems implement optimized quorum reads by reading from the closest nodes, but this requires ensuring version consistency across the cluster. Quorum reads are essential in leaderless systems like DynamoDB where there is no designated leader. In leader-based systems, quorum reads are used when strong consistency is required despite the latency cost.

## Split-Brain Problem in Distributed Systems

Split-brain occurs when a network partition causes a cluster to partition into two or more independent subsets, each believing it is the healthy cluster. If both partitions accept writes and become leaders, data consistency is violated because writes to each partition cannot be merged. A classical example is two database nodes in different data centers that lose network connectivity; both may serve as primary and accept writes, creating conflicting states. Consensus algorithms prevent split-brain by requiring quorum majority agreement for leadership and writes. A partition with minority nodes cannot achieve quorum and must stop accepting writes. This ensures that only one partition can make progress, maintaining consistency at the cost of availability in the minority partition.

## Fencing Tokens and Leases

Fencing tokens are unique, monotonically increasing identifiers issued to a leader by an external system to prevent a former leader from performing operations after being deposed. When a new leader is elected, it receives a higher fencing token. If an old leader sends a request with an old fencing token, recipients reject it because the token is lower than the current leader's token. This approach handles the case where the old leader is live but partitioned from the cluster and unaware it has been deposed. Fencing tokens must be issued by an external source of authority with higher availability than the cluster itself. Leases are time-based alternatives where the leader holds a time-bounded lease; if the lease expires without renewal, the leader must stop. Fencing tokens and leases prevent split-brain despite network partitions, but require additional infrastructure.

## Raft Log Replication Mechanism

In Raft, the leader maintains a log of state machine commands and replicates entries to followers via AppendEntries RPCs. Each log entry contains a term number, index, and command. The leader persists entries to its local log before sending them to followers. Followers append entries to their logs after validating the term and previous index match. The leader tracks the highest index replicated on each follower. Once an entry is replicated to a quorum, the leader commits it by advancing the commit index and applying it to the state machine. Followers learn the commit index from the leader and apply committed entries. Log replication ensures that all replicas eventually apply the same sequence of commands in the same order. This approach is robust to follower failures because lost entries are re-replicated after the follower recovers.

## Raft Safety Guarantees

Raft provides strong safety guarantees: election safety (at most one leader per term), leader append-only property (leaders never overwrite or delete entries), log matching property (if two logs have the same index and term, they are identical), leader completeness (if an entry is committed, future leaders will have that entry), and state machine safety (machines apply the same entries in the same order). These guarantees prevent data loss and consistency violations. Election safety is ensured by requiring majority agreement to elect leaders and restricting votes to one per term. Log matching property is maintained by sending the previous entry's index and term in AppendEntries RPCs; followers reject mismatches. Leader completeness ensures the leader's log contains all committed entries, preventing committed data from being lost when the leader changes.

## Paxos Multi-Paxos and State Machine Replication

Single-Paxos decides a single value; Multi-Paxos extends this to agree on a sequence of values for state machine replication. In Multi-Paxos, a leader is elected and proposes a sequence of commands. Followers replicate these commands to their logs. Once a command is chosen, all replicas apply it to their state machines in order. Multi-Paxos optimizes the prepare phase by having the leader renew its leadership periodically rather than running prepare for every command. This reduces the overhead compared to running Single-Paxos for each command independently. Multi-Paxos has similar properties to Raft but is more difficult to implement and understand.

## Term Numbers and Epochs in Consensus

Term numbers (in Raft) or epochs (in other systems) are logical timestamps that increase monotonically as leadership changes. Each term has at most one leader. Nodes track the current term and reject messages from older terms. Term numbers allow nodes to detect when the cluster has moved forward without them and to detect stale leaders. When a node receives a message with a higher term number, it updates its term and transitions to follower state. Term numbers are persisted to disk to maintain this invariant across restarts. This mechanism ensures that old leaders cannot cause harm because they will learn they are stale and step down. Term numbers partition time logically so that safety properties can be reasoned about within each term.

## Commitment Rules and State Machine Application

A log entry is committed when it has been replicated to a quorum of nodes and the leader has written an entry in the current term. The commit point advances monotonically as more entries become replicated. Followers learn the commit index from AppendEntries RPCs and apply committed entries to their state machines. A critical safety rule in Raft is that leaders cannot commit entries from previous terms directly; they must replicate an entry in the current term first. This ensures that if a new leader is elected, it is aware of all committed entries. Committed entries are guaranteed to survive leader failures and will be applied by future leaders and followers. The state machine must be deterministic so that replicas apply the same effects from the same log entries.

## Linearizability and Consensus

Linearizability is a consistency model where all operations appear to execute at a single point in time in the order observed by clients. Consensus algorithms enable linearizable state machine replication by ensuring all replicas execute the same commands in the same order. A read operation is linearizable if it observes the effects of all writes that completed before the read began. Providing linearizable reads requires either reading from the leader (after confirming leadership) or reading from a quorum. Some systems weaken consistency to improve latency; eventual consistency requires no quorum reads, while causal consistency requires smaller quorums. Linearizability is important for correctness in many applications but comes at a latency cost compared to weaker consistency models.

## Handling Network Partitions

Consensus algorithms handle network partitions by having the majority partition continue operating while the minority partition stops accepting writes. In Raft, a follower in the minority partition cannot achieve quorum votes for itself to become leader, so it remains a follower. The minority partition can serve reads if the application allows stale data. The majority partition elects a new leader if the old leader is in the minority partition and continues making progress. Once the partition heals, the minority partition's log is rolled back to match the majority partition's log, and the partition rejoins. Partitions are inevitable in real networks; consensus algorithms make partitions safe by trading availability for consistency.

## Candidate State and Deadlock Prevention

In Raft, a candidate is a node that is attempting to become leader by requesting votes. A node transitions to candidate state when its election timeout expires and it hasn't received a heartbeat. Multiple nodes may become candidates simultaneously if elections are triggered at the same time. To prevent deadlock from multiple simultaneous elections, Raft uses randomized election timeouts; each follower sets a timeout randomly between T and 2T. This makes it unlikely that multiple nodes become candidates at the same time. If a split-vote occurs, elections time out and new elections begin with higher term numbers. Eventually, one candidate receives a quorum vote and becomes leader. Randomized timeouts are essential for liveness in systems with many nodes.

## Snapshot and Log Compaction in Consensus Systems

Log entries accumulate indefinitely unless compacted. Snapshots capture the state machine state at a specific log index, allowing old log entries to be discarded. When a node creates a snapshot, it writes the state machine state and the index/term of the last included entry. Followers that lag too far behind can be caught up by receiving a snapshot rather than replaying all log entries. Raft defines InstallSnapshot RPC for this purpose. Snapshots must be durable and atomic to maintain safety. The tradeoff is that snapshots are expensive to create (require pausing the state machine) but dramatically reduce disk usage and recovery time. Typical snapshot intervals range from minutes to hours depending on workload and log growth rates.

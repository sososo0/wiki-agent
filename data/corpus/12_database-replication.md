# Database Replication

## Leader-Follower Architecture Basics
Leader-follower replication is a replication pattern where one database instance (the leader) accepts all writes, and one or more follower instances replicate data from the leader. The leader maintains a replication log of all write operations, which followers consume and apply to their local state. This ensures eventual consistency across replicas. Leaders handle all mutations while followers can serve read-only traffic. This architecture is simple to implement and reason about compared to peer-to-peer replication schemes. The primary tradeoff is that the leader becomes a bottleneck for write throughput and represents a single point of failure for write availability, though follower promotion can mitigate this in failover scenarios.

## Synchronous Replication
In synchronous replication, the leader waits for explicit acknowledgment from followers before confirming a write to the client. The leader blocks the write operation until one or more followers have durably persisted the data. This provides strong consistency guarantees—clients can immediately read their writes from any replica. The major tradeoff is latency: write latency increases to account for network round-trips to remote replicas. If any follower becomes unavailable or slow, it can stall all writes on the leader, reducing system availability. Synchronous replication is typically used in scenarios where data consistency is critical and write availability loss is acceptable, such as financial transaction systems.

## Asynchronous Replication
Asynchronous replication allows the leader to acknowledge writes to clients immediately without waiting for followers to replicate the data. The leader sends replication logs to followers, which consume and apply them at their own pace. This minimizes write latency and decouples leader performance from follower performance. The tradeoff is that followers may lag behind the leader, and unacknowledged writes can be lost if the leader crashes before replicating them to any follower. Asynchronous replication is the default in many systems due to its performance benefits and is suitable when eventual consistency is acceptable. It increases operational complexity for handling failover scenarios since lost writes must be managed explicitly.

## Semi-Synchronous Replication
Semi-synchronous replication is a hybrid approach where the leader waits for acknowledgment from at least one follower (or sometimes a quorum) before confirming writes, rather than waiting for all followers. This balances the consistency guarantees of fully synchronous replication with better availability and latency than requiring all replicas. If the designated synchronous follower becomes unavailable, the system can automatically switch to asynchronous replication for that write or promote a new synchronous replica. This approach is used in databases like MySQL with semi-sync plugins and provides reasonable protection against data loss while avoiding the latency penalty of true synchronous replication across many replicas.

## Replication Lag Definition and Measurement
Replication lag is the delay between when data is written to the leader and when that data becomes visible on a follower. It's measured in time units (milliseconds, seconds) or in the number of unprocessed events in the follower's replication queue. Causes include network latency, follower processing capacity constraints, and lock contention on the follower. Replication lag is a critical metric to monitor in systems using asynchronous or semi-synchronous replication. Most monitoring systems track lag by comparing timestamps or sequence numbers between leader and follower state. Understanding lag is essential for operations teams to set appropriate SLOs and for application engineers to decide when to use followers for reads. Very high lag can lead to applications reading stale data.

## Replication Lag and Read Consistency
When clients read from followers during replication lag, they may observe stale data compared to the leader. This violates read-after-write consistency if a client writes to the leader and immediately reads from a follower. Applications must account for this by either reading critical data from the leader, implementing client-side staleness detection, or using mechanisms like read repair. Some systems expose replication lag metrics to applications so they can make routing decisions. For example, an e-commerce system might read inventory from the leader if stock levels are critical but read user profile data from a follower. The architectural decision depends on application consistency requirements and acceptable latency tradeoffs.

## Read Replicas for Load Distribution
Read replicas are follower instances specifically deployed to serve read-only traffic, distributing the read load away from the leader. This allows systems to scale read throughput independently from write capacity. Read replicas receive updates from the leader via replication and can be geographically distributed to reduce read latency for users in different regions. The tradeoff is that reads may observe stale data depending on replication lag, and operating additional replicas increases infrastructure costs and operational complexity. Read replicas are cost-effective when read traffic significantly exceeds write traffic. Common use cases include analytics queries, reporting, and search indexes that can tolerate slightly stale data.

## Geographical Distribution of Replicas
Deploying replicas across geographic regions provides resilience against regional outages and reduces read latency for users accessing nearby replicas. Each region maintains a follower that replicates from the leader in another region. The leader typically resides in a primary region, and followers in secondary regions serve regional read traffic. Replication lag increases with geographic distance due to network latency, so applications must account for stale reads. This architecture also complicates failover: promoting a geographically distant replica to leader increases write latency for all users. Typical deployments use synchronous or semi-synchronous replication to the nearest replica and asynchronous to distant ones.

## Multi-Leader Replication Overview
Multi-leader replication allows multiple database instances to accept writes, each acting as a leader to the others. Changes from each leader are replicated to all other leaders. This provides better write availability and performance since clients can write to a nearby leader rather than always routing to a single leader. Multi-leader is common in geographically distributed systems where each region has a leader. The significant tradeoff is handling write conflicts when multiple leaders receive concurrent writes to the same data. Conflict resolution becomes complex and may require application-level logic. Multi-leader replication increases operational overhead but is necessary for systems requiring high write availability across regions or supporting offline-first applications.

## Write Conflicts in Multi-Leader Systems
When multiple leaders accept writes concurrently, conflicts arise when different leaders receive writes to the same data. For example, two leaders might receive updates to the same record with different values. Unlike single-leader systems where ordering is deterministic, multi-leader systems must define a conflict resolution strategy. Common approaches include last-write-wins (comparing timestamps or sequence numbers), version vectors to detect concurrent writes, or application-defined custom resolution logic. Some systems track all conflicting versions and force application-level resolution. Conflicts are a fundamental challenge in multi-leader replication; ignoring or naively resolving them can silently corrupt data. Operational overhead is significantly higher than single-leader architectures.

## Conflict Resolution Strategies
Last-write-wins (LWW) is a deterministic approach where the write with the largest timestamp or highest sequence number overwrites others. It's simple to implement but can silently discard data if clocks are skewed or if the "latest" write is not semantically correct. Version vectors track causal relationships between writes, allowing systems to distinguish concurrent writes from causally ordered ones. Concurrent writes must be resolved explicitly or stored as conflicts. Custom application logic can be used for domain-aware resolution, such as choosing the larger inventory value. Multi-value resolution stores all conflicting versions and forces application code to choose one. Each strategy has different consistency and operational tradeoffs; LWW is simplest but least safe, while custom logic is safest but most complex.

## Leaderless Replication Systems
Leaderless replication, used in systems like Dynamo and Cassandra, distributes writes to multiple replicas simultaneously with no designated leader. Clients send reads and writes to multiple replicas and use quorum mechanisms to ensure consistency. A write is confirmed once a quorum (e.g., majority) of replicas acknowledge it; reads query multiple replicas and return the most recent value. This eliminates single leader bottlenecks and provides fault tolerance: the system continues operating even if individual replicas fail. The tradeoff is increased complexity in read and write paths, higher latency to contact multiple replicas, and requirements for clock synchronization. Leaderless replication is used in highly available, geographically distributed systems where write availability is critical.

## Quorum-Based Consistency
Quorum-based consistency uses read and write quorums to provide tunable consistency guarantees without a leader. If R is the read quorum size and W is the write quorum size, and both are chosen such that R + W > N (where N is total replicas), then every read will see at least one replica that has the most recent write. Larger quorums provide stronger consistency but increase latency and reduce availability if nodes fail. Smaller quorums improve performance but increase the chance of reading stale data. For example, with 5 replicas, setting R=W=3 ensures strong consistency with tolerance for 2 failures. Quorum reads and writes are fundamental to leaderless replication but introduce latency overhead since multiple replicas must be contacted.

## Replication Log Formats
Replication logs record all writes so followers can replay them. Statement-based logging records the original SQL statement, requiring followers to re-execute it; this has smaller log size but can produce different results if execution is non-deterministic. Write-ahead logs (WAL) record low-level changes to the storage engine; they're more reliable but larger and engine-specific, making cross-engine replication difficult. Row-based logging records changes to individual rows as structured data; it's verbose but deterministic and enables flexible replication. Logical logs use a format independent of the storage engine. Log format choice impacts replication reliability, lag, and compatibility across different versions or database engines. Most modern systems use row-based or logical logs for determinism.

## Replication Lag Mitigation Techniques
Applications can mitigate replication lag effects through several techniques. Read-your-writes consistency can be achieved by routing writes and subsequent reads to the leader, or by tracking write timestamps on the client and waiting for followers to catch up. Monotonic reads ensure clients always see forward progress by routing to the same replica or ensuring their replica has seen all prior operations. Consistent prefix reads ensure causally dependent operations appear in order, important for transactions spanning multiple records. Session-level consistency guarantees can be provided by storing session state that tracks which replicas have which operations. These techniques typically require application involvement rather than being purely infrastructure features, increasing implementation complexity.

## Replication and Transaction Semantics
Replication interacts complexly with transactions, especially distributed ones. A transaction committed on the leader may not have been replicated to followers, creating read-after-write inconsistency if the leader fails. Multi-statement transactions spanning multiple records complicate replication since partial progress may be visible on some followers but not others. Two-phase commit across leader and synchronous followers can ensure atomicity but requires careful handling of failure scenarios. Systems like MySQL Group Replication use consensus protocols to ensure transaction consistency across replicas. Application code must understand whether transactions are committed to the leader only or must be durable on replicas, affecting consistency vs. latency tradeoffs.

## Handling Replication Failures
Replication failures occur when followers fall significantly behind, crash, or become unreachable. The leader must buffer replication log until followers catch up; if the log buffer fills, replication typically stalls or new followers cannot be added. Failed followers must reconnect and either resume from their last checkpoint or perform full resynchronization, which is expensive for large datasets. Detecting replication failures requires monitoring lag and connectivity; threshold-based alerting notifies operators of degraded replication. Some systems automatically promote healthy followers to replace failed ones or prevent stale followers from being promoted to leader accidentally. Replication failure scenarios must be tested and procedures documented, as they're critical operational events that can lead to data loss or inconsistency.

## Promoting a Follower to Leader
Failover involves promoting a follower to leader when the current leader fails. Promotion requires ensuring the new leader has all committed data from the old leader; this is safe with synchronous replication but risky with asynchronous replication. Unacknowledged writes on the failed leader may be lost. Automated promotion reduces downtime but risks promoting stale followers or causing split-brain scenarios where multiple nodes believe themselves leader. Manual promotion is slower but allows operators to verify follower state. DNS or service discovery must be updated so clients route to the new leader. The promoted follower's replication clients (other followers) must point to the new leader. Promotion procedures should be tested regularly to ensure they work reliably under failure conditions.

## Split-Brain and Consensus Mechanisms
Split-brain occurs when the leader becomes unreachable and a follower is promoted without confirming the old leader is truly down, resulting in two nodes accepting writes. This causes unreconcilable divergence and data corruption. Consensus mechanisms like Raft or Paxos prevent split-brain by requiring quorum agreement before any node can become leader. A candidate leader must receive acknowledgment from a majority of nodes, ensuring only one leader can be elected at a time. This adds complexity and latency but provides safety guarantees. Systems using consensus for replication require at least 3 nodes for meaningful fault tolerance (surviving one failure). Consensus is a strong alternative to manual failover for critical systems but introduces dependencies on quorum availability.

## Monitoring and Observability of Replication
Effective replication monitoring tracks replication lag, error rates, and follower connectivity. Lag should be measured on multiple followers and aggregated to understand system-wide replication health. Alerts should trigger when lag exceeds acceptable thresholds for application SLOs. Replication errors (network failures, conflicts, out-of-disk space) must be captured and made visible to operators. Transaction throughput on leaders vs. followers helps identify bottlenecks. Distributed tracing can show client impact of replication failures. Dashboards should display replication topology, showing relationships between leaders and followers. Testing replication failure scenarios regularly validates monitoring and alerting systems catch real problems. Observability infrastructure for replication is often underfunded but critical for reliable operation at scale.

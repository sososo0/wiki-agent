# Backpressure Mechanisms

## Reactive Streams Specification
The Reactive Streams specification defines a standard for asynchronous stream processing with non-blocking backpressure. It establishes contracts between Publishers and Subscribers where demand is signaled explicitly through request(n) calls. Subscribers indicate capacity to receive n elements, allowing publishers to throttle emission rates accordingly. This prevents unbounded buffering and memory exhaustion in asynchronous pipelines. Implementations include RxJava, Project Reactor, and Akka Streams. Use when building observable chains or stream-based architectures where you need language-agnostic backpressure semantics. The tradeoff is additional protocol overhead compared to synchronous blocking; however, it enables efficient resource utilization in I/O-bound systems without thread-per-request models.

## Bounded Queue Pattern
Bounded queues enforce capacity limits on work items in transit between system components. When a queue reaches its configured maximum size, producers either block, receive rejection signals, or time out depending on implementation. This prevents unbounded memory growth and establishes clear resource ownership. Common in thread-pool executors, message brokers, and job queues. Use bounded queues as a foundational backpressure mechanism in any multi-stage pipeline. Tradeoffs include potential producer blocking (which can cascade), complexity in choosing appropriate queue sizes, and the risk of head-of-line blocking if queue consumers stall. Configuration requires understanding both producer rates and consumer throughput.

## Request Rate Limiting
Request rate limiting restricts the number of operations accepted within a time window using algorithms like token bucket or sliding window. Limits can be per-user, per-IP, per-service, or global. When limits are exceeded, requests are rejected with HTTP 429 or equivalent status codes. This protects backend systems from overload and ensures fair resource allocation. Implement at API gateways, service boundaries, or application entry points. Tradeoffs include potential user experience degradation, complexity in distributed rate limiting scenarios (clock skew, coordination overhead), and the challenge of setting appropriate limits that balance availability and protection. Limits that are too aggressive trigger false rejections; too lenient provides insufficient protection.

## Token Bucket Algorithm
Token bucket is a rate-limiting algorithm where tokens accumulate at a fixed rate into a bucket with maximum capacity. Each operation consumes tokens; operations are rejected if insufficient tokens remain. This allows burst traffic up to bucket capacity while maintaining average rate limits. Tokens can represent requests, bytes, or work units depending on context. Use for protecting APIs, database connections, or resource quotas. The algorithm is simple to implement and works well in single-process scenarios. Distributed token bucket implementations require shared state (Redis, ZooKeeper) and coordination, introducing latency and consistency challenges. Bucket size controls burst tolerance; refill rate controls sustainable throughput.

## Sliding Window Rate Limiting
Sliding window rate limiting counts operations within a moving time window and rejects requests that exceed configured thresholds. Unlike token bucket, it maintains precise per-second or per-minute counts. Implementation typically uses sorted sets or queues to track timestamps of recent operations. More accurate than token bucket for strict rate enforcement but requires more memory to maintain request history. Use when exact rate compliance is required, such as billing-critical systems or SLA enforcement. Tradeoffs include higher memory consumption, increased latency checking against historical data, and complexity in distributed scenarios requiring clock synchronization. Window size choice affects granularity; smaller windows detect spikes faster but consume more state.

## Leaky Bucket Algorithm
Leaky bucket models a fixed-capacity container where requests fill the bucket and drain at constant rate. When full, new requests overflow and are discarded or delayed. This provides smooth, predictable traffic shaping and prevents burst traffic from overwhelming downstream systems. Differs from token bucket: it enforces average rate strictly without burst capacity. Use for network traffic shaping, database connection pooling, or systems requiring consistent output rates. Tradeoffs include inability to handle legitimate bursts, potential unfairness if drain rate is too low (requests discarded while capacity exists), and complexity tuning drain rates to match system capacity. Implementation requires timing mechanisms to regulate drain rate.

## Queue Overflow Rejection
Queue overflow rejection drops or refuses incoming work when queues reach capacity, signaling backpressure to producers without blocking them. Rejection can be immediate (fail-fast) or after configured timeout attempts. Implementations use rejection policies like "abort," "discard," or "discard oldest." This prevents memory exhaustion and cascading failures from queue growth. Useful in overload scenarios where dropping non-critical requests is preferable to system failure. Tradeoffs include request loss, potential customer impact, and the requirement to handle rejections gracefully at producers. Without proper monitoring, rejections may be invisible. Requires clear semantics about which requests are safe to drop and whether retry logic should apply.

## Adaptive Timeout Configuration
Adaptive timeout mechanisms adjust request timeouts based on observed latency percentiles, system load, or service health. Rather than static timeouts, the timeout window shrinks when services respond slowly, failing fast to free resources. Implementations track p95, p99, or p99.9 latencies and scale timeouts relative to recent performance. Use when downstream service latency varies significantly or when resource exhaustion manifests as increased latency. Tradeoffs include complexity implementing percentile tracking, risk of timeout cascades if latencies increase system-wide, and difficulty debugging timeout issues when rules change dynamically. Requires careful tuning to avoid timeouts during legitimate slow periods (e.g., backups, deployments).

## Load Shedding Strategies
Load shedding intentionally drops or rejects requests to prevent system overload when capacity is exceeded. Strategies include: reject requests below SLA thresholds, drop non-priority requests, limit concurrency, or shed based on resource utilization. Load shedding differs from rejection in intentionality—it's a deliberate architectural choice rather than a side effect of queue overflow. Use when system load exceeds capacity and partial availability is preferable to total failure. Tradeoffs include determining which requests to shed (fairness concerns), customer communication about shed requests, and potential business impact from dropped work. Without monitoring, shedding may be invisible. Requires clear metrics on what constitutes "overload" and which requests are safe to drop.

## Priority Queue Backpressure
Priority queues admit high-priority requests when capacity is constrained while rejecting low-priority ones. Priority can be determined by request type, user tier, SLA classification, or business value. High-priority requests are processed preferentially, ensuring SLA-critical work completes. Use in systems serving multiple customer classes with different SLAs. Tradeoffs include added complexity in priority assignment and potential starvation of low-priority requests if high-priority traffic remains sustained. Requires monitoring to verify low-priority work isn't indefinitely delayed. Priority decisions may need runtime adjustment as business priorities shift. Implementation overhead is minimal compared to FIFO queues.

## Concurrency Limiting
Concurrency limiting restricts the number of simultaneous operations in flight, rejecting or queueing new requests when limits are reached. Differs from queue-based backpressure: concurrency limits control active work, not queued work. Use for protecting limited resources like database connections, thread pools, or external API quotas. Semaphores or atomic counters typically implement this. Tradeoffs include potential request rejection when concurrency is high, lack of fairness if requests have varying durations, and complexity choosing limits that match actual capacity. Too-low limits reduce throughput; too-high limits risk resource exhaustion. Works well combined with queue mechanisms.

## Quota Management Systems
Quota management allocates and tracks fixed allocations of resources (requests, bytes, compute time) per user, tenant, or service over defined periods. Quotas are decremented as resources are consumed; requests are rejected when quotas are exhausted. Enables fair resource allocation in multi-tenant or shared systems. Use for SaaS platforms, cloud services, or any system where fairness across users is essential. Tradeoffs include state management overhead, complexity distributing quota state across replicas, and the need for quota reset mechanisms aligned with billing periods. Clock skew in distributed systems can cause quota violation. Quotas must be tunable post-deployment to accommodate unexpected usage patterns.

## Backpressure Signaling Protocols
Backpressure signaling protocols explicitly communicate capacity constraints between system components using defined message formats or status codes. Examples include HTTP 429 (Too Many Requests), gRPC flow control headers, or custom protocol messages. Receivers signal they cannot accept more work; senders adjust behavior (retry, queue, or fail). Use when building request-response systems or streaming protocols. Tradeoffs include protocol complexity and requirement that all participants understand backpressure signals. Clients must handle signals gracefully (exponential backoff, queue internally) or failures cascade. Signaling overhead is minimal compared to the value of controlled flow. Monitoring that signals are properly handled is critical.

## Memory-Based Backpressure
Memory-based backpressure triggers when heap usage, RSS memory, or specific memory pool utilization exceeds thresholds. Systems shed load, reduce queue sizes, or reject requests when approaching memory limits. Prevents out-of-memory errors and full GC pauses. Use as a safety mechanism in any system with unbounded queues or caches. Tradeoffs include determining thresholds (GC behavior varies by JVM tuning), lag between high memory usage and response time (GC may occur before backpressure triggers), and potential oscillation if thresholds are too close together. Memory monitoring has overhead; too-frequent checking impacts performance. Works best as last-line defense alongside proactive mechanisms.

## CPU-Based Backpressure
CPU-based backpressure triggers when CPU utilization exceeds configured thresholds, signaling overload and triggering load shedding or request rejection. Monitors system CPU or thread pool metrics; high utilization suggests the system cannot process additional work efficiently. Use to prevent cascade failures where overloaded CPUs cause timeouts and retries. Tradeoffs include lag between high CPU and response (CPU spike might be transient), difficulty distinguishing productive high utilization from wasteful spinning, and platform variance in CPU measurement. CPU thresholds must account for workload characteristics; analytics workloads tolerate higher CPU than latency-sensitive ones. Best combined with response-time-based signals.

## Healthy Hosts / Unhealthy Host Management
Systems track health status of service instances and shed load to unhealthy hosts, distributing backpressure across healthy ones. Health checks can be active (periodic requests) or passive (failure tracking). Unhealthy hosts receive reduced traffic or are excluded entirely until health is restored. Use in load-balanced systems with multiple backend instances. Tradeoffs include complexity implementing distributed health status consensus, potential false positives (transient failures marked unhealthy), and challenges distinguishing "degraded but functional" from fully unhealthy. Health checks add overhead; too-frequent checks consume resources, too-infrequent checks delay detection. Requires monitoring of health decision accuracy.

## Bulkhead Pattern
Bulkhead isolates resources (threads, connections, memory) allocated to different services or request types, preventing one overloaded service from starving others. Each bulkhead has fixed capacity; requests use only resources within their bulkhead. If a bulkhead is exhausted, only that service experiences backpressure, not the entire system. Use in monoliths or shared infrastructure serving multiple services. Tradeoffs include underutilization if one bulkhead has spare capacity while another is exhausted (resources aren't shared), complexity sizing bulkheads appropriately, and monitoring overhead tracking per-bulkhead metrics. Overprovisioning bulkheads wastes resources; underprovisioning defeats isolation. Works well combined with priority queues.

## Graceful Degradation Under Load
Graceful degradation intentionally reduces functionality or quality when systems approach capacity, allowing continued operation at reduced capability. Examples include disabling expensive features, reducing cache refresh rates, returning cached data, or filtering results. Differs from load shedding: requests succeed but with reduced capability rather than being rejected. Use when partial functionality is more valuable than failure. Tradeoffs include complexity implementing degraded modes, testing interactions between normal and degraded paths, and customer communication about capability reduction. Monitoring must distinguish degraded operation from failure. Degradation rules must be pre-configured; runtime decisions are risky.

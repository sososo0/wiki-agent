# Rate Limiting Algorithms

## Token Bucket Overview
The token bucket algorithm maintains a bucket with a fixed capacity that refills at a constant rate. Each request consumes one or more tokens; requests proceed only if sufficient tokens exist. Tokens accumulate up to the bucket's maximum capacity. This algorithm handles bursts well since accumulated tokens allow temporary traffic spikes while maintaining average rate limits. The refill rate determines the sustained throughput, while bucket capacity controls burst allowance. Implementation requires tracking the last refill time and current token count per entity. Token bucket suits APIs needing flexibility for occasional traffic spikes within a defined average rate.

## Leaky Bucket Overview
The leaky bucket algorithm models a bucket that fills with incoming requests and leaks (processes) at a constant rate. Requests enter the bucket; if it's full, they're discarded or rejected. This produces a perfectly smooth outgoing request rate regardless of incoming pattern. The leak rate is fixed and predetermined. Leaky bucket is conceptually simpler than token bucket but less flexible for handling legitimate bursts. It's useful for traffic shaping and smoothing outbound requests to downstream systems. Implementation uses a queue with a background process draining at the leak rate.

## Fixed Window Counter Algorithm
Fixed window counter divides time into fixed-duration windows (e.g., per minute, per hour) and counts requests within each window. When a request arrives, the algorithm checks if the current window's count exceeds the limit. After the window closes, the counter resets. This approach is simple to implement and understand. However, it suffers from the "boundary problem"—a burst of traffic at window boundaries can exceed the intended rate limit. For example, fifty requests in the last second of one minute and fifty in the first second of the next minute would allow 100 requests in two seconds. Fixed window counter works for non-critical rate limiting where boundary spike handling is acceptable.

## Sliding Window Log Algorithm
Sliding window log maintains a timestamp log of all requests within the current time window. When a new request arrives, the algorithm removes timestamps older than the window duration and checks if the request count is under the limit. This approach provides precise rate limiting without boundary problems. However, it consumes memory proportional to request volume, storing every request timestamp. This can become expensive for high-traffic endpoints with strict rate limits. Sliding window log is accurate and fair but impractical for memory-constrained systems or very high request rates. It's best suited for moderate-traffic scenarios where precision is more important than memory efficiency.

## Sliding Window Counter Algorithm
Sliding window counter combines aspects of fixed window and sliding window approaches. It divides time into fixed windows and stores count data for recent windows. When checking limits, it calculates a weighted count based on the overlap between the current time and previous window. The formula interpolates: `current_count + (previous_count * overlap_ratio)`. This reduces memory usage compared to logging every timestamp while avoiding sharp boundary problems. The approach is approximate but practical. Sliding window counter works well for moderate to high-traffic systems where near-perfect accuracy is acceptable and memory efficiency matters.

## Token Bucket Refill Strategies
Token bucket implementations vary in how they refill tokens. The classical approach refills on-demand: when checking tokens, calculate elapsed time since last refill and add tokens accordingly. Alternatively, some systems use a background task periodically adding tokens. On-demand refill is more precise and requires no background process, but adds latency to requests. Periodic refill simplifies the code but may accumulate tokens imprecisely. For distributed systems, on-demand refill is often preferred to avoid clock synchronization issues. Understanding refill strategy matters when choosing token bucket for rate limiting.

## Leaky Bucket Memory Management
Leaky bucket implementations must manage the queue holding pending requests. The queue size directly impacts memory usage. When the queue fills, new requests are dropped. Some implementations allow configurable queue sizes; others use bounded queues by default. The leak rate must be tuned appropriately—too slow causes queues to grow indefinitely, too fast defeats the purpose. In high-concurrency environments, the queue can become a bottleneck. Memory-conscious implementations may drop requests aggressively rather than queue extensively. Understanding memory tradeoffs is essential when deploying leaky bucket in resource-constrained environments.

## Burst Allowance Configuration
Most rate limiting algorithms support burst allowance—temporary traffic increases above the sustained rate. Token bucket's bucket capacity directly controls burst allowance: larger capacity permits larger bursts. Leaky bucket can be modified to allow bursts by increasing leak rate temporarily. Fixed window counter inherently allows bursts at boundaries. Sliding window approaches can be adjusted by expanding the window duration. When configuring burst allowance, consider legitimate use cases: API clients fetching multiple resources, batch processing, or background jobs. Setting burst allowance too high defeats rate limiting; too low frustrates legitimate users. Burst configuration should balance protection against legitimate user needs.

## Per-Tenant Rate Limiting
Multi-tenant systems require isolated rate limits per tenant to prevent one tenant from affecting others. Implementation typically maintains separate rate limit state per tenant identifier. Each tenant has independent token buckets, counters, or logs. The main challenge is scalability—with thousands or millions of tenants, memory and computational overhead grows significantly. Distributed stores (Redis, memcached) often back per-tenant rate limiting in large systems. Access patterns should be considered: evicting inactive tenants' state can reduce memory. Per-tenant limits require careful design to isolate users fairly while maintaining system stability.

## Distributed Rate Limiting with Shared State
Distributed systems with multiple servers need coordination to enforce rate limits consistently. Shared state stores (Redis, memcached, databases) centralize rate limit tracking. Each server queries the shared store before allowing requests. This ensures accurate global limits but introduces latency and creates a potential bottleneck. Redis is commonly used for its speed and atomic operations. Consistency is critical—using eventual consistency can lead to limit bypass. Trade-offs include consistency guarantees versus query latency. Distributed rate limiting adds complexity but is necessary for horizontally scaled systems.

## Distributed Rate Limiting with Local Estimation
Local estimation reduces shared state queries by maintaining approximate counters locally. Servers periodically synchronize with shared state, allowing some drift. This approach reduces load on central stores and latency per request. The tradeoff is reduced accuracy—total rate limit across servers may slightly exceed intended limits. Synchronization frequency controls the accuracy-latency tradeoff. Local estimation works well when slight overages are acceptable. Implementation requires careful synchronization logic and must handle clock skew. This approach suits high-throughput systems where perfect global accuracy is less important than low latency.

## Rate Limit Headers and Client Feedback
Well-designed rate limiting systems communicate limits and remaining capacity to clients via HTTP headers. Common headers include `RateLimit-Limit`, `RateLimit-Remaining`, `RateLimit-Reset`, and `Retry-After`. These headers allow clients to adjust request patterns proactively. When limits are exceeded, returning 429 (Too Many Requests) status with `Retry-After` helps clients coordinate backoff. Clear header semantics reduce client frustration and improve API usability. Headers should reflect the algorithm being used (e.g., remaining tokens for token bucket). Documenting header meanings in API documentation is essential. Comprehensive header support demonstrates respectable rate limiting implementation.

## Adaptive Rate Limiting
Adaptive rate limiting adjusts limits dynamically based on system load, upstream capacity, or observed error rates. When downstream services degrade, limits reduce automatically. During low load, limits may increase. This requires monitoring integration and response thresholds. Adaptive approaches prevent cascading failures by reducing inbound requests when systems struggle. Implementation complexity increases significantly—feedback loops must be designed carefully to avoid oscillation. Adaptive rate limiting is valuable for protecting against resource exhaustion but requires careful tuning and monitoring. Most implementations combine fixed baseline limits with adaptive adjustments.

## Priority-Based Rate Limiting
Some systems need to prioritize certain request types or users. Premium customers might have higher limits; critical operations might bypass limits entirely. Priority-based approaches implement multiple rate limit tiers. Higher priority requests consume shared resources first or have separate limits. Implementation requires identifying priority and enforcing different rules. Kubernetes Quality of Service (QoS) is an example of priority-based resource limiting. Care must be taken to prevent starvation of lower-priority requests. Priority-based rate limiting adds operational complexity and requires clear policies about priority assignment.

## Rate Limiting Warm-Up and Cold Start
When services start or recover from outages, gradual rate limit introduction prevents thundering herd. Cold start rate limiting begins very restrictive and gradually increases toward normal limits. This allows dependent services to scale gradually. Warm-up periods typically last minutes to hours depending on system expectations. Kubernetes features like initial delay periods for rate limit increases exemplify this approach. Implementing warm-up requires timestamp tracking since service start and gradual threshold increases. Without warm-up, simultaneous recovery of multiple systems can cause cascading failures. Warm-up is valuable insurance for production reliability.

## Rate Limiting Quota Systems
Quota systems differ from rate limiting by resetting on calendar boundaries (monthly, daily) rather than sliding windows. A user might have 1000 requests per month rather than 100 per second. Quota tracking requires persistent state and date handling. The primary challenge is fairly distributing quota across time—users shouldn't be forced to use entire quota on day one. Some systems implement daily sub-quotas (e.g., 50 per day for a 1500 monthly quota). Quota systems suit billing-associated limits and longer-term fairness. Implementation typically uses fixed window counters with longer windows, accepting boundary effects as acceptable for weekly/monthly periods.

## Rate Limiting Exception Handling
Rate limiting systems must handle exceptions gracefully. When rate limit state is unavailable (central store down), should requests be allowed or rejected? Conservative systems reject; optimistic systems allow. The choice depends on risk tolerance. Redundancy in rate limit stores (multiple Redis replicas) improves availability. Fallback strategies might use local estimation when distributed state is unavailable. Partial failures require careful handling—some requests succeeding and others failing can confuse clients. Circuit breakers can help: if the rate limit system fails, fallback to less strict local limiting rather than cascading failure. Exception handling philosophy should align with service reliability goals.

## Rate Limiting in Request Pipelines
Rate limiting should be applied at appropriate pipeline stages. Early application (API gateway, edge) reduces unnecessary processing of rejected requests. Late application (application code) allows fine-grained, business-logic-aware limiting. Most systems use multiple layers: coarse API gateway limits and fine-grained application limits. Gateway limits protect infrastructure; application limits protect business logic. Layered rate limiting requires coordination to provide sensible user feedback. Implementing at multiple layers increases operational complexity. The choice should reflect system architecture and which resources need protection.

## Cost-Based Rate Limiting
Not all requests are equal—some are computationally expensive. Cost-based rate limiting assigns costs to operations and enforces limits on total cost rather than request count. A simple read request costs 1 unit; a complex analytical query costs 100 units. Users receive a budget of cost units per period. This aligns rate limits with actual resource consumption. Implementation requires assigning costs to operations and tracking cumulative cost. Cost estimation can be difficult and requires updates when performance characteristics change. Cost-based limiting is valuable for heterogeneous workloads but more complex than simple request counting.

## Rate Limiting Testing and Validation
Rate limiting systems should be thoroughly tested before production. Unit tests verify algorithm behavior: that token bucket refills correctly, that sliding window handles boundaries, etc. Load tests verify distributed systems handle concurrent limit checks without race conditions. Chaos tests verify failure modes: what happens when rate limit store becomes unavailable? Property-based tests can verify that total request volume never exceeds limits across all entities. Monitoring should track rate limit rejections, latency of limit checks, and false positives/negatives. A/B testing can validate limit values against real traffic patterns. Comprehensive testing prevents rate limiting bugs from causing unexpected rejections.

## Rate Limiting Observability and Monitoring
Production rate limiting requires visibility into its behavior. Metrics should track: requests rejected, requests allowed, current token bucket levels, per-tenant limit consumption, and latency of rate limit checks. Dashboards should highlight which endpoints or tenants are hitting limits. Alerts should trigger when rejection rates spike unexpectedly, indicating potential DoS or misconfigured limits. Logs should record rejected requests with reasons for debugging. Heat maps can visualize burst patterns. Rate limiting contributes to latency—P99 latency of rate limit checks should be monitored. Comprehensive observability enables quick identification and resolution of rate limiting issues.

## Rate Limiting and Fairness
Rate limiting implementations should provide fairness: users shouldn't be penalized for others' behavior. Fixed window counter violates fairness at boundaries. Sliding window approaches provide better fairness over time. Some systems implement fairness queuing: requests from users under quota are processed before requests from those over quota. Max-min fairness algorithms ensure no user receives less than their fair share when others aren't using theirs. Fair queuing implementations are more complex than simple rejection. The fairness model should match service goals: some services prioritize simplicity over fairness, others require strong fairness guarantees. Documenting fairness properties helps users understand rate limiting behavior.

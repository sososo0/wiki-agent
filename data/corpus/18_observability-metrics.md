# Observability: Metrics

## Counter Metrics
Counter metrics are monotonically increasing values that measure the total number of occurrences of an event. They start at zero and can only stay the same or increase; they never decrease. Common applications include counting HTTP requests, errors, or completed jobs. Counters are useful for calculating rates over time (requests per second) and identifying trends. The main tradeoff is that you lose granularity about individual event sizes—a counter only tells you how many times something happened, not how much. Implementation is straightforward and has minimal performance overhead. Example: tracking `http_requests_total` with labels for status code and endpoint allows you to compute error rates and request volume trends.

## Gauge Metrics
Gauge metrics represent a single numerical value that can increase or decrease over time, measuring instantaneous state rather than cumulative totals. They capture snapshots of system conditions like memory usage, active connections, or queue depth. Gauges are valuable for understanding current resource utilization and capacity planning. The primary limitation is that gauges represent only a point-in-time measurement; rapid changes between collection intervals may be missed. They require careful handling to avoid misleading conclusions from sampling at inopportune moments. Example: `database_connection_pool_active_connections` shows how many connections are currently in use, helping operators determine if pool sizing is adequate.

## Histogram Metrics
Histograms bucket observations into ranges and count how many fall into each bucket, enabling analysis of value distributions without storing individual data points. They are essential for understanding latency distributions, request sizes, and other continuous values. Histograms automatically enable percentile calculations (p50, p95, p99) and allow rate-of-change queries. The main tradeoff involves choosing appropriate bucket boundaries—poor bucket selection can obscure important details. Histograms consume more storage than counters or gauges but significantly less than storing raw values. Example: a histogram for response times with buckets [10ms, 50ms, 100ms, 500ms, 1s, 5s] reveals whether latency problems affect all requests or only a tail of slow requests.

## Summary Metrics
Summary metrics are similar to histograms but pre-compute quantiles at collection time rather than allowing query-time aggregation. They report configured quantiles directly (e.g., p50, p90, p99) along with sum and count. Summaries are useful when you need specific percentiles but cannot afford the storage overhead of histograms, or when aggregation across time windows is not required. The major limitation is that summaries cannot be aggregated across multiple instances—if you have ten servers, you cannot compute a meaningful p95 across all of them. Summaries also provide less flexibility for analysis since quantiles are pre-selected. Example: application-level reporting of request latency quantiles where you only care about p50, p95, and p99 for alerting.

## Cardinality Explosion
Cardinality explosion occurs when metric labels have unbounded or high-variance values, creating millions of unique metric series and overwhelming storage and query systems. Common causes include adding user IDs, request IDs, or unbounded enum values as labels. This degrades performance, increases costs, and can destabilize monitoring infrastructure. Prevention requires careful label design: only include labels with a small, known, fixed set of values. Use aggregation or sampling for high-cardinality dimensions, or move them to logs or traces instead. Example: labeling `http_request_duration` with a unique request ID would create one series per request; instead, label only by endpoint, method, and status code.

## Rate Calculation from Counters
Rates measure the frequency of events over time, typically calculated from counter metrics using `(counter_value_now - counter_value_past) / time_interval`. Rate calculations are fundamental for understanding throughput, error frequency, and system behavior. Most time-series databases provide rate functions that handle counter resets (when a service restarts) automatically. Rates smooth out the discrete nature of counters, making trends visible and enabling meaningful alerting. The main consideration is choosing the right time window—too short introduces noise, too long hides sudden changes. Example: `rate(http_requests_total[5m])` computes the average request rate over the last five minutes, useful for scaling decisions.

## Service Level Indicator (SLI)
A Service Level Indicator is a measurable metric that reflects a user-visible aspect of service quality, such as availability, latency, or error rate. SLIs are derived from application and infrastructure metrics and form the basis for defining SLOs. Effective SLIs must be observable, directly correlate with user experience, and be influenced by engineering decisions. Common SLIs include request success rate, request latency at specific percentiles, and end-to-end availability. SLIs should focus on user experience rather than implementation details; CPU usage is not an SLI, but the latency it causes is. Example: `(successful_requests / total_requests)` measured over a one-minute window is an SLI for request success rate.

## Service Level Objective (SLO)
A Service Level Objective is a target value for an SLI, expressed as a percentage or threshold over a specified time period. SLOs translate business requirements into measurable engineering goals (e.g., "99.9% of requests complete in under 500ms over a rolling 30-day window"). SLOs must be achievable with current infrastructure, meaningful to users, and aligned with business priorities. They drive resource allocation and on-call practices. SLOs that are too strict waste engineering effort; too loose fail to meet user expectations. Example: an SLO of "99.5% availability, measured monthly" sets a clear target while allowing five 9-minute outages per month.

## Service Level Agreement (SLA)
A Service Level Agreement is a contractual commitment with customers, typically incorporating one or more SLOs, often with financial consequences (credits or penalties) for violations. SLAs are business and legal documents, not purely technical; they reflect commercial commitments. SLAs usually target slightly lower performance than internal SLOs to provide buffer against legitimate incidents. SLAs must be measurable using the same methods customers can verify, creating accountability. Example: a cloud provider's SLA might promise "99.9% availability per month, with service credits of 10% monthly charges for breaches."

## Percentile Latency: P50 (Median)
P50, or the 50th percentile, represents the median latency—the value below which 50% of requests fall. P50 is useful for understanding typical user experience under normal conditions and establishing baseline performance. It is relatively stable and not heavily influenced by outliers. P50 alone is insufficient for production systems because it ignores performance problems affecting the tail; a system with 50% of requests at 100ms and 50% at 10s has a p50 of 100ms but poor user experience. Example: if p50 latency is 150ms, half your users experience requests faster than 150ms and half slower.

## Percentile Latency: P95 (95th Percentile)
P95 represents the latency threshold below which 95% of requests fall, capturing the experience of most users while beginning to include tail latencies. P95 better reflects real-world user experience than p50, revealing when performance degrades for a meaningful percentage of traffic. It balances sensitivity to outliers with stability. P95 is often used for SLO targets because it penalizes poor tail performance without being dominated by rare extreme cases. Example: a p95 latency of 500ms means 95% of requests complete within 500ms, but 5% are slower.

## Percentile Latency: P99 (99th Percentile)
P99 is the latency below which 99% of requests fall, capturing high-percentile tail behavior affecting a small but non-negligible fraction of users. P99 is critical for services where even rare slowdowns impact user experience or for identifying performance problems in specific request types or codepaths. P99 is more volatile than p50 or p95 and influenced more heavily by outliers, so it requires larger sample sizes for stability. Example: if p99 latency is 2 seconds, 1 in 100 requests is slower than 2 seconds.

## Percentile Latency: P99.9 and Beyond
P99.9 and higher percentiles (p99.99, etc.) represent extremely tail latencies affecting very few requests. These high percentiles are important for identifying worst-case scenarios, debugging systematic slowdown patterns, and ensuring consistent user experience for all users. Measuring them accurately requires enormous sample sizes; insufficient data leads to noisy or misleading percentile estimates. High percentiles should inform long-tail optimization efforts but should not drive SLO targets for most services. Example: monitoring p99.9 might reveal a once-per-thousand request pattern of slowness caused by a specific edge case in your code.

## Golden Signals: Request Rate
Request rate measures how many requests the service handles per unit time, typically expressed as requests per second. It is a fundamental signal for understanding system load, capacity planning, and identifying traffic anomalies. Request rate is calculated from counter metrics and should be measured separately by endpoint, method, or other relevant dimensions. Sudden changes in request rate may indicate client issues, upstream failures, or successful scaling. Example: monitoring `rate(requests_total[1m])` per endpoint reveals which parts of your service are under load and helps detect traffic pattern changes.

## Golden Signals: Errors
Error rate measures the proportion of failed requests, expressed as a percentage or count per second. Errors include HTTP 5xx responses, timeouts, exceptions, and application-level failures. Tracking error rates separately by endpoint, error type, and cause enables rapid diagnosis. Errors often precede latency degradation, making error rates valuable early warning signals. The definition of "error" must be consistent across the system; some 4xx responses may be legitimate (client errors) rather than service failures. Example: a sudden spike in 502 Bad Gateway errors indicates potential problems in backend services or load balancers.

## Golden Signals: Latency
Latency measures how long requests take to complete, typically captured as histograms or percentiles. Latency is user-facing and directly impacts experience. Latency should be measured end-to-end from user perspective and also instrumentally for each component. Degradation in latency is often the first symptom of resource exhaustion or code performance problems. Latency metrics should include multiple percentiles (p50, p95, p99) to understand both typical and tail behavior. Example: tracking p95 latency by database query type helps identify which operations need optimization.

## Golden Signals: Saturation
Saturation measures how close to maximum capacity the service is, typically expressed as resource utilization (CPU, memory, disk) or queue depth. Saturation is a leading indicator—when saturation is high, latency and errors will soon follow. Measuring saturation requires understanding what "full" means for each resource; 80% CPU utilization on a sustained batch job is normal, while 80% on a latency-sensitive service suggests imminent degradation. Saturation metrics drive scaling decisions and capacity planning. Example: monitoring connection pool saturation (active connections / max connections) reveals when you need to increase pool size or scale horizontally.

## Metric Cardinality Management
Metric cardinality management involves careful design of metric labels to prevent cardinality explosion while maintaining observability. Strategies include: limiting label values to known, bounded sets; aggregating high-cardinality dimensions before collection; routing high-cardinality data to logs or traces instead; and implementing cardinality limits in instrumentation. Effective cardinality management requires establishing naming conventions and label guidelines across teams. Regular audits of metrics infrastructure identify unexpected high-cardinality series. Example: instead of labeling each request with the user ID (unbounded cardinality), emit a counter with fixed labels and handle user-specific analysis in logs.

## Metric Aggregation and Time Windows
Metric aggregation combines multiple data points into summary statistics over specified time windows, enabling efficient storage and querying. Common aggregations include sum, average, min, max, and percentiles. Time window selection depends on use case: 1-minute windows for alerting, 5-minute or longer for trend analysis. Aggregation reduces storage overhead but loses granularity; a 1-minute average hides problems that occur and resolve within seconds. Time-series databases typically support flexible re-aggregation at query time. Example: aggregating per-second request counts into 1-minute buckets creates a manageable data volume while preserving visibility into service behavior.

## Custom Metric Design
Custom metrics measure application-specific behavior not captured by standard metrics, such as business logic execution (orders processed, feature flag states) or domain-specific performance (cache hit rates, queue sizes). Custom metrics follow the same principles as standard metrics: use counters for totals, gauges for state, histograms for distributions. Design custom metrics with clear naming, documentation, and label strategies. Over-instrumentation creates maintenance burden and increases storage costs; metrics should answer specific operational or business questions. Example: an e-commerce service might track `checkout_completion_rate` as a counter to monitor business impact of latency or error changes.

# Event-Driven Architecture

## Event Sourcing Fundamentals

Event sourcing is an architectural pattern where all changes to application state are captured as a sequence of immutable events stored in an append-only log. Rather than storing only the current state of an entity, the system records every state transition as a discrete event. The current state is derived by replaying events from the start of the log. This creates a complete audit trail of how the system arrived at its present state. Event sourcing enables temporal queries, facilitates debugging, and provides natural support for event replay and recovery. Trade-offs include increased storage requirements for the event log, added complexity in querying current state, and potential consistency challenges when multiple services source from the same events. Example: A banking system records events like "AccountCreated," "DepositMade," and "WithdrawalProcessed" rather than updating a balance field directly. The current balance is computed by summing all relevant events.

## Command Query Responsibility Segregation (CQRS)

CQRS separates read and write operations by using different models for each. Commands modify state through a write model (typically event-sourced), while queries read from one or more optimized read models tailored to specific access patterns. The read models are eventually consistent projections built from events produced by the write model. This separation allows independent scaling of read and write paths and enables optimization of each for its specific concerns. Trade-offs include eventual consistency between read and write models, increased operational complexity, and potential data staleness in read models. CQRS works well with event sourcing but can be applied independently. Example: An e-commerce system uses a write model for order processing and maintains denormalized read models for product catalogs, user wishlists, and order history, each optimized for their respective query patterns.

## Outbox Pattern for Reliable Message Publishing

The outbox pattern ensures reliable publication of events by storing them in a local database table (the outbox) as part of the same transaction that modifies application state. A separate process polls the outbox and publishes events to external systems (message brokers, other services). This guarantees at-least-once delivery semantics without relying on distributed transactions. If the outbox poller fails, it can resume without losing events. The pattern is particularly valuable when you need guaranteed event delivery alongside transactional consistency. Trade-offs include the need for an additional polling mechanism and outbox table cleanup. Example: A user service creates a new user and inserts both a Users table row and an OutboxEvents row in a single database transaction, ensuring no user creation events are lost even if message broker publishing fails.

## Event Versioning Strategies

Event versioning addresses the reality that event schemas evolve over time as systems change. Versioning strategies include explicit versioning (including version numbers in events), upcasting (transforming old event formats to new formats on read), and supporting multiple event versions simultaneously. Proper versioning prevents deserialization failures when replaying old events with newer code. Common approaches include semantic versioning of events, maintaining migration functions, and designing events to be forward-compatible. Trade-offs involve added complexity in event handling code and the need to maintain backward compatibility. Example: A ProductPriceChanged event initially contains only product_id and new_price; version 2 adds currency field. When replaying v1 events with v2 code, an upcast function provides a default currency value.

## Event Replay and Temporal Recovery

Event replay is the process of reprocessing a sequence of events from the past to reconstruct system state or rebuild projections. It's fundamental to event-sourced systems and enables recovery from bugs, data corruption, or projection failures. Replaying events from a known point-in-time reconstructs exactly what the system's state was then. This supports debugging complex distributed system issues and rebuilding read models from scratch. Challenges include ensuring replayed events produce idempotent results, handling events with time-dependent logic, and managing replay performance over large event streams. Example: If a user service projection becomes corrupted, replaying all UserCreated, UserUpdated, and UserDeleted events reconstructs the correct user database without requiring a separate backup.

## Choreography-Based Event Orchestration

Choreography is a decentralized approach to event-driven coordination where services react to events published by other services without a central coordinator. When service A completes an action, it publishes an event; services B and C subscribe independently and react. Each service knows what events it produces and which ones it listens for, but there is no explicit orchestration logic. This promotes loose coupling and allows services to evolve independently. Trade-offs include difficulty in understanding overall workflows (events scatter across multiple services), potential for cascading failures, and complexity in implementing compensating transactions for rollback. Example: Order placement triggers OrderCreated; payment service listens and processes payment; inventory service listens and reserves stock; notification service sends confirmations—each reacting independently to events.

## Orchestration-Based Event Coordination

Orchestration uses a central component (orchestrator or saga) that explicitly directs the sequence of events and operations across services. The orchestrator receives an initial request and issues commands to other services, receives their responses (often as events), and determines next steps. This centralizes workflow logic and makes system behavior explicit and auditable. Trade-offs include creating a potential single point of failure, coupling services to the orchestrator, and increased operational complexity. Orchestration is well-suited for complex, ordered workflows where understanding the complete flow is critical. Example: An order orchestrator receives a new order request, commands payment service to charge, waits for payment confirmation event, then commands inventory service to allocate stock, ensuring a defined sequence.

## Event Sourcing with Snapshots

Snapshots are periodic captures of aggregate state in event-sourced systems, eliminating the need to replay all events from the beginning each time state is needed. Rather than replaying events from time zero, the system loads the most recent snapshot and replays only newer events. This dramatically improves performance for long-lived aggregates with thousands of events. Snapshots introduce trade-offs: they consume storage, require careful management to ensure consistency, and add complexity to the event replay logic. Snapshots are typically taken at regular intervals or thresholds (e.g., every 100 events). Example: A shopping cart aggregate might be snapshotted every 50 item additions/removals; loading the cart replays only the events after the snapshot rather than the entire session history.

## Event Schema Design and Contracts

Proper event schema design establishes clear contracts between event producers and consumers, enabling independent evolution. Events should include essential context (timestamps, correlation IDs, user context), be self-describing, and avoid embedding large objects. Schema design decisions affect downstream flexibility and system resilience. Using explicit schema formats (JSON Schema, Protobuf, Avro) enforces contracts and enables validation. Trade-offs include stricter validation upfront versus more flexible runtime handling. Example: A UserRegistered event should include user_id, email, timestamp, and region, but not the user's address—a separate UserProfileUpdated event handles that, keeping events cohesive and focused.

## Idempotency and Exactly-Once Semantics

Event-driven systems typically offer at-least-once delivery guarantees, meaning events may be processed multiple times. Idempotency ensures that processing the same event multiple times produces the same result as processing it once. Strategies include tracking processed event IDs, making operations naturally idempotent (e.g., Set operations), and using idempotency keys. Exactly-once semantics are expensive to achieve in distributed systems and are often not necessary if idempotency is properly implemented. Trade-offs involve added tracking overhead and complexity in determining whether an operation is naturally idempotent. Example: A payment event includes an idempotency_key; the payment service records processed keys and rejects duplicate processing of the same key.

## Dead Letter Queues and Error Handling

Dead letter queues (DLQs) are holding areas for events that fail processing after retries are exhausted. Rather than silently dropping or endlessly retrying failed events, they're moved to a DLQ for inspection and manual remediation. DLQs enable systems to remain resilient while acknowledging that some events require special handling. Proper DLQ strategy includes clear alerting, analysis of failure patterns, replay capabilities, and eventual resolution processes. Trade-offs involve the need to monitor and manage DLQs and potential data loss if DLQ contents expire without resolution. Example: If an order event cannot be processed due to an inventory service being down, after exponential backoff retries, it moves to an order DLQ where operators can investigate and replay once the service recovers.

## Causal Ordering and Consistency Guarantees

Causal ordering ensures that related events maintain their logical dependencies across a distributed system. If event B depends on event A (e.g., an order cannot be shipped before it's created), the system must preserve this ordering. Mechanisms include sequence numbers, vector clocks, or lamport timestamps to track causal relationships. Systems must balance causal consistency guarantees against performance and complexity. Complete causal ordering across all events is expensive; selective ordering for critical relationships is often sufficient. Example: Using a correlation ID linking UserCreated → UserProfileUpdated → NotificationSent ensures these logically dependent events maintain their ordering even if processed by different services.

## Event Store Implementation Patterns

An event store is the database that persistently records all events. Implementation patterns include append-only tables with surrogate keys, partitioned event logs, and specialized event store systems (EventStoreDB, Axon Framework). Key considerations are write performance (events are typically written sequentially but at high volume), read performance (replaying thousands of events), and efficient querying by aggregate ID or timestamp. Trade-offs include choosing between general-purpose databases and specialized systems, managing table growth over time, and indexing strategies. Example: An event store table has columns: event_id (auto-incrementing), aggregate_id, aggregate_type, event_type, payload, timestamp, version—with a clustered index on aggregate_id for efficient replay.

## Event Filtering and Projections

Projections are read models derived by processing a filtered stream of events. Rather than subscribing to all events, projections filter based on event type, aggregate, or other criteria and build targeted denormalized views. This allows multiple specialized read models to coexist, each optimized for specific queries. Filters reduce processing overhead and improve projection-building performance. Trade-offs include managing multiple projection pipelines and ensuring consistency across them. Example: A user projection filters only UserCreated, UserUpdated, and UserDeleted events to build a normalized users table, while an audit projection captures all events for compliance.

## Distributed Saga Pattern for Long-Running Transactions

The saga pattern coordinates long-running transactions across multiple services using events and compensating actions. When a step fails, compensating transactions undo previous steps to maintain consistency. Sagas can be implemented choreography-style (services react to events) or orchestrator-style (central coordinator directs steps). This pattern is essential for maintaining eventual consistency across distributed boundaries without distributed transactions. Trade-offs include complexity in implementing compensations and difficulty reasoning about failures when many services are involved. Example: A travel booking saga coordinates flights, hotels, and car rentals; if hotel booking fails, previous flight and car reservations are automatically compensated with cancellations.

## Event Correlation and Tracing

Event correlation uses unique identifiers (correlation IDs, trace IDs) to link related events across services, enabling end-to-end tracing of workflows. Each event includes the correlation ID of the operation it's part of, allowing reconstruction of complete request flows. This is critical for debugging distributed systems and understanding causal chains. Correlation IDs should be generated at entry points and propagated through all downstream events. Trade-offs include managing ID generation and propagation across service boundaries and potential overhead of storing and querying by correlation ID. Example: A user registration request generates a trace_id; it flows through UserServiceEvents, PaymentServiceEvents, and NotificationServiceEvents, allowing correlation of all related events.

## Event-Driven Scaling and Backpressure

Event-driven systems must handle varying event production rates without overwhelming consumers or accumulating unbounded queues. Backpressure mechanisms allow consumers to signal when they're overloaded, causing producers to slow down. Without backpressure, fast producers can exhaust memory or cause cascading failures. Strategies include queue depth monitoring, rate limiting at consumers, and feedback signals to producers. Trade-offs involve coordinating backpressure across multiple services and potential latency increases when systems are under load. Example: An inventory service consuming OrderCreated events monitors its processing queue depth; when depth exceeds a threshold, it signals to the order service to throttle event publishing.

## Event Retention and Archival

Event stores grow indefinitely if events are never removed, creating storage and performance challenges. Retention policies determine how long events are kept in the hot event store. Archival strategies move older events to cheaper storage while maintaining replay capability. Considerations include legal/compliance retention requirements, query performance impact of large event stores, and recovery time objectives. Trade-offs involve the complexity of managing multiple storage tiers and potential slow replay of archived events. Example: A payment system keeps events in the hot store for 2 years, then archives to cold storage; event replay loads recently archived events from cold storage on demand.

## Testing Event-Driven Systems

Testing event-driven architectures requires strategies beyond traditional unit tests, including event-based testing (verifying correct events are produced), projection testing (validating read models), and end-to-end workflow tests. Test doubles like in-memory event stores or test event buses enable isolated testing. Property-based testing can verify invariants across event sequences. Trade-offs involve the complexity of setting up comprehensive test scenarios and potential brittleness of tests coupled to event formats. Example: Testing an order workflow involves verifying OrderCreated events trigger payment processing events, which trigger OrderConfirmed events, with assertions at each stage.

## Event-Driven Monitoring and Observability

Monitoring event-driven systems requires tracking event production rates, processing latencies, queue depths, and projection lag (delay between event production and availability in read models). Key metrics include event throughput, end-to-end latency across event chains, DLQ rates, and projection staleness. Structured event logging with correlation IDs enables tracing individual workflows. Trade-offs include the overhead of comprehensive monitoring and complexity in setting meaningful alerting thresholds. Example: Dashboards track order event throughput (events/sec), order processing latency (from OrderCreated to OrderShipped), and whether the user projection is more than 5 seconds behind the event log.

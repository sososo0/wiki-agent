# Database Indexing

## B-tree Indexes

B-tree indexes organize data in a self-balancing tree structure where each node can contain multiple keys and pointers to child nodes. All leaf nodes are at the same depth, ensuring consistent O(log n) lookup performance. B-trees are the default index type in most relational databases including PostgreSQL, MySQL, and Oracle. They efficiently support range queries, prefix searches, and sorted iteration because keys are maintained in sorted order. B-trees perform well for both point lookups and range scans. The trade-off is that they require more I/O operations for writes compared to hash indexes, and they consume more disk space than some alternatives. B-trees work best when you need ordered access patterns or frequently perform range queries like `WHERE age BETWEEN 25 AND 35` or `WHERE created_at > '2024-01-01'`.

## Hash Indexes

Hash indexes use a hash function to map keys directly to bucket locations, enabling constant-time O(1) point lookups. They are ideal for equality comparisons like `WHERE user_id = 12345`. Hash indexes consume less disk space than B-trees and offer faster single-key lookups in many cases. However, they cannot support range queries, prefix searches, or ordering operations. Hash indexes also require careful management of hash collisions and bucket resizing during growth. They perform poorly with skewed data distributions that create uneven bucket utilization. Use hash indexes when you have predictable access patterns limited to exact-match queries and need maximum lookup speed. Most modern databases restrict hash indexes to specific scenarios due to their limited functionality compared to B-trees.

## Composite Indexes

Composite indexes (also called multi-column indexes) index multiple columns together in a single index structure. The order of columns in a composite index matters significantly—the database uses the leftmost columns for filtering before examining subsequent columns. A composite index on `(department, salary, hire_date)` efficiently supports queries filtering on department, or department and salary, but not queries filtering only on salary or hire_date. Composite indexes reduce the need for multiple single-column indexes and can enable index-only query execution. The trade-off is increased write overhead and disk space consumption. Use composite indexes when you have common query patterns that consistently filter on the same set of columns. The selectivity of the leftmost column is particularly important—put the most selective column first to eliminate the most rows early in the index scan.

## Covering Indexes

A covering index includes all columns needed to satisfy a query without accessing the base table. This allows the database to execute queries entirely from the index, avoiding expensive table lookups. For example, an index on `(user_id, status, created_at)` can fully answer the query `SELECT status, created_at FROM users WHERE user_id = 5` without touching the main table. Covering indexes reduce I/O operations and latency for targeted queries. The trade-off is significantly increased index size and write amplification, since all included columns must be maintained in the index. Some databases support non-key columns that are included in the index leaf but not used for tree navigation, reducing some overhead. Use covering indexes for high-frequency queries on specific columns, or queries on tables where the index is much smaller than the full table. Be careful not to create too many covering indexes, as the write cost compounds quickly.

## Index Selectivity

Index selectivity measures how effectively an index narrows down the result set—the ratio of distinct values in the indexed column to total rows. High selectivity (many distinct values) means the index can eliminate many rows, making it efficient. Low selectivity (few distinct values) means the index may scan many rows even after being used. A boolean column with only two values has very low selectivity and may not benefit from indexing. A user ID column with one unique value per row has perfect selectivity. The query optimizer uses selectivity estimates to decide whether using an index is faster than a full table scan. Databases with outdated statistics may make poor index choices. Regularly update table statistics so the optimizer has accurate selectivity information. Focus indexing effort on high-selectivity columns, and avoid indexing low-selectivity columns unless they are part of a larger composite index with high-selectivity leading columns.

## Write Amplification from Indexes

Write amplification refers to the overhead that indexes add to insert, update, and delete operations. Every index on a table must be maintained during modifications, so a table with five indexes incurs roughly five times the write cost compared to an unindexed table. For write-heavy workloads, this becomes a significant performance bottleneck. B-tree indexes must rebalance nodes and may split/merge during updates, compounding the cost. Hash indexes similarly require bucket management. Each index also consumes disk space and memory, putting pressure on caches. The trade-off is that indexes dramatically improve read performance. The optimal index strategy balances read and write patterns—add indexes for critical read queries, but avoid indexing columns used primarily in write operations or with very low selectivity. Monitor index usage and remove unused indexes to reduce write amplification. Consider the ratio of reads to writes; heavily write-heavy tables may benefit from fewer indexes.

## Index on Foreign Keys

Foreign key columns should almost always be indexed to support efficient join operations and foreign key constraint checks. When a foreign key constraint is violated, the database must look up the referenced table to validate the constraint—an unindexed foreign key column forces a full table scan. Similarly, when joining tables on foreign keys, an unindexed foreign key requires expensive nested loop joins. Indexing foreign keys enables index-based joins and hash joins, dramatically improving performance. The write cost is minimal since foreign keys are typically only modified during deletes or updates of related records. Most databases recommend indexing the referencing side (the table containing the foreign key). Some databases automatically create indexes on foreign key columns; others require manual creation. Check your database documentation, as missing foreign key indexes is a common source of slow joins and cascading performance issues during delete operations.

## Partial Indexes

Partial indexes index only a subset of rows that match a WHERE condition, reducing index size and write overhead. For example, `CREATE INDEX active_users ON users(email) WHERE status = 'active'` only indexes active users. Partial indexes are useful when you frequently query a specific subset of data (active records, recent records, high-value customers). They consume less disk space and require fewer write operations since inactive rows are not indexed. The trade-off is that the optimizer must recognize that a query's WHERE clause matches the partial index's condition—some databases are limited in their ability to use partial indexes. Partial indexes work best when the subset is well-defined and permanent, not when filtering on dynamic values. Use them for common scenarios like filtering deleted records, status-based queries, or temporal data. Not all databases support partial indexes equally; PostgreSQL supports them well, while MySQL and Oracle support them with limitations.

## Index Statistics and Query Planning

The query optimizer uses index statistics to estimate the cost of different execution plans and decide whether an index should be used. Statistics include cardinality (distinct value count), selectivity, and data distribution across index values. Outdated or inaccurate statistics cause the optimizer to make poor decisions, such as choosing a full table scan over an available index. Most databases automatically gather statistics during table creation, but these can become stale as data changes. Manual statistics updates are necessary when data distribution shifts significantly or after bulk operations. Some databases use histograms to track value distributions and detect skew. Running `ANALYZE` in PostgreSQL or `ANALYZE TABLE` in MySQL forces statistic recalculation. Set up regular maintenance jobs to update statistics on frequently-modified tables. Query slow logs can reveal cases where the optimizer made suboptimal choices, often due to stale statistics. Accurate statistics are foundational to good query performance.

## Index Fragmentation

Index fragmentation occurs when index pages are not physically sequential on disk, causing additional I/O operations during index scans. Fragmentation builds up over time as rows are inserted, updated, and deleted, leaving gaps in the index structure. High fragmentation (typically >30% on conventional systems) noticeably degrades scan performance. Some databases measure fragmentation as the percentage of out-of-order pages. Rebuilding an index rewrites it from scratch in sequential order, eliminating fragmentation but locking the table during the operation. Reorganizing an index compacts pages without a full rebuild and is less disruptive. The trade-off is that maintenance operations consume I/O and may impact active workloads. For large tables with heavy write activity, periodic maintenance is necessary. Smaller tables or read-only tables rarely need attention. Monitor fragmentation levels and rebuild or reorganize indexes when fragmentation exceeds acceptable thresholds. Some databases support online index operations that don't lock the table, allowing maintenance during business hours.

## Unique Indexes

Unique indexes enforce uniqueness constraints while simultaneously serving as regular indexes for query optimization. A unique index on `email` prevents duplicate email addresses and makes point lookups on email fast. Unique indexes have identical performance characteristics to non-unique B-tree indexes for reads. The enforcement of uniqueness adds minimal overhead to write operations since the uniqueness check is part of the constraint validation that would occur anyway. Unique indexes are typically created implicitly when defining primary keys or unique constraints. They can be composite, enforcing uniqueness across multiple columns. Unique indexes are useful for enforcing business rules while gaining index benefits. Partial unique indexes are possible on some databases, allowing unique enforcement only within a subset of rows (e.g., only active records). Use unique indexes wherever your schema requires unique values, both for data integrity and query performance.

## Index-Only Scans (Index Covering Optimization)

Index-only scans execute queries entirely using index data without accessing the underlying table. This dramatically reduces I/O since the database reads only the compact index, not the larger table. The query `SELECT COUNT(*) FROM users WHERE status = 'active'` can execute as a pure index scan if an index on `status` exists. For index-only scans to work, the index must include all columns referenced in the query—either as key columns or as included non-key columns. Databases track which index pages contain visible (non-deleted) rows; when all rows on an index page are deleted or hidden to other transactions, the index page still exists but contributes no results. This means index-only scans are not always possible for time-sensitive data with rapid deletions. The performance gain is substantial—index-only scans can be 10-100x faster than table scans for selective queries. Design indexes with index-only scans in mind for high-frequency queries, particularly aggregates and counts on frequently-accessed subsets of data.

## Descending Indexes

Descending indexes sort key values in descending order, enabling efficient queries with ORDER BY DESC clauses. Without a descending index, queries like `SELECT * FROM logs ORDER BY timestamp DESC LIMIT 10` require the database to scan the entire index in reverse or perform an expensive sort operation. Descending indexes consume the same disk space as ascending indexes and have identical write costs. Many databases can use an ascending index in reverse order with minimal performance penalty, making explicit descending indexes less critical than they appear. However, composite indexes with mixed sort orders (e.g., `(status ASC, created_at DESC)`) benefit from explicit direction specification on each column. Use descending indexes when you have frequent DESC queries on large result sets or when mixed sort orders are common. Check your database's documentation on whether it efficiently reverses index scans, as this varies between PostgreSQL, MySQL, and other systems. For most use cases, ascending indexes are sufficient, but descending indexes are a targeted optimization for specific query patterns.

## Expression Indexes (Functional Indexes)

Expression indexes index computed values or transformations of columns rather than raw column values. For example, `CREATE INDEX lower_email ON users(LOWER(email))` enables efficient case-insensitive email searches without applying the LOWER function during every query. Expression indexes make queries faster when you consistently apply the same transformation. The trade-off is increased write cost, since the expression must be evaluated for every row insertion or update. Complex expressions with function calls can significantly impact write performance. Expression indexes are useful for normalized searches, date transformations, or computed fields. However, simple transformations may be better handled at the application level if the overhead is significant. Not all databases support expression indexes equally—PostgreSQL supports them well, while MySQL has limited support depending on version. Keep expressions simple and deterministic; avoid volatile functions like `NOW()` or `RANDOM()`. Expression indexes shine for common normalization patterns like case-insensitive searches or timezone conversions.

## Conditional Indexes (Filtered Indexes)

Conditional indexes apply a WHERE clause to index only rows meeting specific criteria, reducing index size and write overhead. This is similar to partial indexes but with specific use in conditional filtering. A conditional index on `(user_id, order_total)` where `order_total > 100` efficiently supports high-value customer queries without indexing low-value transactions. Conditional indexes are especially useful for time-series data where you typically query recent records but may occasionally query historical records. They reduce index maintenance cost by excluding millions of rows that never participate in queries. The optimizer must recognize the query's WHERE clause and correctly match it to the conditional index definition. Use conditional indexes when you have clear, permanent filtering criteria that significantly reduce the indexed row set. Combine conditional indexes with application-level query routing to ensure queries against the full dataset don't accidentally miss the conditional index. This technique is sometimes called "partial indexing" and is well-supported in PostgreSQL but less so in other databases.

## Index Hints and Query Optimizer Overrides

Index hints allow developers to explicitly instruct the optimizer to use (or avoid) specific indexes. Hints are necessary when the optimizer makes poor decisions despite accurate statistics, or when testing alternative index strategies. Different databases use different hint syntax: MySQL uses `USE INDEX (index_name)`, SQL Server uses `WITH (INDEX=index_name)`, and Oracle uses `/*+ INDEX(table index_name) */`. Hints should be used carefully as they reduce optimizer flexibility and may cause problems if data distributions change significantly. Hard-coded hints also complicate maintenance and upgrades. The better approach is to investigate why the optimizer made a poor choice—usually stale statistics, missing indexes, or incorrect cardinality estimates. Use hints temporarily for debugging and as a last resort when the optimizer is provably suboptimal. Document why hints were necessary so future maintainers understand the reasoning. Most production systems should minimize hint usage and instead focus on correct statistics, schema design, and comprehensive index strategies.

## Index Compression

Index compression reduces disk space and improves cache efficiency by storing index pages in compressed format. Prefix compression stores common prefixes once and references them, working well for sorted string data. Dictionary compression replaces repeated values with smaller tokens. Compression reduces I/O and memory usage at the cost of increased CPU usage for decompression during index scans. Modern systems with fast CPUs and limited I/O often see net performance gains from compression. Large indexes on SSD systems with capacity constraints benefit most from compression. Small, frequently-scanned indexes may suffer if decompression overhead exceeds I/O savings. Compression is transparent to application code—the database handles compression and decompression automatically. Most modern databases support compression on indexes; enable it when disk space is limited or when index size significantly impacts memory usage. Measure compression ratios and performance impacts in your specific workload before enabling widely. Compression trades CPU for I/O, which is appropriate when I/O is the bottleneck.

## Bitmap Indexes

Bitmap indexes store a bitmap (array of bits) for each distinct value in a column, with each bit representing a row. They excel in data warehouses and analytical systems with many distinct columns but relatively few distinct values per column. A bitmap index on a gender column uses only one bit per row, extremely space-efficient. Bitmap indexes efficiently support complex boolean queries combining multiple conditions (e.g., `WHERE gender='M' AND region='West' AND age_group='30-40'`). They can scan multiple bitmaps and combine them with bitwise operations much faster than traditional indexes. However, bitmap indexes perform poorly on columns with high cardinality (many distinct values) since one bitmap per distinct value becomes impractical. They also incur significant overhead during updates when many bitmaps must be modified. Bitmap indexes are rarely used in OLTP systems but are standard in data warehouses using databases like Oracle or PostgreSQL with specialized extensions. Use bitmap indexes only when you have low-cardinality columns in analytical workloads with frequent multi-column filtering.

## Index Maintenance and Cleanup

Regular index maintenance prevents performance degradation and reclaims unused resources. Unused indexes consume disk space, memory, and impose write overhead without providing benefits. Most databases provide tools to identify unused indexes—examine query logs and index usage statistics. Drop indexes that haven't been used in months or years. Rebuilding fragmented indexes restores sequential page layout; reorganizing compacts pages with less disruption. Update statistics after bulk data loads or significant modifications so the optimizer has current information. Monitor index bloat on tables with frequent updates and deletes; expired rows may occupy index space but not contribute to queries. Some systems support online index operations that don't lock tables, allowing maintenance during business hours. Schedule index maintenance during low-traffic periods to minimize impact. Implement alerting for high fragmentation or missing statistics. A disciplined maintenance approach prevents indexes from becoming performance liabilities. Document your maintenance policies and automate routine tasks through scheduled jobs.

## Multi-level Indexes and Index Structures

Most indexes use tree structures with multiple levels where higher levels provide quick navigation to lower levels. A B-tree with millions of rows typically has 3-4 levels: root, intermediate, and leaf nodes. The root level fits in memory; subsequent levels are fetched from disk as needed. Lookup performance remains O(log n) regardless of table size because each level eliminates a large portion of candidates. Hash indexes typically use single-level flat structures unless they support dynamic resizing. Some databases use specialized structures like T-trees or skip lists in memory-based systems. The depth of the index directly impacts lookup cost—deeper indexes require more disk I/O. Databases optimize tree structure by choosing high branching factors (many keys per node) to minimize depth. Key compression and page size tuning affect how many keys fit in each node. Understanding your index's depth and structure helps diagnose performance issues. Tools like `EXPLAIN` output often show index depth or page statistics. Analyze your most critical indexes' structure and consider optimizations like key compression or page size tuning for heavily-used indexes.

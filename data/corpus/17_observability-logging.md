# Observability: Logging

## Structured Logging Fundamentals

Structured logging is the practice of emitting logs as machine-parseable data structures (typically JSON) rather than free-form text strings. Each log entry contains explicit fields like timestamp, service name, request ID, and event details, enabling programmatic analysis and aggregation. This contrasts with unstructured logging, where information is embedded in human-readable text that must be parsed with regex patterns or heuristics. Structured logging is essential for distributed systems because it allows centralized log aggregation systems to index and query logs consistently across thousands of services. Tradeoffs include slightly larger log size (mitigated by compression), initial setup overhead, and developer discipline to include relevant fields. Example: instead of logging `"User login failed due to invalid password"`, emit `{"event":"login_failed", "user_id":"12345", "reason":"invalid_password", "timestamp":"2024-01-15T10:30:00Z"}`. This enables querying all failed logins for a user or filtering by failure reason across all services.

## Log Levels and Severity Classification

Log levels provide a standardized way to classify the severity and importance of log entries. Common levels in order of increasing severity are: DEBUG (detailed diagnostic info), INFO (general informational messages), WARN (potentially problematic situations), ERROR (error events that may be recoverable), and CRITICAL/FATAL (severe errors that may cause service degradation). Choosing appropriate levels is crucial: too many WARN/ERROR logs create alert fatigue and obscure real issues, while insufficient logging misses important failures. DEBUG logs are typically disabled in production to reduce volume. INFO level should capture significant state changes and important business events. WARN level is appropriate for recoverable errors or unusual conditions that warrant investigation but don't immediately impact service availability. ERROR level indicates failures that need operator attention. Different components may have different appropriate thresholds based on their criticality and expected failure rates.

## Correlation IDs for Request Tracing

A correlation ID (also called trace ID or request ID) is a unique identifier assigned to a request when it enters the system and propagated across all services, components, and subsystems that handle that request. This enables tracking a single user action through an entire distributed system. Correlation IDs are typically generated at the entry point (API gateway, load balancer) and passed through request headers, message queues, and RPC calls. All logs related to that request include the correlation ID, allowing operators to reconstruct the complete flow by searching for a single ID across multiple log streams. This is invaluable for debugging user-reported issues. Implementation requires discipline: every outbound request must include the correlation ID header, and every logging statement must include it. Standards like W3C Trace Context provide interoperability guidelines. Example: a user reports slow checkout; searching for their request ID reveals the request passed through the payment service where it spent 5 seconds waiting for a response from the fraud detection service.

## Log Sampling Strategies

Log sampling reduces logging volume by selectively emitting logs rather than logging every event. Common strategies include: uniform sampling (log every Nth occurrence), probabilistic sampling (log with probability P), and decision-tree sampling (sample based on attributes like user ID, endpoint, or error status). Sampling is necessary because high-volume systems may generate billions of log entries daily, making full logging prohibitively expensive in storage and processing. Tradeoffs include potential loss of visibility into low-frequency events and increased complexity in analysis (accounting for sampling rates when aggregating). Adaptive sampling adjusts rates dynamically: increase sampling for error conditions, decrease for routine operations. Always log ERROR and higher severity levels completely; sample DEBUG and INFO. When implementing, ensure sampled logs include metadata indicating the sampling rate so analysis can extrapolate correctly. Example: sample INFO logs at 1%, but log all ERRORs. A rare but important error condition may still be captured even though 99% of routine requests are dropped.

## Correlation IDs in Microservices

In microservices architectures, a single user request typically involves multiple service calls, message queue operations, and asynchronous tasks. Propagating correlation IDs across service boundaries enables tracing these distributed transactions. Implementation requires: generating a correlation ID at the system entry point (API gateway), passing it in request headers for synchronous calls, including it in message attributes for asynchronous messaging, and passing it to background jobs. Each service must extract the correlation ID from incoming requests and include it in all outbound calls and logs. Libraries and frameworks often provide middleware to automate this. Service meshes like Istio can propagate correlation IDs automatically. Without this discipline, reconstructing request flows becomes extremely difficult. Example: User submits an order via the API gateway (generates correlation ID "req-12345"). The order service calls inventory service and payment service in parallel, both including "req-12345" in headers. Payment service emits a message to the fraud-check topic with "req-12345" as a field. All resulting logs from all services include this ID, enabling end-to-end tracing.

## Log Levels in Production vs. Development

Production and development environments typically use different log level configurations. Development environments usually configure logging at DEBUG level to provide maximum visibility for developers working on features and debugging issues. This high verbosity is acceptable because development services are low-volume and serve individual engineers. Production environments typically use INFO level as default, significantly reducing log volume and storage costs while still capturing important business events and state changes. ERROR and higher levels are always captured. The transition between environments should be intentional and configured (e.g., via environment variables, configuration files, or feature flags) rather than hard-coded. Some systems implement dynamic log level adjustment, allowing operators to temporarily increase verbosity for specific services when investigating production issues without redeploying. This should be time-limited to prevent accidentally leaving verbose logging enabled. Documentation should clearly specify expected log levels for each environment. Example: A service defaults to INFO in production but can be switched to DEBUG through a configuration API for 30 minutes to investigate a specific failure.

## PII Redaction Fundamentals

Personally Identifiable Information (PII) includes data that directly or indirectly identifies individuals: names, email addresses, phone numbers, social security numbers, IP addresses, authentication tokens, credit card numbers, and similar data. Logging PII creates compliance risks (GDPR, CCPA, HIPAA), security vulnerabilities (logs are often less protected than transactional data), and increases liability if logs are breached. PII redaction involves removing or masking this sensitive information before logs are persisted. Strategies include: token replacement (replace email with "[REDACTED_EMAIL]"), masking (show only last 4 digits), hashing, or complete removal. Redaction must be consistent and applied before logs reach storage systems. Tradeoffs include performance overhead of regex scanning and potential difficulty debugging issues where redacted information was relevant (mitigated by separate, highly-protected logs for authorized operators). Example: log `"User authentication failed"` instead of `"User john.doe@example.com authentication failed"`, or show only `"User [email protected] authentication failed"`.

## Automatic PII Detection and Masking

Automated PII detection uses pattern matching, regular expressions, and machine learning to identify sensitive information in logs without requiring developers to explicitly exclude it. Tools scan logs in real-time or during ingestion to detect common patterns like email addresses (regex), credit card numbers (Luhn algorithm), phone numbers (digit patterns), and common field names known to contain PII. Once detected, the information is masked or redacted before storage. This approach reduces developer burden compared to manual redaction but requires careful tuning to avoid both false negatives (missing PII) and false positives (redacting legitimate data that happens to match patterns). High-entropy fields like authentication tokens are relatively easy to detect; distinguishing legitimate data from PII is harder (is "123-45-6789" a user ID or SSN?). Automated systems should be supplemented with developer awareness and explicit guidance for high-risk code paths. Regular audits should verify that common PII patterns are being caught. Example: a log aggregation system uses regex patterns to detect and mask email addresses, phone numbers, and credit card numbers, reducing them to patterns like "user@[REDACTED].com" before indexing.

## Manual PII Redaction in Code

Manual redaction requires developers to explicitly identify and exclude or mask PII when writing logging statements. This involves code reviews, developer training, and guidelines about what information should never be logged. Developers must consciously exclude PII from log messages: instead of logging a full user object, log only the user ID; instead of logging request bodies, log relevant fields while excluding passwords and tokens. This approach places responsibility on developers and can be error-prone, but it ensures developers think carefully about what information is necessary for debugging. Best practices include: maintaining a checklist of sensitive fields, using wrapper functions for common redactions, and automated scanning in code review. Tradeoff is higher upfront effort but potentially better understanding of which information is actually needed for troubleshooting. Example: instead of `logger.info(f"User signed up: {user_object}")` which might include email and phone, write `logger.info(f"User signed up", {"user_id": user.id, "signup_source": user.source})`.

## Token and Secret Redaction

Authentication tokens, API keys, database credentials, and other secrets must never appear in logs. These secrets enable attackers to impersonate users, access systems, or exfiltrate data if log files are compromised. Redaction strategies include: hashing tokens to a consistent value (allows tracing token reuse across logs), replacing with "[REDACTED_TOKEN]", or removing entirely. For debugging, consider storing a hash of the token that operators can correlate with token management systems, enabling investigation without exposing the actual token. Common patterns include JWT tokens, OAuth tokens, API keys, and Bearer tokens in HTTP headers. Developers should avoid logging request/response bodies directly; instead, log only relevant fields and explicitly exclude Authorization headers. Automated detection can identify common secret patterns (strings starting with "sk_", "pk_", lengthy alphanumeric sequences in auth headers). Example: log `"API call to payment service with token=[REDACTED_TOKEN_HASH:abc123]"` allowing correlation with the token database without exposing the actual token.

## Log Retention Policies

Log retention policies define how long logs are stored before deletion or archival. Decisions balance: regulatory requirements (some regulations mandate minimum retention), debugging and forensics needs (incidents often investigated days or weeks after occurrence), storage costs, and compliance risk (shorter retention reduces breach exposure). Typical retention periods range from 7 days to 2 years depending on context. Regulatory requirements (HIPAA, PCI-DSS, SOX) often mandate minimum retention; some require 7 years for certain logs. Operational logs may be retained shorter (14-30 days), security logs longer (90-365 days). Hot storage (immediately accessible) is typically expensive; logs often transition to cold storage (archived) after a period. Implementation usually involves log aggregation systems with tiered storage: recent logs in searchable indices, older logs compressed and archived, very old logs deleted. Example: application logs retained 30 days in searchable form, archived to S3 for 1 year, deleted after 1 year; security audit logs retained 2 years.

## Compliance-Driven Retention Requirements

Different regulations impose specific log retention requirements. GDPR requires retaining logs only as long as necessary; personal data should be deleted when no longer needed, which can conflict with longer retention for debugging. HIPAA typically requires 6 years of audit log retention for healthcare data. PCI-DSS requires 1 year of log retention with at least 3 months immediately available. SOX requires 7 years of financial audit trail retention. CCPA requires ability to delete personal data, necessitating log purging or anonymization capabilities. These requirements vary by jurisdiction and apply only if you handle relevant data types. Non-compliance can result in significant fines and legal liability. Implementation must track: what data is logged, what regulations apply, retention timelines, deletion processes, and audit trails of deletions. This often requires close coordination between engineering, legal, and compliance teams. Example: a healthcare platform must retain HIPAA audit logs 6 years, GDPR personal data logs only as long as the user relationship exists, and implement data deletion mechanisms to comply with CCPA deletion requests.

## Log Storage Cost Optimization

Log storage is often a significant operational expense, especially in high-volume systems. Optimization strategies include: sampling (discussed separately), compression (reduces size 10-100x), filtering (not logging unnecessary events), and tiered storage (hot/warm/cold). Compression is applied during ingestion and at rest; modern formats like gzip or zstd provide excellent ratios for structured logs. Filtering removes low-value logs: perhaps sample DEBUG logs at 1%, limit specific verbose components, or conditionally disable logging for certain high-volume operations. Tiered storage keeps recent logs in expensive but searchable systems (Elasticsearch, Datadog), transitions older logs to compressed object storage (S3, GCS) after days/weeks, and deletes beyond retention period. Cost also includes egress charges if logs are moved between regions. Tools should expose cost-per-service or cost-per-component visibility so teams understand which services are expensive to log. Example: by implementing 1% sampling of INFO logs while keeping ERROR logs at 100%, reducing retention from 90 to 30 days, and compressing archived logs, a system reduced logging costs by 70% while maintaining sufficient visibility for troubleshooting.

## Log Aggregation and Centralization

Centralized log aggregation collects logs from all services, servers, and components into a single searchable system rather than leaving logs scattered across individual machines. This is essential for distributed systems where a single request spans multiple services, making local logs insufficient. Centralization enables: searching across all services simultaneously, correlating events across components, identifying patterns, and retaining logs beyond individual server lifespans. Components of a typical setup include: agents on servers collecting logs, a message queue (Kafka, Pub/Sub) buffering logs, an aggregation service (Logstash, Fluentd), a search/storage backend (Elasticsearch, BigQuery), and a UI (Kibana, Grafana). Tradeoffs include increased infrastructure complexity, network overhead (logs must be transmitted), and cost. Resilience is critical: logs shouldn't cause availability issues if aggregation fails; buffering on agents prevents data loss if aggregation is down. Monitoring the logging pipeline itself is important. Example: a 100-service system generates 100GB of logs daily; aggregation enables finding "all failures in checkout flow" across all services in seconds rather than manually checking 100 servers.

## Real-Time Log Streaming and Analysis

Real-time log streaming sends logs immediately to analysis systems as they're generated, enabling detection of issues within seconds. Architectures typically use message queues (Kafka) or pubsub systems (Google Cloud Pub/Sub, AWS Kinesis) to stream logs for processing. Consumers can: alert on error patterns, compute metrics, trigger automated responses, or populate dashboards. Real-time analysis enables detection of anomalies, cascading failures, and user-impacting issues immediately rather than waiting for batch analysis. Tradeoffs include increased infrastructure complexity, higher operational costs, and potential overhead if streaming isn't properly back-pressured. Failures in the streaming pipeline shouldn't cause log loss; logs should be persisted locally and replayed if streaming fails. Use cases include: detecting repeated errors within seconds, identifying traffic anomalies, triggering automated scaling, or notifying on-call engineers. Batch analysis of the same logs (processed overnight) provides complementary insights for historical trending. Example: a stream processor detects when error rate for a specific endpoint exceeds 5% within a 1-minute window and automatically pages the on-call engineer.

## Log Query and Search Capabilities

Log aggregation systems must support flexible querying to enable operators to find relevant logs quickly. Essential capabilities include: full-text search (find logs containing specific terms), field filtering (logs where `status_code=500`), range queries (timestamps within a range), and logical operators (AND, OR, NOT). Advanced features include: regex matching, nested field queries, aggregations (counting occurrences, computing statistics), and faceted search. Query performance is critical; systems with gigabytes of daily logs must return results in seconds, requiring indexing and optimization. Most systems use inverted indices (map terms to documents) or columnar storage (organize data by field). Query languages vary: some use simple filter UIs, others support structured query languages (Lucene, SQL). Operators should be able to construct queries without programming knowledge but power users need advanced capabilities. Documentation and example queries are valuable. Saved searches enable sharing common queries across teams. Example: `status_code: 500 AND service: payment AND timestamp:[2024-01-15T10:00Z TO 2024-01-15T10:30Z]` finds all errors in the payment service during a specific incident window.

## Log Parsing and Field Extraction

Unstructured log data must be parsed to extract meaningful fields for indexing and querying. For applications already emitting structured logs (JSON), parsing is simple: fields are directly available. For unstructured text logs, parsers use patterns (Grok patterns, regex) to extract fields. Example: text log `"2024-01-15T10:30:00Z ERROR user=12345 action=login reason=invalid_password"` requires parsing to extract timestamp, level, user, action, and reason fields. This is error-prone and inflexible when log formats change. Best practice is to emit structured logs from applications; if you inherit unstructured logs, invest in robust parsing. Parsing happens during log ingestion (at the aggregation system) or on the application side before transmission. Parsing failures (logs that don't match patterns) should be handled: logs remain searchable but as raw text. Document your log formats and parsing rules. Changes to application log formats should include updates to parsing rules and testing. Example: a legacy application emits text logs; ingestion rules use Grok patterns to extract timestamp, service, request_id, and message fields, enabling structured querying despite unstructured source.

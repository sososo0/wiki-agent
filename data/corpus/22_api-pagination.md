# API Pagination

## Offset Pagination Fundamentals

Offset pagination uses `offset` and `limit` parameters to define which slice of results to return. The client specifies how many records to skip (`offset`) and how many to retrieve (`limit`). For example, `GET /users?offset=20&limit=10` skips the first 20 records and returns the next 10. This approach is intuitive and works well for small datasets or when users need to jump to specific page numbers. However, offset pagination becomes problematic at scale because the database must scan and discard all skipped rows before returning results, making deep pagination slow. Additionally, offset pagination is inconsistent under concurrent writes—if records are inserted or deleted between requests, users may see duplicates or miss records.

## Cursor-Based Pagination Overview

Cursor-based pagination encodes the position of the last retrieved item into an opaque token (the cursor) that the client passes in subsequent requests. The server decodes the cursor, determines where to resume, and returns the next batch of results along with a new cursor. This avoids rescanning skipped records and performs consistently even when data changes between requests. Cursors typically encode the primary key or a sortable field value. The tradeoff is that clients cannot jump to arbitrary pages—they must traverse sequentially from the beginning or use a stored cursor. Cursor pagination requires more sophisticated implementation but scales better and prevents duplicate or missing records during concurrent modifications.

## Keyset Pagination Strategy

Keyset pagination (also called "seek method") uses the values of the last row's sort keys to determine the next fetch. Instead of a position offset, the client sends the actual values—for example, `GET /posts?after_id=142&after_timestamp=2024-01-15T10:30:00Z`. The server constructs a query with a WHERE clause that fetches rows lexicographically after those values. This is a specific implementation of cursor-based pagination and is highly efficient because databases can use indexes directly without scanning. Keyset pagination works well for sorted result sets and is particularly effective when pagination keys are indexed. The limitation is that it only supports forward navigation and requires the client to maintain the previous row's key values. It's ideal for feed-style interfaces where users scroll forward through time-ordered content.

## Page Size Limits and Defaults

API pagination requires enforcing maximum and default page sizes to prevent resource exhaustion. A typical pattern is to set a default limit (e.g., 20 items) and a maximum limit (e.g., 100 items), rejecting requests that exceed the maximum. Without these guardrails, clients can request millions of records in a single query, consuming excessive memory and CPU. The default limit should balance responsiveness with server load—too small and mobile clients make excessive requests; too large and the response time increases. Documentation must clearly specify these limits. Some APIs expose the enforced limit in response headers or metadata so clients know whether they received a truncated result set. Enforcement should happen at the API layer before database queries execute.

## Pagination Metadata in Responses

Responses should include metadata indicating whether more results exist and how to fetch them. For offset pagination, include `total_count`, `offset`, and `limit`. For cursor-based pagination, include the `next_cursor` (and optionally `previous_cursor`) and a boolean `has_more`. Total counts are expensive to compute for large tables with filters, so some APIs omit them and only indicate whether a next page exists. Metadata should be in a consistent structure, either as a wrapper object or as HTTP headers. Including too much metadata increases response size; including too little forces clients to make additional requests to determine pagination state. The response should make it clear when the last page is reached so clients stop making requests.

## Offset Pagination at Scale

While offset pagination is conceptually simple, it degrades severely with deep pagination. A query like `OFFSET 1000000 LIMIT 10` requires the database to examine and discard 1,000,010 rows, even if only 10 are returned. For large tables, this causes full table scans or index scans that lock resources. Mitigation strategies include implementing a maximum offset (e.g., reject `offset > 100000`), caching computed offsets, or migrating to cursor-based pagination entirely. Some systems combine offset pagination with filters to reduce the working set. For use cases where users genuinely need random access (like admin dashboards), offset pagination may be acceptable if limited to small offsets. However, for user-facing APIs serving large datasets, cursor-based alternatives are preferred.

## Cursor Encoding and Opaqueness

Cursors should be opaque tokens that clients treat as black boxes, never attempting to parse or construct them. Common encoding approaches include base64-encoding serialized data structures, encrypting pagination state, or using a pointer/reference to a server-side cache entry. Opaque cursors prevent clients from making assumptions about pagination implementation and allow the server to change pagination logic without breaking clients. If a cursor format changes, old cursors should gracefully fail with a clear error rather than producing incorrect results. Documentation should explicitly state that cursors are not meant to be human-readable or reversible. Using UUIDs or session-based cursor references adds security by preventing clients from guessing or crafting cursors to access unauthorized data.

## Consistent Pagination During Concurrent Writes

When records are inserted, updated, or deleted during pagination, offset-based approaches suffer from inconsistencies. A user might see duplicate records if items are inserted before their current position, or miss records if deletions occur. Cursor-based pagination handles writes more gracefully because the cursor binds to specific row values rather than positions. If a previously-seen row is deleted, the cursor still moves forward logically. To achieve strong consistency guarantees, implement snapshot isolation—execute all pagination queries against a consistent database snapshot (e.g., a specific transaction timestamp or MVCC version). This prevents both duplicate and missing results. The tradeoff is that snapshots may increase resource usage and prevent pagination of real-time changing data.

## Filtering and Sorting with Pagination

Pagination interacts with filtering and sorting in complex ways. When paginating filtered results, the limit applies to filtered records, not total records, and deep pagination becomes problematic again. Sorting defines the order in which cursor-based pagination proceeds—if results are sorted by creation date, the cursor should track that field. Changing sort order across pagination requests breaks consistency. Best practice is to require immutable sort order throughout a pagination session. For keyset pagination, pagination keys must match the sort order. For offset pagination with filters, compute total counts against the filtered set, which is expensive. Document clearly which fields can be used for sorting and whether sort order changes are permitted mid-session.

## Cursor Invalidation and Expiry

Cursors should have a bounded lifetime to prevent stale pagination state. If a cursor references a row that no longer exists or a snapshot that's no longer available, the pagination should fail gracefully. Time-based expiry (e.g., cursors valid for 1 hour) is a simple approach. Server-side cursor storage with garbage collection is another pattern. Clients should treat cursor expiry as a signal to restart pagination from the beginning. Clearly document the cursor validity period and error messages when cursors expire. For stateless cursor implementations (e.g., encrypted state), expiry can be encoded in the cursor itself. For user-facing APIs, longer expiry periods reduce frustration; for internal APIs, shorter periods are acceptable. Always provide a clear error code (e.g., `400 Bad Request` with `invalid_cursor` reason) when cursors are invalid.

## Backward Pagination

Most pagination systems support moving forward through results, but backward pagination is less common. Offset pagination supports it trivially by decreasing the offset. Cursor-based pagination requires a `previous_cursor` field and queries that fetch results lexicographically before the cursor position. Implementing backward pagination doubles complexity and may not be necessary for many use cases. If supported, document clearly whether backward pagination is consistent (some systems only guarantee forward consistency). For keyset pagination, backward navigation requires different database queries and index utilization. Unless the use case explicitly requires backward browsing (e.g., chat history), forward-only pagination reduces implementation complexity. When backward pagination is needed, consider storing cursors for already-visited pages rather than recomputing them.

## Pagination Performance and Database Indexes

Pagination performance depends heavily on index design. For offset pagination, a covering index on the sort columns helps the database avoid row lookups. For cursor-based pagination, an index on the pagination key columns (in the same order as the sort) allows the database to seek directly to the cursor position. Composite indexes (multiple columns) are necessary when pagination keys include multiple fields. Without proper indexes, any pagination method degrades to full table scans. Monitor query execution plans to ensure the database uses indexes. For filtered pagination, indexes on both filter columns and sort columns are needed. As table size grows, verify that index performance remains constant—some databases degrade with very large indexes. Periodically analyze and rebuild indexes to maintain performance.

## Pagination State and Session Management

Some pagination implementations maintain server-side state to track cursor position, snapshot versions, or session context. This enables rich features but introduces complexity: state must be stored (in-memory, database, or cache), garbage collected when unused, and replicated across servers in distributed systems. Stateless pagination, where all context is encoded in cursors, avoids these issues but may be less flexible. Hybrid approaches store minimal state (e.g., only a session ID) and encode the actual pagination position in the cursor. For stateless designs, ensure cursors are tamper-proof (signed or encrypted) to prevent abuse. State management affects scalability—in-memory state doesn't scale horizontally unless sticky sessions are used. For high-scale systems, stateless pagination is often preferred despite slightly higher computational cost.

## Pagination with Aggregations and Joins

Paginating results from complex queries (aggregations, joins, GROUP BY) is more difficult than simple row pagination. When paginating grouped results, the cursor must identify which group to resume from, and keyset pagination becomes problematic if the sort key is not unique per group. For aggregated data, consider whether pagination is semantically meaningful—users may not need to paginate through thousands of daily summaries. If pagination is necessary, ensure the cursor keys uniquely identify result rows. Avoid paginating before aggregation (which wastes resources) and instead paginate after aggregation. Complex queries may benefit from materialized views or denormalized tables that simplify pagination logic. Document which fields can be used as pagination keys and whether all result columns are deterministically sortable.

## API Documentation for Pagination

Documentation must clearly specify pagination mechanism, parameters, limits, and response format. For each endpoint, explain whether offset or cursor pagination is used, define required and optional parameters, and show example requests and responses. Document the response metadata format and what fields indicate whether more results exist. Specify default and maximum page sizes. Show how to detect the end of results and explain error codes for invalid pagination requests. Include examples of navigating forward (and backward if supported). Document any pagination limitations—for instance, whether filtered pagination is supported, whether sort order changes are allowed mid-session, or if there's a maximum offset. Provide code examples in common languages demonstrating proper pagination usage. Update documentation whenever pagination behavior changes.

## Pagination in GraphQL APIs

GraphQL APIs typically use cursor-based pagination following the Relay Cursor Connections Specification. Results are wrapped in a connection object containing edges (individual results with cursors) and page information (has_next_page, has_previous_page, end_cursor, start_cursor). This structure is more verbose than REST but standardized across implementations. GraphQL allows clients to request specific pagination metadata fields, avoiding over-fetching. Implementing Relay-style pagination requires wrapping query results and generating cursors for each edge. The benefit is consistency across different GraphQL APIs and client library support. Some GraphQL implementations support both cursor and offset pagination, giving clients flexibility. Document the exact cursor encoding and whether backward pagination is supported, as implementations vary.

## Pagination and Caching

HTTP caching interacts with pagination in subtle ways. Paginated responses should not be cached long-term because subsequent pages depend on consistent sorting and filtering. Set appropriate Cache-Control headers (e.g., `no-cache`, `max-age=0`) to prevent stale pages from being served. Cursor-based pagination is more cache-friendly than offset pagination because cursors are typically unique—different cursors produce different cache keys. However, if cursors are derived from timestamps or IDs, the same logical page might have different cursors across requests, reducing cache hit rates. Consider implementing cache revalidation (ETags) for paginated responses to allow clients to check if results have changed. For client-side caching, discourage clients from caching paginated results indefinitely. Some systems use cache invalidation strategies where certain write operations invalidate pagination caches.

## Pagination with Search and Full-Text Queries

Full-text search and pagination introduce additional complexity. Search engines (Elasticsearch, Solr) have built-in pagination but different semantics than SQL databases. Cursor-based pagination in search contexts often uses internal state like search context IDs (in Elasticsearch, the `search_after` parameter combined with `pit` for point-in-time consistency). Offset pagination in search can be expensive when offset is large because the search engine must rank and discard many results. For search-backed APIs, document whether pagination is available for all queries or only specific ones. Ensure search results are sorted consistently to maintain pagination integrity. If exposing search pagination, clarify that result ordering may not be stable across pagination because relevance scores can shift. Consider limiting search result pagination depth (e.g., only allowing access to top 10,000 results) to prevent exhaustion attacks.

## Pagination Error Handling

Pagination requests can fail in several ways: invalid parameters (offset/limit out of range), expired cursors, malformed cursors, sort fields not available for pagination, or backend errors during result assembly. Return specific, actionable error messages: `400 Bad Request` with reason "offset_out_of_range" is more helpful than a generic error. Include hints in error responses (e.g., "maximum offset is 100000, requested 500000"). For cursor errors, distinguish between expired cursors (`410 Gone`) and malformed cursors (`400 Bad Request`). Document all possible pagination-related error codes. Handle edge cases gracefully: if a filter produces zero results, return an empty result set with appropriate metadata, not an error. Test error paths thoroughly—pagination error handling is often overlooked but critical for client stability. Provide retry guidance in error responses (e.g., whether to retry or start fresh pagination).

## Pagination Best Practices Summary

Use cursor-based pagination for user-facing APIs serving large datasets to ensure consistency and scalability. Enforce reasonable page size limits with clear defaults. Include appropriate metadata in responses to indicate pagination state. Use keyset pagination when sort order is stable and indexed. Implement strong error handling for invalid pagination parameters. Document pagination behavior thoroughly. For small datasets or internal APIs, offset pagination is acceptable. Monitor pagination query performance and maintain proper database indexes. Avoid building pagination state into client logic—treat cursors as opaque. Consider caching implications and use appropriate HTTP headers. Test pagination behavior under concurrent writes and data changes. When migration from offset to cursor pagination is necessary, support both approaches temporarily to avoid breaking existing clients.

# Backpressure Mechanisms

## gRPC Server-Side Flow Control

gRPC implements backpressure through HTTP/2 flow control at the transport layer. The server advertises a receive window size to clients, and when this window is exhausted, clients must wait before sending additional frames. This prevents server-side buffer exhaustion in streaming scenarios. When a server processes messages slower than a client sends them, the transport layer automatically throttles the client without application-level intervention. The main tradeoff is that flow control operates at the transport layer and may not account for application-specific processing constraints—a server could exhaust memory before HTTP/2 flow control triggers. Implementers should pair transport-level backpressure with application-level checks. For example, a server receiving a client stream of sensor data can rely on HTTP/2 window management to slow clients, but should also implement application logic to drop or queue messages if internal processing queues exceed thresholds.

## Client-Side Request Rate Limiting

Client-side backpressure involves the client adjusting its request rate based on server responses or local queue saturation. This can be implemented through token bucket algorithms, sliding window counters, or adaptive rate limiting that responds to server latency. The client is responsible for not overwhelming the server and can react faster than waiting for server-side rejection. Tradeoffs include added client-side complexity and the need to coordinate rate limits across distributed clients. Without coordination, multiple clients can still overload a server even if each respects individual limits. A concrete example: a batch processing client tracks in-flight requests and pauses submission when a threshold is reached, resuming when responses complete. This prevents the client from buffering unbounded pending work.

## Server-Side Queue Management and Rejection

Server-side backpressure involves the server refusing or delaying new work when internal queues or resources are saturated. This can be implemented through rejection policies (returning error codes like 503 Service Unavailable), request queuing with maximum limits, or graceful degradation. The server has authoritative knowledge of its capacity and can make optimal rejection decisions. However, clients must handle rejections gracefully and implement retry logic, which complicates client code. Additionally, rejecting requests under load can cascade failures if clients immediately retry. A concrete example: an API server maintains a bounded queue of pending database connections. When the queue reaches 90% capacity, new requests receive a 429 Too Many Requests response, signaling clients to back off.

## Backpressure in Streaming Pipelines

Streaming data pipelines (like Kafka consumers, message queue processors, or stream processors) require backpressure to prevent intermediate stages from buffering unbounded data. A slow downstream stage should signal upstream stages to pause reading or reduce batch sizes. Many streaming frameworks (Kafka, RabbitMQ, Akka Streams) implement backpressure through consumer lag monitoring and dynamically adjusting fetch sizes or processing batch windows. The tradeoff is between latency (larger batches process faster but increase end-to-end latency) and memory usage. Without backpressure coordination, a fast producer can fill consumer buffers and crash the pipeline. Example: a Kafka consumer that processes records into a database implements backpressure by pausing poll() calls when the processing queue depth exceeds a configured threshold, allowing the broker to retain messages until the consumer catches up.

## Backpressure Metrics and Observability

Effective backpressure management requires instrumentation to detect when backpressure is active and measure its impact. Key metrics include queue depths, request rejection rates, latency percentiles (especially tail latency during backpressure), and flow control window utilization. Alerts should trigger when rejection rates spike or when queues consistently exceed safe thresholds. Without visibility, backpressure can degrade silently, causing slow client timeouts rather than fast rejections. Implementers should expose metrics for queue occupancy, requests dropped, and time spent waiting on backpressure signals. A concrete example: a service exposes a metric for HTTP/2 flow control window exhaustion events; when this metric exceeds a baseline, on-call engineers investigate whether downstream processing is degraded or client behavior has changed.

## Backpressure in Batch ETL Pipelines

Batch ETL pipelines apply backpressure differently than streaming systems because jobs process discrete batches rather than continuous flows. Backpressure mechanisms include scheduling constraints (limiting concurrent job stages), resource quotas (CPU, memory, disk I/O), and checkpoint-based flow control where a stage pauses reading input until the previous stage completes writing checkpoints. The tradeoff is between throughput (more parallelism) and resource stability (fewer failures due to resource exhaustion). Batch systems often tolerate higher latency than streaming systems, allowing aggressive queueing before triggering backpressure. A concrete example: a data warehouse ETL pipeline limits the number of concurrent extract stages to 5 to avoid overwhelming source databases, and each stage waits for previous stage checkpoints before proceeding, preventing intermediate data accumulation.

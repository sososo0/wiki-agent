# Database Sharding

## Hash-Based Sharding Fundamentals
Hash-based sharding applies a hash function to a shard key to determine which partition receives a write or read. The hash function distributes keys uniformly across available shards, reducing the likelihood of uneven data distribution. Common hash functions include consistent hashing, MurmurHash, and MD5. This approach works well when the shard key has high cardinality and relatively uniform access patterns. The primary tradeoff is that range queries become inefficient because logically adjacent keys may hash to different shards, requiring queries to fan out across multiple partitions. Hash sharding also complicates resharding operations since most keys will map to different shards when the shard count changes.

## Range-Based Sharding Fundamentals
Range sharding partitions data by assigning contiguous ranges of shard key values to specific shards. For example, user IDs 1-1,000,000 go to shard A, 1,000,001-2,000,000 to shard B, and so on. This approach preserves locality for range queries and simplifies sequential key access patterns. However, range sharding is vulnerable to uneven distribution if the shard key doesn't distribute uniformly. If many users sign up within a narrow ID range, one shard may become significantly larger than others. Range queries are efficient, but the tradeoff is that hot spots are more likely to emerge when certain key ranges experience disproportionate traffic.

## Directory-Based Sharding Architecture
Directory-based sharding maintains a lookup table or metadata service that maps shard keys to their owning shards. When a request arrives, the system queries the directory to locate the appropriate shard before executing the operation. This approach decouples logical key ranges from physical shard assignments, enabling flexible resharding without rewriting all data. A centralized directory service becomes a potential bottleneck and single point of failure, so implementations typically use replicated metadata stores or caching strategies to mitigate latency. This method offers maximum flexibility for rebalancing but adds operational complexity and requires consistent directory updates during topology changes.

## Shard Key Selection Criteria
Choosing the right shard key is fundamental to sharding success. Ideal shard keys have high cardinality (many distinct values), distribute uniformly across shards, and align with query patterns. Common shard key choices include user ID, tenant ID, customer ID, or geographic region. A poor shard key choice like gender or country will create severe hot spots if distribution is skewed. The shard key should also be immutable or change infrequently, since updating it typically requires re-sharding data. Additionally, queries should ideally filter by the shard key to avoid fanning out across multiple shards. Selecting shard keys requires understanding both data characteristics and application access patterns.

## Resharding Operations and Strategies
Resharding redistributes data across a different number of shards, either to scale horizontally or to rebalance after discovering an unsuitable shard key choice. During resharding, each row must be reassigned to a new shard based on updated partition logic. The operation can occur online (with minimal downtime) or offline. Online resharding typically involves running both old and new shard assignments in parallel, migrating data in the background, and switching traffic once migration completes. This requires careful coordination to ensure no data is lost or duplicated. Resharding is expensive because it touches every row in the database, making it important to select shard keys carefully upfront to minimize resharding frequency.

## Hot Shard Identification and Detection
A hot shard receives disproportionate read or write traffic compared to peers, becoming a throughput bottleneck. Detection involves monitoring shard-level metrics including query latency, CPU utilization, disk I/O, and connection counts. Comparing these metrics across shards reveals outliers. Hot shards often emerge from uneven data distribution (range sharding with skewed keys), uneven access patterns (certain user segments generate more traffic), or both. Early detection is critical because a severely hot shard may degrade performance for the entire system if it becomes overloaded. Time-series monitoring and alerting should track per-shard metrics to catch hot spots before they impact application SLOs.

## Hash Function Selection and Consistency
The hash function used in hash-based sharding directly impacts data distribution and resharding complexity. Common candidates include consistent hashing (which minimizes key remapping during shard count changes) and modulo-based hashing (which is simpler but remaps most keys when shards are added or removed). Consistent hashing uses a virtual ring where each shard occupies multiple points; adding a shard only affects a fraction of keys. The tradeoff is increased implementation complexity. The hash function must be deterministic and produce uniform output distribution across the output range. Testing hash function distribution with real shard keys is essential to verify even spreading and identify potential pathological cases.

## Rebalancing Through Shard Splitting
Shard splitting divides an overloaded or unevenly-sized shard into two or more smaller shards, typically when a shard exceeds capacity. The split can be based on the shard key range or hash space partition. For range-based sharding, this might mean splitting a shard with IDs 1M-2M into shards with IDs 1M-1.5M and 1.5M-2M. The operation requires copying data, updating routing logic, and verifying consistency. Splitting is generally less disruptive than full resharding but still requires careful coordination. Multiple splits may be necessary during rapid scaling. The frequency of splitting depends on growth rate and initial shard count; undersizing initial shard counts leads to frequent splits.

## Rebalancing Through Shard Merging
Shard merging combines multiple underutilized shards into a single shard, reducing operational overhead and resource consumption. Merging becomes necessary when shards are created too aggressively or when traffic patterns shift. For instance, if three shards each hold only 10GB of data and receive minimal traffic, merging them into one shard reduces infrastructure costs. Merging requires reading data from multiple source shards, validating consistency, and decommissioning empty shards. The operation is the inverse of splitting and has similar coordination requirements. Merging is less common than splitting in scaling scenarios but important for cost optimization in declining workloads.

## Mitigating Hot Shards Through Data Replication
Replicating hot shard data across additional read-only replicas distributes read traffic without resharding. Each replica serves a subset of read requests, increasing aggregate throughput. This approach is most effective when the workload is read-heavy and the hot shard isn't write-saturated. Write traffic still concentrates on the primary shard, limiting this technique's effectiveness for write-heavy hot spots. Replicas introduce eventual consistency challenges and replication lag. This strategy is complementary to other hot shard mitigation approaches and works well as a tactical fix while longer-term resharding is planned.

## Mitigating Hot Shards Through Caching Layers
Caching frequently accessed hot shard data in a distributed cache (Redis, Memcached) reduces load on the shard by absorbing read traffic. This is effective when the hot shard serves a working set of data smaller than total shard size. Cache invalidation and consistency become operational concerns. If the hot shard is write-heavy, caching provides limited benefit unless the cache is updated on writes. Caching is typically a short-term mitigation while resharding or splitting is implemented. Monitoring cache hit rates helps determine if caching is effectively reducing shard load.

## Mitigating Hot Shards Through Request Routing
Sophisticated request routing can distribute traffic more evenly by directing requests to specific shard replicas or by applying local load balancing logic. Some systems use application-level routing that accounts for per-shard latency and capacity. This complements infrastructure-level load balancing and can redirect requests away from overloaded shards. However, routing cannot address data hotness from a single logical key receiving all traffic (like a popular user or account). Routing is most effective when hot shards are caused by uneven distribution of data volume rather than uneven request distribution to appropriately-sized shards.

## Shard Key Immutability and Update Constraints
Shard keys must remain stable or change infrequently because updates require re-sharding the affected row. If shard key changes are common, the system incurs constant resharding overhead. For example, using a user's geographic region as a shard key becomes problematic if users frequently relocate. Immutable identifiers like user ID or customer ID are more reliable. Some systems enforce shard key immutability at the schema level or application layer. When shard key updates are unavoidable, they must be handled as delete-insert operations with careful transaction handling to ensure consistency. Documenting shard key immutability requirements is essential for application developers.

## Cross-Shard Query Execution
Queries not filtered by shard key must execute across multiple shards (fan-out), retrieving results from each and aggregating them. Examples include range queries that span multiple key ranges or searches on non-shard-key fields. Fan-out queries are more expensive than single-shard queries because they increase latency (waiting for slowest shard response), network traffic, and compute load. Optimization strategies include early filtering on shards to reduce result sets before aggregation and parallel execution. Some systems use secondary indexes on non-shard-key fields to optimize certain cross-shard queries. The cost of cross-shard queries should factor into shard key selection.

## Directory Service for Metadata Management
A directory service (or metadata store) maintains the mapping of shard keys to physical shard locations. This service is critical for distributed system health, requiring replication for fault tolerance and typically strong consistency guarantees. Updates must be coordinated across replicas. Clients can cache directory information locally to avoid per-request directory lookups, trading off staleness for latency. Directory services must handle cache invalidation when shard topology changes. Examples include using a replicated key-value store, a distributed consensus system (etcd, ZooKeeper), or a custom metadata service. Directory availability directly impacts system availability, requiring careful design and monitoring.

## Shard Topology Management and Monitoring
Maintaining awareness of which shards exist, their capacity, health status, and load characteristics is essential for sharding operations. A topology management system tracks shard metadata, monitors shard-level metrics, and orchestrates topology changes. This system must detect shard failures, coordinate rebalancing, and enforce consistency during resharding. Monitoring should track metrics per shard including storage size, query latency, connection count, and error rates. Alerting on metric thresholds enables proactive intervention before problems cascade. Topology changes must be coordinated carefully to avoid inconsistency or data loss, often requiring state machine coordination across the distributed system.

## Eventual Consistency in Sharded Systems
Sharded databases often employ replication within each shard for high availability, introducing eventual consistency challenges. Updates to a shard's primary replica must propagate to replicas, creating a window where reads from different replicas may return different values. Applications must tolerate this inconsistency or use read-your-writes guarantees (reading from primary after writes). Shard-level replication lag varies, so different shards may be at different consistency points. This is distinct from consistency across shards, which is inherently limited in sharded architectures due to lack of global transactions. Understanding consistency semantics is critical for correct application design on sharded systems.

## Distributed Transactions Across Shards
Transactions spanning multiple shards are difficult to implement correctly because there is no global transaction coordinator with visibility into all shards. Two-phase commit (2PC) is the traditional approach but is slow and can block shards if a participant fails. Many modern sharded systems avoid multi-shard transactions, instead requiring application logic to handle cross-shard consistency. Some systems implement weaker consistency models like eventual consistency or causal consistency. When multi-shard transactions are necessary, careful design and thorough testing are essential to handle failure modes. The cost of cross-shard transactions argues for shard key selection that minimizes multi-shard operations.

## Shard Count Planning and Scaling
Initial shard count selection balances capacity headroom against operational complexity. Too few shards exhaust capacity quickly, necessitating frequent resharding. Too many shards increase operational overhead and per-shard data size may become too small to manage efficiently. Common approaches include sizing shards for 1-3 years of growth and adding shards as needed, or using a fixed shard count with overflow capacity. Shard count changes require resharding, which is expensive, so planning upfront reduces frequency. As the system scales, the cost-benefit of sharding must be continuously evaluated; very large shards may reach a point where resharding becomes prohibitively expensive.

## Handling Shard Failures and Recovery
When a shard becomes unavailable due to hardware failure or corruption, the system must either serve stale data from replicas or temporarily accept reduced availability. Shard recovery involves re-replicating data from replicas to new hardware and validating consistency. This process can be slow for large shards, prolonging unavailability. Design choices include maintaining multiple replicas per shard (increasing capacity costs), geographic replication across datacenters (increasing consistency complexity), or accepting temporary unavailability for critical shards. Automation is essential; manual recovery is too slow and error-prone. Regular testing of failure and recovery procedures ensures RTO and RPO targets are met.

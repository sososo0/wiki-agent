# Caching Strategies

## Cache-Aside Pattern
Cache-aside (also called lazy loading) is a pattern where the application code is responsible for loading data from the primary data store into the cache. On a cache miss, the application fetches the data from the source, stores it in the cache, and returns it to the caller. This pattern decouples the cache from the data layer and allows flexibility in cache implementation. The primary tradeoff is increased application complexity and potential latency on cache misses. Cache-aside works well when access patterns are unpredictable or when you want to cache selectively based on application logic. For example, a user profile service might check the cache for a user ID; if absent, query the database, populate the cache with a 1-hour TTL, and return the result.

## Read-Through Pattern
Read-through caches sit between the application and data source, transparently handling cache population on misses. When data is requested, the cache checks its store; on a miss, it automatically fetches from the underlying data source, caches the result, and returns it. This pattern simplifies application code since cache operations are transparent. The tradeoff is that the cache library or abstraction must understand your data source and loading logic. Read-through is useful when you have a consistent data source and want to minimize application-level cache management. Many cache-as-a-service solutions implement read-through by allowing you to define a data loading function that executes on cache misses.

## Write-Through Pattern
Write-through ensures data consistency by writing to both the cache and the primary data store synchronously before acknowledging the write to the client. This pattern guarantees that cached data is always consistent with the source of truth. The tradeoff is increased write latency since both stores must complete before returning. Write-through is appropriate for scenarios where consistency is critical and writes are not the primary bottleneck. For example, a financial transaction system might use write-through to ensure that balance updates are immediately visible in both cache and database, preventing stale reads in subsequent queries.

## Write-Behind Pattern
Write-behind (write-back) caching asynchronously flushes writes from the cache to the underlying data store. The application writes to the cache immediately and receives acknowledgment, while a background process eventually persists changes to the primary store. This pattern offers high write throughput and low write latency. The significant tradeoff is risk of data loss if the cache fails before changes are persisted, and temporary inconsistency between cache and source. Write-behind is suitable for use cases where some data loss is tolerable and throughput is critical, such as analytics event collection or non-critical operational metrics. Implementation requires careful handling of failure scenarios and batching strategies.

## Cache Invalidation Fundamentals
Cache invalidation is the process of removing or marking cached entries as stale when the underlying data changes. Invalidation can be explicit (application-triggered) or implicit (time-based or size-based eviction). The fundamental challenge is determining when and how to invalidate without over-invalidating (removing useful entries) or under-invalidating (serving stale data). Invalidation strategies range from simple time-based TTLs to event-driven invalidation systems. Proper invalidation is critical for maintaining application correctness. For example, when a product price is updated in an e-commerce system, the cached price entry must be invalidated to prevent customers from seeing outdated prices.

## Passive Invalidation with TTL
Time-to-live (TTL) invalidation removes cached entries after a fixed duration without explicit triggers. An entry expires automatically regardless of whether the underlying data has changed. This passive approach requires no external signaling and is simple to implement. The tradeoff is potential staleness (serving old data until TTL expires) and potential cache thrashing if TTL is too short. TTL works well for data with predictable change rates and high tolerance for eventual consistency. For example, a service might cache geolocation data with a 24-hour TTL since location mappings change infrequently. Setting TTL requires domain knowledge about acceptable staleness; too short wastes cache benefits, too long risks serving outdated information.

## Active Invalidation and Event-Driven Patterns
Active invalidation removes cache entries immediately when source data changes, triggered by explicit events or database notifications. This approach maintains fresher data and can prevent stale reads entirely. Implementation complexity increases significantly, requiring coordination between data stores and cache systems, such as event streaming or database triggers. The tradeoff involves operational overhead and potential race conditions if invalidation and updates are not properly sequenced. Event-driven invalidation works well for critical data where staleness is unacceptable. For example, an inventory system might publish "stock updated" events that trigger immediate cache invalidation, ensuring users see current availability within milliseconds.

## TTL Strategy: Short vs. Long Duration
Choosing TTL duration requires balancing staleness tolerance against cache efficiency. Short TTLs (seconds to minutes) minimize staleness but reduce cache effectiveness and increase backend load from frequent cache misses. Long TTLs (hours to days) maximize cache hits and reduce load but allow longer periods of serving stale data. The optimal TTL depends on data volatility, acceptable staleness, and system load characteristics. A practical approach is to set different TTLs for different data types: user profile data might use 1-hour TTL while static reference data uses 24-hour TTL. Monitoring cache hit rates and application performance helps identify whether TTL adjustments would improve efficiency.

## Variable TTL Based on Data Characteristics
Rather than fixed TTLs, variable TTL assigns different expiration times based on data type, update frequency, or business logic. Infrequently-changing data receives longer TTLs, while volatile data receives shorter ones. This approach optimizes cache efficiency by keeping high-value entries around longer while reducing staleness risk for critical data. Implementation requires logic to determine appropriate TTL at cache write time. For example, a news website might cache homepage content with 5-minute TTL but cache article text with 2-hour TTL since articles rarely change after publishing. Variable TTL strategies reduce the need for active invalidation while maintaining reasonable freshness guarantees.

## Thundering Herd Problem
The thundering herd (cache stampede) occurs when a cache entry expires simultaneously for many requests, causing all of them to miss and request the same data from the backend simultaneously. This creates a sudden load spike that can overwhelm the data source. The problem emerges when many clients share the same cache TTL and the underlying data is expensive to fetch. The impact ranges from degraded performance to complete backend failure. For example, if 10,000 users all cached a popular item with identical TTL, all 10,000 cache misses could hit the database at the same moment, causing connection pool exhaustion.

## Probabilistic Early Expiration
Probabilistic early expiration mitigates thundering herd by randomizing cache expiration times slightly. Rather than all entries expiring at exactly the same moment, each entry gets a small random TTL variation. This spreads out cache misses over time, distributing backend load more evenly. The tradeoff is slightly increased staleness variation and implementation complexity. The algorithm typically adds a random jitter (10-20% variation) to the base TTL, so if base TTL is 3600 seconds, actual expiration might be between 3240-3960 seconds. This simple technique significantly reduces synchronized spikes while preserving average cache freshness.

## Xfetch and Stale-While-Revalidate
Xfetch and stale-while-revalidate patterns mitigate thundering herd by serving cached data even after expiration while asynchronously refreshing it. When a request hits an expired entry, the system returns the stale data immediately and triggers a background refresh. This approach provides low latency and prevents stampedes by allowing one background process to refresh rather than many concurrent requests. The tradeoff is brief periods of serving knowingly-stale data. This pattern is effective for data where slight staleness is acceptable. For example, a rate limiting service might serve stale rate limit counters while asynchronously updating them, preventing cache-miss storms when limits reset.

## Locking and Single Requestor Pattern
The locking pattern prevents thundering herd by ensuring only one request fetches fresh data while others wait for the cached result. When a cache miss is detected, the first request acquires a lock, fetches data, and populates the cache. Other concurrent requests wait for the lock to release, then retrieve the populated cache. This eliminates redundant backend requests but introduces wait latency. Implementation requires careful lock management to avoid deadlocks and timeout handling. For example, in a user service, if 100 requests for the same user arrive during a cache miss, a lock ensures only one database query executes while others wait for the cache to be populated.

## Distributed Cache Invalidation
Distributed invalidation broadcasts invalidation signals across multiple cache instances, ensuring consistency across a cluster. When one cache invalidates an entry, other caches in the cluster receive notification and remove or mark it stale. Implementation methods include message brokers (Redis Pub/Sub, Kafka), database triggers, or polling mechanisms. The tradeoff is increased operational complexity and potential consistency windows where some nodes haven't yet received the invalidation. Distributed invalidation is essential for systems using replicated caches where a single cache node cannot serve all requests. For example, a web service with caches on 50 edge servers broadcasts invalidation messages when product data changes to keep all caches synchronized.

## Cache Warming Strategies
Cache warming pre-populates the cache with expected-to-be-needed data before serving user traffic, rather than waiting for cache misses to occur. Warming can be scheduled (daily warm-up before peak hours) or triggered by system events (service restart). This approach reduces initial cache miss rates and prevents early load spikes. The tradeoff is complexity in determining what data to pre-warm and resource usage during warming. Cache warming is valuable for systems with predictable access patterns or expensive computations. For example, an e-commerce site might warm the cache with top-100 products and promotions each morning, ensuring low latency when customer traffic begins.

## Cache Coherence in Multi-Layer Caching
Multi-layer caching systems (L1 application cache, L2 distributed cache, L3 database) require coherence mechanisms to prevent inconsistency where different layers serve conflicting versions. Coherence strategies include write-through to all layers, invalidation cascades, or versioning schemes. Maintaining coherence across layers adds complexity but prevents serious correctness bugs. The primary tradeoff is coordination overhead. For example, a system might implement invalidation where L2 cache invalidation automatically triggers L1 invalidation across all application instances, preventing stale application-local caches after data updates.

## Conditional Cache Invalidation
Conditional invalidation removes cache entries only when specific conditions are met, rather than invalidating all potentially-affected entries. For example, a user's "recommended products" cache might only invalidate when the user's preference profile significantly changes, not on every product update. This approach preserves cache efficiency by avoiding over-invalidation. Implementation requires defining conditions that accurately capture true dependencies. The risk is under-invalidation where conditions miss required invalidations. Conditional invalidation works well when invalidation dependencies are complex and some staleness is acceptable.

## Cache Size and Eviction Policies
Cache capacity is finite, so eviction policies determine which entries to remove when the cache is full. Common policies include LRU (Least Recently Used), LFU (Least Frequently Used), and FIFO (First In First Out). LRU evicts entries not accessed recently and generally performs well for typical workloads. LFU evicts infrequently-accessed entries, preserving high-value data. Different policies have different CPU costs and memory characteristics. The choice impacts both hit rates and performance. For example, a cache with 1GB capacity and 10-million entries might use LRU to automatically evict older entries, ensuring the most-recently-used data stays cached.

## Cache Consistency Models
Consistency models define guarantees about what data cached entries might contain relative to the source. Strong consistency guarantees cached data matches the source at all times (difficult with caches). Eventual consistency allows temporary divergence but guarantees convergence over time. Weak consistency offers no guarantees. Different application requirements demand different models. Strong consistency requires aggressive invalidation or write-through, increasing overhead. Eventual consistency is easier to implement but requires applications to tolerate stale reads. For example, a user profile cache might use eventual consistency (stale profile for a few seconds is acceptable) while a payment system uses strong consistency (account balance must always be accurate).

## Monitoring Cache Health and Hit Rates
Effective caching requires monitoring cache behavior through hit rates, miss rates, eviction rates, and latency. Hit rate (successful cache retrievals / total requests) indicates cache effectiveness; rates below 80% suggest configuration issues. Miss rates reveal access patterns and invalidation frequency. Eviction rates indicate whether cache size is appropriate. Monitoring helps identify whether TTL, size, or invalidation strategies need adjustment. For example, observing a 40% hit rate on a cache expected to achieve 90% indicates either insufficient capacity, overly-aggressive invalidation, or wrong TTL duration. Regular monitoring allows data-driven cache optimization rather than guesswork.

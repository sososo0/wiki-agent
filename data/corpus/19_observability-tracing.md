# Observability: Distributed Tracing

## Span Basics and Structure

A span represents a single unit of work within a distributed system, typically corresponding to a function call, RPC, database query, or other discrete operation. Each span contains metadata including a unique identifier, parent span reference, operation name, timestamps, and key-value tags. Spans are the fundamental building blocks of traces. Understanding span structure is essential because improper span creation or instrumentation can result in incomplete traces that obscure system behavior. The tradeoff involves instrumentation overhead—creating too many fine-grained spans increases memory and network overhead, while too few spans lose important context. For example, instrumenting every internal function call in a hot path may generate excessive spans, whereas instrumenting only service boundaries may miss critical performance bottlenecks within a service.

## Trace Context Propagation Standards

Trace context propagation refers to passing trace identifiers and metadata across service boundaries so that related spans can be correlated into a single logical trace. Standards like W3C Trace Context and OpenTelemetry define HTTP headers and formats for this propagation. Proper propagation is essential in microservice architectures where requests traverse multiple services. The tradeoff involves header overhead and compatibility with legacy systems that don't support standard propagation formats. When implementing propagation, you must decide which headers to include (traceparent, tracestate) and how to handle services that strip or modify headers. A concrete example: an HTTP request enters service A, which injects a W3C traceparent header before calling service B; service B extracts this header and includes it when calling service C, allowing all three services' spans to appear in one trace.

## Manual Instrumentation vs. Automatic Instrumentation

Manual instrumentation means explicitly adding tracing code to your application using a tracing SDK, while automatic instrumentation uses libraries and agents that intercept calls without source code changes. Manual instrumentation provides precise control over what gets traced and allows domain-specific context injection, but requires development effort and can clutter code. Automatic instrumentation reduces code changes and works with third-party libraries, but may miss application-specific operations or create unwanted spans. In practice, most systems use both: automatic instrumentation for standard operations (HTTP, database calls) and manual instrumentation for business logic. For example, you might rely on an HTTP instrumentation library to trace incoming requests automatically, while manually instrumenting a complex data processing pipeline to capture intermediate steps.

## Sampling Strategies and Rate Selection

Sampling reduces tracing overhead by recording only a subset of traces, essential for high-traffic systems where tracing every request is prohibitively expensive. Common strategies include head-based sampling (decide at request entry whether to trace the entire request), tail-based sampling (sample based on trace characteristics after completion), and probabilistic sampling (fixed percentage of requests). Head-based sampling is simpler to implement but risks missing interesting traces that become problematic later, while tail-based sampling captures more relevant data but requires buffering traces temporarily. The tradeoff involves accuracy versus cost—higher sampling rates provide more visibility but increase overhead. For example, a payment processing service might use head-based sampling at 10% for normal transactions but enable 100% sampling for failed transactions, or use tail-based sampling to always capture traces exceeding a latency threshold.

## Distributed Trace Identifiers and Unique IDs

Trace and span IDs are unique identifiers that correlate related spans into a single trace. A trace ID identifies the entire request flow across all services, while each span has a unique span ID and references its parent span ID. These IDs must be collision-resistant (typically 64 or 128-bit values), globally unique, or unique within a time window. Proper ID generation and propagation is fundamental to tracing—collision or reuse of IDs causes spans to be incorrectly grouped. Most tracing libraries use cryptographically random generation for IDs. The tradeoff involves ID size (longer IDs are more collision-resistant but add overhead) and generation performance. In practice, a typical trace ID is a 128-bit value represented as hexadecimal, and parent-child relationships are maintained through parent span ID references rather than creating a centralized registry.

## Trace-Log Correlation Techniques

Trace-log correlation links logs generated during a request with the corresponding trace data, allowing engineers to view logs and trace spans together. This typically involves injecting the trace ID into logging context so that log lines automatically include it. Correlation bridges the gap between two complementary observability tools—logs capture discrete events and messages, while traces capture request flow and timing. Implementation approaches include injecting trace ID into Mapped Diagnostic Context (MDC) in Java or context variables in Python, then configuring log formatters to include it. The tradeoff involves log volume and searchability—adding trace IDs increases log size, but significantly improves debuggability. For example, if a distributed trace shows latency in a database query, the corresponding logs (indexed by trace ID) reveal the exact SQL, connection pool state, and error messages that caused the slowdown.

## Baggage and Context Propagation

Baggage is application-defined metadata propagated across service boundaries alongside trace context. Unlike trace IDs which are tracing infrastructure, baggage carries business or operational context such as user ID, tenant ID, request priority, or feature flags. Baggage must be explicitly defined and propagated; standards like W3C Baggage define the format. Baggage is powerful for correlating traces with business context but introduces security and privacy concerns—sensitive data in baggage can leak across services or be logged unintentionally. The tradeoff involves utility versus size and security—excessive baggage increases header overhead and payload size, while insufficient baggage misses important context. A concrete example: a multi-tenant SaaS service propagates tenant ID in baggage, allowing traces to be filtered by customer; a feature experimentation platform propagates experiment assignment as baggage so traces can be grouped by experiment variant.

## Span Tagging and Attributes

Tags (or attributes) are key-value pairs attached to spans that provide additional context about the operation. Common tags include HTTP method, response status code, database query type, error information, or custom business metrics. Tagging enables filtering and aggregating traces—for example, finding all traces with status 500 or all database queries exceeding 100ms. The tradeoff involves cardinality and overhead—high-cardinality tags (like unique user IDs or exact query strings) can cause memory and indexing problems in tracing backends, while sanitized or low-cardinality tags reduce pressure but lose detail. Best practice is to tag with bounded-cardinality values; use sampling or separate storage for high-cardinality data. For example, tag with HTTP status code (low cardinality: 200, 404, 500) rather than full response body, and tag with database table name but hash long query strings.

## Critical Path Analysis Through Traces

Critical path analysis uses distributed traces to identify the sequence of operations that determines overall request latency. By examining a trace, you can visualize the timeline of all spans, identify which operations run serially (blocking the total latency) versus in parallel (not affecting latency), and pinpoint optimization targets. Traces make critical path visible in a way that local profiling cannot, because request latency often depends on remote service latencies rather than CPU time. The tradeoff involves trace depth and complexity—very deep traces with many parallel spans are harder to interpret, while traces lacking sufficient span boundaries miss critical path segments. For example, a trace showing a web request might reveal that latency is dominated by three sequential database queries (the critical path) while parallel logging and caching operations don't affect total time, immediately suggesting the optimization target.

## Span Events and Structured Annotations

Span events (also called logs or structured annotations) record discrete occurrences within a span, such as checkpoints, state transitions, or error conditions. Unlike tags which are static metadata about the span, events have timestamps and convey "something happened at this point during the span." Events are useful for capturing transient state or explaining why a span took time. The tradeoff involves span size and query complexity—excessive events inflate span data and make storage/retrieval slower, while too few events lose important diagnostic information. A typical approach is to reserve events for significant occurrences (e.g., cache miss, retries, circuit breaker activation) rather than logging every line of execution. For example, a retry span might have events timestamped as: "attempt_1_failed", "backoff_delay_start", "backoff_delay_end", "attempt_2_started", capturing the retry flow without instrumenting every millisecond.

## Tail-Based Sampling Implementation

Tail-based sampling defers the sampling decision until after a trace completes, allowing the decision to be based on trace characteristics (latency, error status, annotations). A tail-based sampler collects complete traces in a buffer, applies sampling rules, and exports only the sampled traces. This captures more relevant traces than head-based sampling—for example, automatically sampling all traces containing errors or exceeding latency thresholds—but introduces memory overhead and latency in exporting traces. Implementation complexity is higher: you must manage buffering, define sampling rules, and handle memory pressure if traces arrive faster than they can be processed. The tradeoff is between visibility and overhead; tail-based sampling provides better signal but requires more resources. A concrete example: a tail-based sampler configured to always sample traces with latency > 1s, any trace with an error span, and 1% of remaining traces, ensuring high-value traces are captured while managing overall volume.

## Probabilistic Sampling and Random Decisions

Probabilistic sampling makes a random decision at the trace entry point to sample the request with a fixed probability (e.g., sample 10% of traces). This is the simplest sampling strategy to implement and has minimal overhead, but cannot guarantee capture of rare or interesting events. Because the decision is made early and deterministically (same request always gets the same sampling decision), all downstream services can inherit it via trace context. The tradeoff is simplicity and predictability versus relevance—you get consistent sampling rates but may miss important traces. For example, probabilistic sampling at 1% on a system handling 100k requests/sec yields ~1000 sampled traces/sec, providing broad visibility into normal behavior but potentially missing individual rare failure modes. Probabilistic sampling works well as a baseline, combined with other sampling strategies for specific high-value cases.

## Dynamic Sampling Based on Request Attributes

Dynamic sampling adjusts the sampling probability based on request attributes (user, endpoint, environment, error status) rather than using a fixed rate. Rules might be: "sample 100% of admin endpoints", "sample 50% of checkout flows", or "sample 100% of failed requests". Dynamic sampling provides more control and can adapt to operational needs without code changes. However, it introduces rule complexity and potential overhead from attribute evaluation. Implementing dynamic sampling requires a configuration mechanism (centralized rules service or local config) and decision-making logic on the sampling path. The tradeoff involves flexibility versus operational burden—more rules capture more relevant traces but require ongoing maintenance. For example, a SaaS platform might sample 100% of requests from high-value customers, 100% of payment transactions, 10% of normal user requests, and 100% of errors, targeting sampling effort toward business-critical flows.

## Latency-Based Tail Sampling Rules

Latency-based tail sampling rules sample traces based on observed duration, ensuring that slow requests are always captured. Rules might be: "sample any trace > 500ms" or "sample p95+ latency traces for each endpoint". This strategy captures performance issues directly and is particularly useful for detecting slow degradation. Because latency is measured at trace completion, you see actual values rather than predictions. The challenge is defining latency thresholds appropriately—too low and you sample excessive traces, too high and you miss subtle slowness. Implementation requires buffering complete traces and evaluating latency before making the export decision. A concrete example: an API service with latency-based tail sampling might sample all traces exceeding 1 second, plus 10% of traces under 1 second, ensuring that performance regressions are always visible while maintaining affordable sampling rates for normal requests.

## Error-Based and Anomaly Sampling

Error-based and anomaly sampling prioritize sampling traces that indicate problems. Error sampling ensures 100% of traces containing errors or exceptions are captured, while anomaly sampling uses statistical models or heuristics to detect unusual behavior (unexpected latency increase, unusual attribute combinations). These strategies maximize the chances of capturing actionable data—engineers investigating issues find traces immediately. The tradeoff is implementation complexity and potential overhead—anomaly detection requires comparing traces against baselines or statistical models. Error sampling is straightforward to implement, while anomaly sampling requires careful tuning to avoid false positives. For example, error-based sampling with 100% capture of any trace containing a span with error status code ensures all failures are traceable; anomaly sampling might additionally capture traces with latency > mean + 2 standard deviations, catching performance degradation before errors occur.

## Correlating Traces to Metrics

While traces capture request-level details, metrics aggregate numerical measurements over time. Correlation between traces and metrics amplifies both: metrics reveal "what happened at scale", while traces explain "why it happened for this specific request". Implementation involves emitting metrics from trace data (e.g., request count, latency percentiles, error rate) or tagging metric events with trace IDs. Correlation allows engineers to see a metric spike, drill into sampled traces from that time window, and understand the root cause. The tradeoff involves metric cardinality—if you emit metrics with high-cardinality tags (user ID, endpoint), they become unmanageable; thus correlation typically uses low-cardinality aggregations in metrics with spot-checking via trace sampling. For example, detecting a latency spike in metrics, then querying traces filtered to that time window and endpoint to find traces showing the slowness, revealing a specific downstream service or query as the culprit.

## Trace Sampling at Multiple Levels

Systems often implement sampling at multiple points: head-based sampling at trace entry, local sampling rules in each service, and additional sampling at the tracing backend. Multi-level sampling provides defense-in-depth against excessive trace volume, where each layer applies its own rules. However, this can lead to unintuitive behavior—a trace might be dropped at the service level before reaching the backend, even if the backend's sampling rules would have kept it. The tradeoff involves control and predictability versus simplicity; coordination between levels is necessary. A best practice is to use head-based sampling to make the core decision, allowing all services to inherit it, and avoid additional sampling points unless necessary. For example, entry service decides to sample (propagates decision via trace context), service B and C forward the sampled trace, and the backend applies tail-based sampling only to unsampled traces to catch errors, ensuring head-based decisions are respected.

## Trace-Log Filtering and Sampling

Beyond correlation, logs and traces can be filtered together—for example, querying for logs from traces matching certain criteria, or viewing logs only from sampled traces. This filtering helps manage log volume; if you're only analyzing sampled traces, viewing logs from unsampled requests is unnecessary. Implementation involves ensuring logs include trace context (trace ID, span ID) and configuring log indexing to support filtering by trace fields. The tradeoff involves index size and query flexibility—indexing all logs by trace ID increases storage and index overhead, while sampling logs based on trace decisions reduces storage but loses detail for unsampled traces. A common approach is to include trace IDs in all logs (minimal overhead), sample which logs to index based on trace sampling decisions, and reconstruct unindexed logs by re-parsing raw log files if needed. For example, index logs from all sampled traces, keeping rich detail; defer indexing logs from unsampled traces until needed for specific investigations.

## Root Cause Analysis Using Trace Patterns

Distributed traces reveal patterns that indicate root causes: cascading latencies (child spans each add latency in series), resource exhaustion (growing span count or timing at same lock point), external dependency issues (consistent timeout latencies), and retry storms (repeated span execution). By visualizing trace patterns, engineers can quickly narrow the root cause hypothesis. For example, if multiple traces show the same sequence of operations with consistent latency from a particular service, the cause is likely in that service; if traces show increasing latency correlated with increasing span count, the cause is likely queueing or lock contention. The challenge is identifying patterns at scale—manual visual inspection works for individual traces but doesn't discover systemic patterns. Tools that aggregate trace data and highlight statistical anomalies help surface root causes. This is particularly powerful combined with other observability data (metrics, logs) to validate hypotheses.

## Trace Sampling Bias and Blind Spots

Sampling traces always risks missing important events. Head-based probabilistic sampling creates biased samples—slow requests are indistinguishable from fast ones during the decision, so sampling artifacts don't correlate with performance. However, other sampling strategies can introduce bias: latency-based sampling overrepresents slow requests (skewing latency distributions), error-based sampling overrepresents failures. Understanding bias is essential for interpreting sampled trace data correctly—if 100% of errors are sampled but 1% of successes are sampled, observed error rates in traces will be misleadingly high. Mitigating blind spots involves: combining multiple sampling strategies, using statistical adjustments to account for bias, or maintaining unsampled metrics alongside sampled traces. For example, if you implement 1% head-based sampling plus 100% error sampling, be aware that error rates calculated from traces are inflated; use separate error rate metrics (unsampled) as ground truth while using traces for error investigation.

# Rate Limiting Algorithms

## Token Bucket Overview
The token bucket algorithm allows requests up to a configured rate by issuing tokens at regular intervals. Tokens accumulate in a bucket up to a maximum capacity. Each incoming request consumes one token; if no tokens remain, the request is rejected or queued. The algorithm is defined by two parameters: the refill rate (tokens per second) and bucket capacity (maximum tokens). This design permits burst traffic up to the bucket size while maintaining an average rate limit. Token bucket is widely used in API gateways and rate limiting services because it handles both sustained and bursty traffic patterns elegantly. The main tradeoff is implementation complexity compared to simpler algorithms, though the logic remains straightforward for single-machine deployments.

## Token Bucket Implementation Details
Implementing token bucket requires tracking the timestamp of the last refill and current token count. On each request, calculate elapsed time since last refill, add (elapsed_time × refill_rate) tokens, cap at maximum capacity, then decrement by request cost. For precision, use floating-point arithmetic or scale tokens to integer units. Thread-safety requires locks or atomic operations in concurrent environments. A practical example: a bucket with 10 tokens maximum and 2 tokens/second refill allows immediate bursts of 10 requests, then 2 requests per second thereafter. Implementation can be done with in-process memory for single instances or backed by Redis for distributed scenarios.

## Leaky Bucket Algorithm
The leaky bucket algorithm processes requests at a constant rate by maintaining a queue. Requests are added to the queue; a background process removes items at a fixed rate (the leak rate). If the queue reaches capacity, new requests are rejected. This guarantees a smooth, constant output rate regardless of input variability. Leaky bucket is suitable when you need strict traffic shaping and cannot tolerate bursts. The primary advantage is output rate predictability. The disadvantage is that it cannot prioritize requests or handle sudden legitimate spikes well—all excess traffic is rejected even if temporary capacity exists. Leaky bucket is less common in modern systems than token bucket because most use cases benefit from burst tolerance.

## Fixed Window Counter Algorithm
Fixed window counter divides time into fixed intervals (windows) and counts requests per window. At the start of each window, the counter resets to zero. Requests are allowed until the limit is reached. This is the simplest rate limiting approach and requires minimal state: only a counter and window start timestamp. However, it has a significant flaw: the "boundary problem." A client can exploit window edges to exceed the intended rate by making requests at the end of one window and the start of the next. For example, with a 60-second window and 100 request limit, a client could make 100 requests at second 59 and another 100 at second 60, totaling 200 in two seconds. Use fixed window counter only for non-critical limits or when the boundary problem is acceptable.

## Sliding Window Log Algorithm
Sliding window log maintains a timestamp record of each request within the current window. When a new request arrives, remove all timestamps older than the window size, then check if the count of remaining timestamps is below the limit. This approach eliminates the boundary problem because the window truly slides over time rather than resetting at fixed intervals. The accuracy is perfect—the algorithm enforces exact rate limits over any one-second (or other unit) period. The tradeoff is memory overhead: storing timestamps for every request can consume significant space at high request rates. A system with 100,000 requests/second over a 60-second window requires storing up to 6 million timestamps. Sliding window log is practical for lower-traffic systems or scenarios where accuracy is critical.

## Sliding Window Counter Algorithm
Sliding window counter is a hybrid approach combining fixed windows with sliding window concepts. It maintains counters for the current and previous window, then calculates allowed requests by weighting the previous window's traffic based on overlap with the current window. The formula is: allowed = limit × (1 - overlap_ratio) + previous_count × overlap_ratio. This method requires only two counters instead of a full timestamp log, dramatically reducing memory usage. The accuracy is approximate but typically within 10-15% of true sliding window behavior. Implementation is straightforward and performant. The tradeoff is complexity compared to fixed window and some accuracy loss compared to sliding window log. Use sliding window counter when you need better accuracy than fixed window but cannot afford sliding window log's memory overhead.

## Token Bucket for Burst Handling
Token bucket excels at allowing controlled bursts while maintaining average rate limits. The burst capacity (bucket size) determines how many requests can be handled immediately. With a refill rate of 5 tokens/second and capacity of 50, a client can send 50 requests at once, then must wait for refill. This is valuable for real-world traffic patterns where requests arrive in batches. Configure burst capacity based on acceptable delay: larger capacity trades memory for better immediate throughput. In practice, set burst capacity to 1-2 seconds of traffic at the limit rate. For example, a 1000 request/second limit might have 1000-2000 token capacity, allowing 1-2 seconds of burst before throttling. This prevents rejecting legitimate batch workloads while still controlling sustained load.

## Leaky Bucket for Traffic Shaping
Leaky bucket is intentionally designed for traffic shaping—smoothing variable input into constant output. Unlike token bucket which permits bursts, leaky bucket enforces strict rate consistency. Requests queue until the leak rate processes them. This is valuable when downstream systems require predictable, constant load. For example, a message processing pipeline might use leaky bucket to ensure a database receives writes at a steady 500 requests/second rather than bursty traffic that causes performance dips. The queue acts as a buffer absorbing input variance. Requests exceeding queue capacity are rejected; monitor queue depth to detect backpressure. The tradeoff is potential request rejection and added latency from queueing, which is acceptable when output consistency is paramount.

## Fixed Window Counter Edge Cases
Fixed window counter's boundary problem manifests when clients coordinate requests across window boundaries. With a 1-minute window and 1000 limit, a client making 1000 requests at second 59 and another 1000 at second 60 consumes double the intended capacity in two seconds. This violates the rate limit's spirit. However, the boundary problem severity depends on window size and limit magnitude. A 1-hour window has less exploitable edge than a 1-second window. For non-critical limits (e.g., UI rate limiting where rejection is merely inconvenient) or when windows are large relative to typical request rates, the boundary problem is negligible. For revenue-critical limits (e.g., billing-sensitive APIs), avoid fixed window counter entirely. Testing should explicitly check behavior at window boundaries to assess impact.

## Sliding Window Log Memory Considerations
Sliding window log's memory cost is proportional to request rate and window size: memory ≈ request_rate × window_size × bytes_per_timestamp. At 10,000 requests/second over a 60-second window with 8 bytes per timestamp, expect ~4.8 MB per rate limit bucket. With many clients or tenants, this multiplies quickly. Storage can be optimized by storing only minute-level precision (removing sub-second granularity) or using compressed formats. For distributed systems, centralizing timestamps in Redis requires network round-trips per request. Consider sliding window log only when: (1) request rates are moderate, (2) accuracy is critical, or (3) memory is not a constraint. For high-traffic systems, sliding window counter or token bucket are more practical.

## Distributed Rate Limiting Challenges
Distributed systems cannot store rate limit state locally on each server; clients might hit different servers, causing limits to be enforced per-server rather than globally. For example, with two servers and a 100 request/minute limit, a client could make 100 requests to each server, totaling 200. Solutions require coordinated state: (1) centralized storage (Redis), (2) consensus protocols, or (3) probabilistic algorithms. Centralized storage is most common; Redis handles thousands of rate limit checks per second. The tradeoff is latency (network round-trip required per request) and availability (if Redis is down, rate limiting fails). Some systems accept weakly-consistent rate limiting using local state and periodic syncing, useful when occasional overage is acceptable. Choose based on consistency requirements and tolerance for brief rate limit exceedance.

## Redis-Based Distributed Rate Limiting
Redis is the standard backend for distributed rate limiting due to its in-memory speed, atomic operations, and high throughput. Implement token bucket by storing token count and last-refill timestamp as Redis keys, using Lua scripts for atomic updates. Implement sliding window log by storing request timestamps in a Redis sorted set, removing old entries in a single operation. Redis supports thousands of concurrent rate limit checks per second with sub-millisecond latency. Key considerations: (1) use Redis pipelining to batch multiple client checks, (2) use TTL on keys to auto-expire old data, (3) use a Redis cluster for high availability and partition by client ID for scalability. Monitor Redis memory usage and eviction policies. The tradeoff is operational overhead of running Redis versus simplicity of having a single source of truth for rate limit state.

## Per-Tenant Rate Limiting
Multi-tenant systems require independent rate limits per tenant, with possibly different limits for different tenant tiers. Implement by keying rate limit state by tenant ID: e.g., `rate_limit:{tenant_id}:tokens` in Redis. This isolates quotas so one tenant's traffic doesn't affect another. Tiering allows different limits for free vs. premium tenants. Example: free tier gets 1000 requests/day, premium gets 100,000. Monitor per-tenant usage to detect abuse or misconfiguration. Challenges include: (1) ensuring tenant IDs are correctly identified (authentication), (2) handling tenant-specific edge cases (burst limits vary by tier), (3) billing integration if limits tie to revenue. Implement gradual degradation: when a tenant approaches limits, return HTTP 429 with `Retry-After` header. Some systems enforce soft limits (warnings) versus hard limits (rejections) based on tier.

## Token Bucket with Different Request Costs
Standard token bucket assumes each request costs one token. Extended implementations allow variable costs: a read operation costs 1 token, a write costs 5, a bulk operation costs 100. This reflects real-world resource consumption differences. Implement by checking if available_tokens >= request_cost before allowing the request. Clients can query cost before sending requests to handle rejection gracefully. This increases fairness by preventing cheap-operation clients from monopolizing limits. Tradeoff is client complexity; clients must be aware of costs and plan accordingly. Example: a data export API charges more tokens for larger exports, incentivizing clients to batch requests efficiently. Be cautious with highly variable costs; large-cost requests might frequently exceed available tokens, causing rejection. Consider guaranteeing minimum available capacity for any single request.

## Rate Limiting Strategies for APIs
API rate limiting typically combines multiple strategies: per-IP limits, per-API-key limits, and global limits. Per-IP prevents single machines from overwhelming the service; per-key limits fairly allocate quota to authenticated users; global limits protect backend infrastructure. Implement using layered checks: reject if any layer's limit is exceeded. Use tiered limits: anonymous users get lower limits than authenticated users. Return informative HTTP headers: `RateLimit-Limit`, `RateLimit-Remaining`, `RateLimit-Reset` indicate current quota state. On exceeding limits, return HTTP 429 (Too Many Requests) with `Retry-After` header suggesting when to retry. Monitor limit violations; spikes indicate attacks or misconfiguration. Adjust limits based on observed patterns: if 99% of traffic is below 100 requests/minute, set limits at 500 to avoid false positives while still protecting infrastructure.

## Adaptive Rate Limiting
Static rate limits don't adapt to system load. Adaptive rate limiting adjusts limits dynamically based on backend health, CPU, memory, or request queue depth. When system load is low, increase limits; when load is high, decrease them. Implement using feedback: measure latency, error rates, or resource usage, then adjust limit parameters. For token bucket, lower the refill rate when overloaded. For fixed window, reduce the limit count. This prevents cascading failures during load spikes; by lowing limits, the system sheds excess traffic before becoming critically overloaded. Tradeoff is complexity and risk of oscillation if adjustment is too aggressive. Use gradual changes (e.g., 10% decrease per minute) to avoid thrashing. Adaptive limiting is particularly valuable for systems with variable workload patterns or where infrastructure capacity changes over time.

## Rate Limiting for Background Jobs
Background job systems benefit from rate limiting to prevent resource exhaustion. Unlike request-response APIs, background jobs can have large latency variance and unpredictable duration. Rate limit job submission (requests/second) or concurrency (simultaneously executing jobs). Token bucket suits submission limits; leaky bucket suits concurrency limits. Example: allow 10,000 job submissions per minute but execute at most 100 concurrent jobs. This prevents a spike in submissions from overwhelming workers. Use queue depth as feedback: if queue depth exceeds threshold, lower submission limits. Monitor per-job-type limits separately if job types have different resource costs. Implement backpressure: when limits are hit, return feedback to job submitters (e.g., HTTP 429 or message broker nack) so they can retry later rather than queuing unbounded. This prevents queue explosion and out-of-memory errors.

## Rate Limiting Metrics and Monitoring
Instrument rate limiting to detect issues: count rejected requests, track limit violations per client, measure latency added by limit checks. Use time-series metrics to identify trends. Alert on sudden violation spikes (possible attack) or systematic violations (possible legitimate limit misconfiguration). Expose metrics per-tenant and per-endpoint to pinpoint problems. Example: if one endpoint's rejection rate jumps from 0.1% to 5%, investigate whether traffic pattern changed or limit configuration is wrong. Track the distribution of requests over time to validate that limits match actual usage. Audit logs should record decisions on requests near or at limits to debug client complaints. Use these metrics to refine limits: if limits are consistently hit but no rejections occur, limits are appropriately tight; if violations are rare, limits may be unnecessarily strict.

## Rate Limiting Failures and Fallbacks
Rate limit enforcement can fail: centralized storage (Redis) becomes unavailable, clocks skew, or bugs cause miscalculation. Define graceful degradation: fail open (allow requests) or fail closed (reject requests)? Most systems fail open because rejecting valid traffic is worse than temporarily exceeding limits, assuming failures are brief. However, fail closed is acceptable for critical security limits. Implement circuit breakers: if rate limit checks fail repeatedly, stop checking temporarily to avoid cascading failures. Cache rate limit decisions: if Redis is unavailable but was recently reachable, use cached state (slightly stale) rather than failing. Implement client-side rate limiting as fallback: clients self-limit based on 429 responses, reducing server-side work. Combine strategies: assume most clients respect rate limits, so server-side limits catch only abusive clients. Monitor fallback paths—if they activate frequently, underlying issues need investigation.

# Graceful Degradation

## Circuit Breaker Pattern
A circuit breaker is a state machine that prevents cascading failures by stopping requests to a failing service. The circuit operates in three states: closed (normal operation), open (requests fail immediately without calling the service), and half-open (limited requests allowed to test recovery). When error rates exceed a threshold, the circuit opens, protecting downstream systems from wasted resources. After a timeout period, the circuit transitions to half-open to test if the service has recovered. This pattern is essential in microservice architectures where one service failure can propagate through the system. The tradeoff is added latency from circuit breaker checks and potential staleness if the service recovers while the circuit remains open. Libraries like Resilience4j and Hystrix provide battle-tested implementations.

## Timeout Configuration
Setting appropriate timeouts at each layer of a distributed system prevents requests from hanging indefinitely and consuming resources. Timeouts should be configured at the network level (TCP timeout), application level (request timeout), and circuit breaker level (timeout before opening). Conservative timeouts ensure failed requests fail fast, but overly aggressive timeouts cause unnecessary failures during legitimate slow responses. Each tier should have progressively shorter timeouts than layers below it, preventing a cascade of waiting requests. A common pattern is setting a downstream service timeout shorter than the timeout offered to callers. Timeout configuration requires monitoring actual latency percentiles and adjusting based on operational needs rather than guessing.

## Fallback Cache
When a service becomes unavailable, serving stale data from cache is often preferable to returning an error. Fallback caches can be populated during normal operation and returned when the primary service fails. This approach works well for reference data, user profiles, and other information that tolerates some staleness. The tradeoff is increased cache storage costs and the complexity of distinguishing between fresh and fallback data when needed. TTLs on fallback cache entries should be longer than normal cache TTLs to maximize availability. This pattern is particularly effective for read-heavy services where slight data staleness is acceptable; it's inappropriate for transactional or time-sensitive operations.

## Read-Only Degradation
When write operations fail due to service degradation, allowing read-only access maintains partial functionality and user visibility. This requires separating read and write paths at the application level and implementing logic to detect when writes are unavailable. Users can still view data even if they cannot make changes, providing significant value in many applications. Implementing this pattern requires clear communication to users about the degraded state and preventing confusion when writes silently fail. Read-only mode is particularly useful during database failovers, replication lag issues, or when writes are failing due to insufficient resources. The primary challenge is handling user expectations and preventing data corruption from retried writes.

## Asynchronous Request Queueing
Instead of failing immediately when a service is overloaded, requests can be queued asynchronously for later processing. This converts a hard failure into a degraded-but-functional state where work completes with higher latency. Message queues like RabbitMQ, Kafka, or SQS act as buffers between producers and consumers. The tradeoff is that users must tolerate delayed responses and the system must handle cases where queued work cannot eventually be processed. Queueing works well for background jobs, notifications, and non-critical data updates. This pattern requires careful monitoring of queue depth to detect when the system cannot keep up with demand and alert operators before the queue grows unbounded.

## Reduced Consistency Levels
Some systems can operate at lower consistency levels during degradation. For example, eventual consistency instead of strong consistency, or returning data that hasn't been fully synchronized across replicas. This allows the system to serve requests when the strong consistency requirements cannot be met. The primary tradeoff is potential conflicts or stale reads that applications must handle gracefully. This pattern is most applicable to systems that can tolerate temporary inconsistency, such as social media feeds or analytics dashboards. Operators must clearly document which consistency level is in effect and applications must either accept the reduced guarantees or fail explicitly rather than silently serving inconsistent data.

## Feature Flag Disabling
Feature flags allow specific features or endpoints to be disabled without redeploying code. During degradation, features that consume significant resources or depend on failing services can be disabled to preserve system capacity for critical operations. This requires comprehensive feature flag infrastructure that can be toggled in real-time. The implementation complexity depends on how deeply the feature is integrated; features designed with toggleability in mind can be disabled cleanly, while features bolted onto existing code paths are difficult to turn off. Feature flags provide a powerful tool for graceful degradation but should not be confused with dependency injection or proper separation of concerns; they work best for truly optional functionality rather than core system behavior.

## Resource Shedding Strategies
Under extreme load, systems can shed non-critical requests to preserve capacity for important operations. Common strategies include rejecting requests with lower priority, dropping requests from less valuable users, or reducing sample rates for telemetry. This ensures the system remains responsive for critical work rather than degrading uniformly. Implementing resource shedding requires defining priority or importance and being transparent about what is being shed. The ethical implications must be considered: a service should document that some requests may be dropped and provide an explicit mechanism for users to opt into best-effort or priority tiers. Load shedding should be a last resort after other degradation strategies; it's most appropriate for services where uneven treatment is acceptable and predictable.

## Degraded API Response Format
APIs can return partial or simplified responses when full data is unavailable. For example, returning a user object without nested details, or omitting computed fields when the computation service is down. Clients must be designed to handle missing or null fields gracefully. This requires clear API contracts documenting which fields may be absent during degradation and what clients should do in those cases. The alternative of returning an error prevents any client functionality, while partial responses allow partial functionality. This pattern requires upfront API design to identify which fields are optional and which are critical. Version negotiation or feature detection can help clients determine which fields are present in each response.

## Graceful Load Shedding with Percentile Queues
Instead of rejecting all excess requests uniformly, a system can accept a percentage of requests above its normal capacity while rejecting the remainder. This can be implemented using token bucket algorithms, fair queuing, or random sampling. Percentile-based shedding is fairer than first-come-first-served rejection during overload since it doesn't penalize requests that arrived slightly later. The tradeoff is that some requests deterministically fail while the system has spare cycles. This approach works well when user experience degrades gracefully as success rate drops; it's inappropriate for applications where users expect 100% success or none. Monitoring must clearly distinguish between dropped requests and legitimate failures.

## Fallback to Default Values
When a service provides enhanced data or computed values that aren't available, returning sensible defaults allows basic functionality to continue. For example, returning a generic recommendation when a personalization service is down, or using standard pricing when a dynamic pricing service fails. This pattern preserves the core user experience while reducing richness. Defaults should be chosen to be safe and inoffensive rather than optimized, since they're used as a fallback. The system must clearly distinguish between real and default values when this matters for audit or compliance. This approach is most effective when the enhanced data provides incremental value rather than being essential to the feature.

## Bulkhead Pattern
The bulkhead pattern isolates failures by partitioning resources into separate pools so that failure in one area doesn't consume all resources. For example, dedicating separate thread pools for different request types or isolating different customer workloads onto different hardware. If one partition becomes saturated or fails, other partitions continue operating normally. The tradeoff is increased complexity and potential underutilization if partitions are sized conservatively. Kubernetes node selectors and resource quotas provide infrastructure-level bulkheads; thread pools and separate service instances provide application-level bulkheads. The pattern is essential in multi-tenant systems and when preventing cascading failures across different service types.

## Adaptive Timeout Adjustment
Instead of fixed timeouts, some systems dynamically adjust timeout values based on observed latency percentiles and system load. Under high load, timeouts might be shortened to fail fast; during stable periods, longer timeouts allow time for slow responses. This requires continuous measurement of latency distribution and automated adjustment of timeout configurations. The complexity is implementing adjustment logic that doesn't oscillate or cause excessive failures. This pattern is useful for systems with variable load patterns where a fixed timeout is either too aggressive or too lenient. Care must be taken to ensure timeout adjustments are monotonic and don't create thrashing behavior where timeouts keep changing.

## Dependency Isolation with Separate Deployments
Isolating services onto separate deployments prevents a single service failure from bringing down dependent services. This includes running each service in its own process, container, or VM rather than sharing resources. Isolation increases operational complexity and resource overhead but provides strong failure boundaries. The pattern is fundamental to microservice architecture and is typically paired with circuit breakers and timeouts. Shared resource pools (databases, caches, queues) can still create coupling and must be monitored separately. Isolation is not a substitute for graceful degradation logic but a prerequisite that makes degradation strategies effective.

## Graceful Service Shutdown
When a service must shut down, gracefully draining in-flight requests prevents data corruption and provides a better user experience than abrupt termination. A graceful shutdown stops accepting new requests, waits for in-flight requests to complete within a timeout period, then terminates. This requires implementing shutdown hooks and coordinating with load balancers to remove the instance from rotation. Kubernetes provides native support through preStop hooks and terminationGracePeriodSeconds. The tradeoff is that shutdown takes longer and some long-running requests may not complete within the grace period. This pattern is essential for zero-downtime deployments and rolling updates in distributed systems.

## Monitoring and Alerting for Degradation States
Effective graceful degradation requires comprehensive monitoring to detect when the system enters degraded states and alert operators. Metrics should track error rates, latency percentiles, queue depths, circuit breaker state, dropped requests, and cache hit rates. Alerts should trigger before the system reaches critical states, giving operators time to respond. The challenge is tuning alert thresholds to avoid false positives while catching real issues early. Degradation often involves multiple metrics changing simultaneously; correlation analysis or composite alerting rules can identify patterns better than individual metrics. Clear alerting and runbook documentation are essential so operators can respond appropriately to degradation events.

## Request Deduplication During Retries
When clients retry failed requests, the service may process the same request multiple times. Deduplication using request IDs ensures that retried requests produce idempotent results without duplicating side effects. This is particularly important for non-idempotent operations like payments or inventory updates. Deduplication requires storing request IDs and their results for a window of time sufficient for any reasonable retry pattern. This adds complexity and storage overhead but prevents data corruption from retried requests. The pattern is essential when combining graceful degradation with client-side retries, since degradation scenarios naturally increase retry rates.

## Cascading Timeout Values
In a chain of services, each timeout should be shorter than the timeout from its caller to ensure failures are detected and reported quickly. If Service A calls Service B with a 10-second timeout, and Service B calls Service C, then Service C should have a timeout shorter than 10 seconds (accounting for Service B's overhead). This prevents a scenario where A waits for B to timeout, then B waits for C to timeout, causing slow failure cascades. Implementing cascading timeouts requires explicit timeout propagation through request contexts or headers. Without this pattern, timeout values accumulate through the call chain and failures take longer to surface. This is particularly important in microservice architectures with deep call chains.

## Partial Index Serving
In search or database systems, a service can continue operating on partial indexes when full indexes are unavailable. For example, serving search results from a recent shard when older shards are offline, or returning results from a subset of fields when field indexes fail. This allows the service to degrade to reduced scope rather than complete failure. The tradeoff is that results are incomplete or don't cover the full dataset, which must be communicated to users. This pattern requires index design that supports partial serving and clear labeling of when results are partial. It's particularly useful in large-scale search systems where full index availability is difficult to guarantee.

## Graceful Memory Pressure Handling
When a service approaches memory limits, it can proactively shed cache entries, reduce batch sizes, or reject new requests before triggering out-of-memory conditions. Monitoring heap usage and other memory metrics allows the service to enter a degraded state rather than crashing. This can involve implementing custom memory monitors or using language runtime capabilities like GC tuning. The pattern prevents cascading failures from process crashes and allows the service to degrade controllably. The tradeoff is increased complexity in memory management and potential increased CPU usage from more frequent garbage collection. This is particularly important for long-running services that operate near memory limits.

# 파일+git을 DB로 쓰는 프로토타입 — 결론

`python experiments/file_git_store/demo.py` 실행 결과를 바탕으로 정리. 목적은
SQLite의 두 가지 한계(① 단일 writer 락, ③ 복제/네트워크 접근 불가)가 파일+git으로
**실제로** 풀리는지 코드로 확인하는 것 — `core/wiki_store.py`(운영 코드)는 전혀
건드리지 않았다.

## ① 단일 writer 락 — 부분적으로 해결, 직렬화 지점이 옮겨갈 뿐

- **파일쓰기 자체는 완전히 락프리다.** 시나리오 1: 다른 entry_id 10개를 스레드
  10개로 동시에 써도 0.002초에 전부 성공(에러 0건). SQLite였다면 전부 같은 DB
  파일 락을 다퉜을 상황 — 이 부분은 명확한 개선.
- **하지만 git commit은 그 자체로 새로운 직렬화 지점이다.** 시나리오 2: 같은
  파일쓰기+커밋을 10개 스레드로 동시에 하면 **9/10이 `.git/index.lock` 충돌로
  실패**(실측, exit status 128). 즉 "파일+git"이 단일 writer 문제를 통째로
  없애는 게 아니라, **잠금의 위치를 DB 파일에서 git index로 옮길 뿐**이다.
- **해법은 있다 — 배치 커밋.** 시나리오 2b: 파일쓰기는 그대로 병렬로 두고,
  커밋만 사이클당 1번으로 모으면(`commit_changes`를 한 번만 호출) 충돌이
  사라진다. 이건 이미 검토했던 "전용 writer 큐"(대안 C) 패턴을 커밋 단계에만
  적용한 것과 같다 — `run_update_cycle.py`처럼 "여러 변경을 모아 한 번에
  처리"하는 배치 작업 구조와 자연스럽게 맞는다.

**결론**: 실시간 다중 요청이 "즉시 커밋"을 요구하는 구조(예: 데모 채팅처럼 매
요청마다 즉시 영속화)라면 SQLite보다 별로 나아지지 않는다. 반면 이 프로젝트의
실제 쓰기 경로(`run_update_cycle.py`/`ingest_doc.py`처럼 배치로 도는 파이프라인)
구조에는 잘 맞는다 — 한 사이클 동안의 모든 변경을 파일로 쌓고 끝에 커밋 1번.

## ③ 복제/네트워크 접근 — 해결

- `git log --follow`가 `wiki_store.py`의 정수 `version` 컬럼 없이도 "몇 번,
  언제, 왜 바뀌었는지"를 그대로 보여준다(시나리오 3, 실제로 3개 커밋 확인).
- `git clone`이 복제를 그대로 대신한다(시나리오 3, 별도 디스크에 클론해 파일
  존재 확인) — 별도 복제 메커니즘을 만들 필요가 없다. `git push`/`pull`로
  원격 동기화도 거의 공짜.

## 검색 — DB 없이도 기존 로직 재사용 가능

`core/retrieval.py`의 `hybrid_search`/`_dense_rank`는 처음부터 SQLite/FTS5와
무관한 순수 함수였다(entries를 list[dict]로만 받음) — 파일 기반 entries에
그대로 적용해 검색 결과가 나왔다(시나리오 4). 잃는 건 BM25/FTS5 키워드 검색
**색인**뿐 — 이번엔 표준 라이브러리만으로 만든 단순 토큰 겹침 랭커로 대체했고,
실제로 쓰려면 별도 키워드 색인(예: 파일 메타데이터 캐시 + 간단한 inverted
index)을 새로 만들어야 한다.

## 실제로 마이그레이션한다면 추가로 필요한 것

1. **배치 커밋 큐** — 위 ①의 해법을 `run_update_cycle.py`/`ingest_doc.py`/
   데모의 즉시-쓰기 경로(피드백 제출 등) 각각에 맞게 설계해야 함. 즉시 영속화가
   꼭 필요한 경로(예: 사용자 피드백)는 여전히 어딘가에서 직렬화가 필요하다.
2. **검색 색인 영속화** — 지금은 매번 전체 디렉터리를 스캔(`list_entries`)+
   재인코딩. 코퍼스가 커지면 SQLite의 FTS5/인덱스가 했던 일을 다시 만들어야
   한다(파일 기반 inverted index, 임베딩 캐시 파일 등).
3. **트랜잭션/원자성** — SQLite는 여러 행 변경을 하나의 트랜잭션으로 묶을 수
   있지만, 파일+git은 "여러 파일을 쓰고 커밋 1번"으로 흉내내야 하고, 커밋 직전에
   프로세스가 죽으면 일부 파일만 바뀐 상태로 남을 수 있다(SQLite의 WAL/롤백이
   주는 원자성 보장이 없음) — 별도 처리 필요.
4. **gate.py/promote.py 등 파이프라인 연동** — 지금 프로토타입은 CRUD+검색만
   확인했고, 실제 마이그레이션은 `core/pipeline/*`가 기대하는 `wiki_store.*`
   함수 시그니처에 맞춰 어댑터를 만들거나 파이프라인 코드 자체를 수정해야 한다.

## 종합 결론

파일+git은 ③(복제)은 거의 공짜로 해결하지만, ①(단일 writer)은 "없앤다"가 아니라
"배치 커밋으로 설계해서 회피한다"가 정확한 표현이다. 즉시-쓰기가 잦은 워크로드
(공개 데모 채팅)에는 SQLite보다 뚜렷이 낫다고 보기 어렵고, 배치형 파이프라인
워크로드(갱신 사이클)에는 잘 맞는다. 전면 마이그레이션보다는, **이 프로젝트처럼
"서빙은 읽기 위주 + 갱신은 배치"인 구조에서 시도해볼 가치가 있는 방향**이라는
결론.
